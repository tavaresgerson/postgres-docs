### 5.3. Colunas de Identidade [#](#DDL-IDENTITY-COLUMNS)

Uma coluna de identidade é uma coluna especial que é gerada automaticamente a partir de uma sequência implícita. Ela pode ser usada para gerar valores de chave.

Para criar uma coluna de identidade, use a cláusula `GENERATED ... AS IDENTITY` em `CREATE TABLE`, por exemplo:

```
CREATE TABLE people (
    id bigint GENERATED ALWAYS AS IDENTITY,
    ...,
);
```

ou, como alternativa

```
CREATE TABLE people (
    id bigint GENERATED BY DEFAULT AS IDENTITY,
    ...,
);
```

Veja [CREATE TABLE](sql-createtable.md "CREATE TABLE") para mais detalhes.

Se um comando `INSERT` for executado na tabela com a coluna de identidade e não for especificado explicitamente um valor para a coluna de identidade, então um valor gerado pela sequência implícita será inserido. Por exemplo, com as definições acima e assumindo colunas adicionais apropriadas, a escrita

```
INSERT INTO people (name, address) VALUES ('A', 'foo');
INSERT INTO people (name, address) VALUES ('B', 'bar');
```

geraria valores para a coluna `id`, começando em 1, e resultaria nos seguintes dados da tabela:

```
 id | name | address
----+------+---------
  1 | A    | foo
  2 | B    | bar
```

Como alternativa, a palavra-chave `DEFAULT` pode ser especificada no lugar de um valor para solicitar explicitamente o valor gerado pela sequência, como

```
INSERT INTO people (id, name, address) VALUES (DEFAULT, 'C', 'baz');
```

Da mesma forma, a palavra-chave `DEFAULT` pode ser usada em comandos `UPDATE`.

Assim, de muitas maneiras, uma coluna de identidade se comporta como uma coluna com um valor padrão.

As cláusulas `ALWAYS` e `BY DEFAULT` na definição da coluna determinam como os valores especificados explicitamente pelo usuário são tratados nos comandos `INSERT` e `UPDATE`. Em um comando `INSERT`, se `ALWAYS` for selecionado, um valor especificado explicitamente pelo usuário é aceito apenas se a declaração `INSERT` especificar `OVERRIDING SYSTEM VALUE`. Se `BY DEFAULT` for selecionado, então o valor especificado pelo usuário tem precedência. Assim, usar `BY DEFAULT` resulta em um comportamento mais semelhante aos valores padrão, onde o valor padrão pode ser sobrescrito por um valor explícito, enquanto `ALWAYS` oferece alguma proteção adicional contra a inserção acidental de um valor explícito.

O tipo de dados de uma coluna de identidade deve ser um dos tipos de dados suportados por sequências. (Veja [CREATE SEQUENCE](sql-createsequence.md). As propriedades da sequência associada podem ser especificadas ao criar uma coluna de identidade (veja [CREATE TABLE](sql-createtable.md)) ou alteradas posteriormente (veja [ALTER TABLE](sql-altertable.md)).

Uma coluna de identidade é automaticamente marcada como `NOT NULL`. No entanto, uma coluna de identidade não garante a unicidade. (Uma sequência normalmente retorna valores únicos, mas uma sequência pode ser redefinida ou valores podem ser inseridos manualmente na coluna de identidade, conforme discutido acima.) A unicidade precisaria ser aplicada usando uma restrição `PRIMARY KEY` ou `UNIQUE`.

Nas hierarquias de herança de tabelas, as colunas de identidade e suas propriedades em uma tabela filha são independentes das de suas tabelas pai. Uma tabela filha não herda automaticamente colunas de identidade ou suas propriedades das tabelas pai. Durante `INSERT` ou `UPDATE`, uma coluna é tratada como uma coluna de identidade se essa coluna for uma coluna de identidade na tabela nomeada na declaração e as propriedades de identidade correspondentes são aplicadas.

As partições herdam as colunas de identidade da tabela particionada. Elas não podem ter suas próprias colunas de identidade. As propriedades de uma coluna de identidade dada são consistentes em todas as partições da hierarquia de partições.
