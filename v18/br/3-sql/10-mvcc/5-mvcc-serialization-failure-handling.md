## 13.5. Gerenciamento de falha de serialização [#](#MVCC-SERIALIZATION-FAILURE-HANDLING)

Tanto os níveis de isolamento de leitura repetida quanto os de serialização podem produzir erros projetados para prevenir anomalias de serialização. Como mencionado anteriormente, as aplicações que utilizam esses níveis devem estar preparadas para repetir as transações que falham devido a erros de serialização. O texto da mensagem desse erro variará de acordo com as circunstâncias precisas, mas sempre terá o código SQLSTATE `40001` (`serialization_failure`).

Também pode ser aconselhável tentar novamente as falhas de bloqueio. Essas têm o código SQLSTATE `40P01` (`deadlock_detected`).

Em alguns casos, também é apropriado repetir os erros de chave única, que têm o código SQLSTATE `23505` (`unique_violation`), e os erros de restrição de exclusão, que têm o código SQLSTATE `23P01` (`exclusion_violation`). Por exemplo, se o aplicativo selecionar um novo valor para uma coluna de chave primária após inspecionar as chaves armazenadas atualmente, ele pode obter um erro de chave única porque outra instância do aplicativo selecionou a mesma nova chave simultaneamente. Isso é efetivamente um erro de serialização, mas o servidor não o detectará como tal, porque não pode "ver" a conexão entre o valor inserido e as leituras anteriores. Há também alguns casos raros em que o servidor emitirá um erro de chave única ou restrição de exclusão, mesmo que, em princípio, tenha informações suficientes para determinar que um problema de serialização é a causa subjacente. Embora seja recomendável apenas repetir os erros `serialization_failure` incondicionalmente, é necessário mais cuidado ao repetir esses outros códigos de erro, pois eles podem representar condições de erro persistentes, em vez de falhas transitórias.

É importante repetir a transação completa, incluindo toda a lógica que decide qual SQL emitir e/ou quais valores usar. Portanto, o PostgreSQL não oferece uma funcionalidade de repetição automática, uma vez que não pode fazer isso com qualquer garantia de correção.

A tentativa de refazer uma transação não garante que a transação retificada seja concluída; pode ser necessário realizar várias tentativas. Em casos com alta concorrência, é possível que a conclusão de uma transação leve muitas tentativas. Em casos que envolvem uma transação preparada em conflito, pode não ser possível fazer progresso até que a transação preparada seja confirmada ou revertida.
