## Apêndice A. Códigos de Erro do PostgreSQL

Todas as mensagens emitidas pelo servidor PostgreSQL recebem códigos de erro de cinco caracteres que seguem as convenções do padrão SQL para códigos “SQLSTATE”. As aplicações que precisam saber qual condição de erro ocorreu geralmente devem testar o código de erro, em vez de olhar para a mensagem de erro textual. Os códigos de erro têm menos probabilidade de mudar nas versões do PostgreSQL, e também não estão sujeitos a mudanças devido à localização das mensagens de erro. Note que alguns, mas não todos, dos códigos de erro produzidos pelo PostgreSQL são definidos pelo padrão SQL; alguns códigos de erro adicionais para condições não definidos pelo padrão foram inventados ou emprestados de outras bases de dados.

De acordo com a norma, os dois primeiros caracteres de um código de erro indicam uma classe de erros, enquanto os três últimos caracteres indicam uma condição específica dentro dessa classe. Assim, uma aplicação que não reconheça o código de erro específico ainda pode inferir o que fazer a partir da classe de erro.

[Tabela A.1](errcodes-appendix.md#ERRCODES-TABLE) lista todos os códigos de erro definidos no PostgreSQL 18.4. (Alguns não são realmente usados atualmente, mas são definidos pelo padrão SQL.) As classes de erro também são mostradas. Para cada classe de erro, há um código de erro “padrão” com os três últimos caracteres `000`. Esse código é usado apenas para condições de erro que se enquadram na classe, mas não têm nenhum código mais específico atribuído.

O símbolo mostrado na coluna “Nome da condição” é o nome da condição a ser usado no PL/pgSQL. Os nomes das condições podem ser escritos em maiúsculas ou minúsculas. (Observe que o PL/pgSQL não reconhece nomes de condições de alerta, ao contrário dos nomes de condições de erro; esses são os tipos 00, 01 e 02.)

Para alguns tipos de erros, o servidor reporta o nome de um objeto de banco de dados (uma tabela, coluna de tabela, tipo de dados ou restrição) associado ao erro; por exemplo, o nome da restrição única que causou um erro `unique_violation`. Esses nomes são fornecidos em campos separados da mensagem do relatório de erro, para que as aplicações não precisem tentar extraí-los do texto possivelmente localizado e legível pelo ser humano da mensagem. A partir do PostgreSQL 9.3, a cobertura completa para este recurso existe apenas para erros na classe SQLSTATE 23 (violação de restrição de integridade), mas isso provavelmente será expandido no futuro.
