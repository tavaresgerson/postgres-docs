## 13.4. Verificação de Consistência de Dados no Nível da Aplicação [#](#APPLEVEL-CONSISTENCY)

* [13.4.1. Exigir consistência com transações serializáveis](applevel-consistency.md#SERIALIZABLE-CONSISTENCY)
* [13.4.2. Exigir consistência com bloqueios explícitos](applevel-consistency.md#NON-SERIALIZABLE-CONSISTENCY)

É muito difícil aplicar regras de negócios em relação à integridade dos dados usando transações de Comprometimento de Leitura, porque a visão dos dados muda a cada declaração, e até mesmo uma única declaração pode não se restringir ao instantâneo da declaração se ocorrer um conflito de escrita.

Embora uma transação de Leitura Repetível tenha uma visão estável dos dados durante sua execução, há um problema sutil ao usar instantâneos MVCC para verificações de consistência de dados, envolvendo algo conhecido como *conflitos de leitura/escrita*. Se uma transação escreve dados e uma transação concorrente tenta ler os mesmos dados (seja antes ou depois da escrita), ela não pode ver o trabalho da outra transação. O leitor então parece ter executado primeiro, independentemente de qual tenha começado primeiro ou qual tenha sido executado primeiro. Se for isso, não há problema, mas se o leitor também escreve dados que são lidos por uma transação concorrente, agora há uma transação que parece ter sido executada antes de qualquer uma das transações mencionadas anteriormente. Se a transação que parece ter sido executada por último realmente comprometer primeiro, é muito fácil que um ciclo apareça em um gráfico da ordem de execução das transações. Quando tal ciclo aparece, as verificações de integridade não funcionarão corretamente sem alguma ajuda.

Como mencionado na [Seção 13.2.3][(transaction-iso.md#XACT-SERIALIZABLE "13.2.3. Serializable Isolation Level")], as transações serializáveis são apenas transações de Leitura Repetível que adicionam monitoramento não bloqueável para padrões perigosos de conflitos de leitura/escrita. Quando é detectado um padrão que poderia causar um ciclo na ordem aparente de execução, uma das transações envolvidas é revertida para quebrar o ciclo.

### 13.4.1. Exigir Consistência com Transações Serializáveis [#](#SERIALIZABLE-CONSISTENCY)

Se o nível de isolamento de transação Serializable for usado para todas as escritas e para todas as leituras que precisam de uma visão consistente dos dados, não é necessário nenhum outro esforço para garantir a consistência. O software de outros ambientes que é escrito para usar transações serializáveis para garantir a consistência deve "funcionar" nesse sentido no PostgreSQL.

Ao usar essa técnica, evitará criar uma carga desnecessária para os programadores de aplicativos, caso o software de aplicativo passe por uma estrutura que retome automaticamente as transações que são revertidas com falha de serialização. Pode ser uma boa ideia definir `default_transaction_isolation` para `serializable`. Também seria sábio tomar algumas ações para garantir que nenhum outro nível de isolamento de transação seja usado, seja inadvertidamente ou para subverter as verificações de integridade, por meio de verificações do nível de isolamento de transação em gatilhos.

Veja [Seção 13.2.3][(transaction-iso.md#XACT-SERIALIZABLE "13.2.3. Serializable Isolation Level")] para sugestões de desempenho.

### Aviso: Transações serializáveis e replicação de dados

Esse nível de proteção de integridade usando transações serializáveis ainda não se estende ao modo standby quente ([Seção 26.4][(hot-standby.md "26.4. Hot Standby")]) ou réplicas lógicas. Por isso, aqueles que usam standby quente ou replicação lógica podem querer usar Leitura Repetível e bloqueio explícito no primário.

### 13.4.2. Exigir consistência com bloqueios explícitos [#](#NON-SERIALIZABLE-CONSISTENCY)

Quando são possíveis escritas não serializáveis, para garantir a validade atual de uma linha e protegê-la contra atualizações concorrentes, é necessário usar `SELECT FOR UPDATE`, `SELECT FOR SHARE` ou uma declaração apropriada `LOCK TABLE`. (`SELECT FOR UPDATE` e `SELECT FOR SHARE` bloqueiam apenas as linhas devolvidas contra atualizações concorrentes, enquanto `LOCK TABLE` bloqueia toda a tabela.) Isso deve ser levado em consideração ao migrar aplicativos para PostgreSQL de outros ambientes.

Vale ressaltar para aqueles que estão fazendo a conversão de outros ambientes o fato de que `SELECT FOR UPDATE` não garante que uma transação concorrente não irá atualizar ou excluir uma linha selecionada. Para fazer isso no PostgreSQL, você deve, na verdade, atualizar a linha, mesmo que não seja necessário alterar nenhum valor. `SELECT FOR UPDATE` *bloqueia temporariamente* outras transações de adquirir o mesmo bloqueio ou executar uma `UPDATE` ou `DELETE` que afetaria a linha bloqueada, mas assim que a transação que detém este bloqueio se compromet ou se desfaz, uma transação bloqueada prosseguirá com a operação conflitante, a menos que uma `UPDATE` real da linha tenha sido realizada enquanto o bloqueio era mantido.

Os verificações de validade global exigem um pensamento adicional em MVCC não serializável. Por exemplo, um aplicativo bancário pode desejar verificar se a soma de todos os créditos em uma tabela é igual à soma dos débitos em outra tabela, quando ambas as tabelas estão sendo atualizadas ativamente. Comparar os resultados de dois comandos sucessivos do `SELECT sum(...)` não funcionará de forma confiável no modo de compromisso de leitura, uma vez que a segunda consulta provavelmente incluirá os resultados de transações que não foram contadas pelo primeiro. Fazer as duas somas em uma única transação de leitura repetitiva dará uma imagem precisa apenas dos efeitos das transações que se comprometaram antes do início da transação de leitura repetitiva — mas alguém pode legitimamente se perguntar se a resposta ainda é relevante no momento em que é entregue. Se a própria transação de leitura repetitiva aplicou algumas mudanças antes de tentar fazer a verificação de consistência, a utilidade da verificação se torna ainda mais discutível, uma vez que agora inclui algumas, mas não todas, mudanças pós-início da transação. Nesses casos, uma pessoa cuidadosa pode desejar bloquear todas as tabelas necessárias para a verificação, a fim de obter uma imagem indiscutível da realidade atual. Um bloqueio no modo `SHARE` (ou superior) garante que não haja mudanças não comprometidas na tabela bloqueada, além daquelas da transação atual.

Observe também que, se se está confiando em bloqueio explícito para impedir mudanças concorrentes, deve-se usar o modo de Comprometimento de Leitura, ou, no modo de Leitura Repetível, ter cuidado para obter blocos antes de realizar consultas. Um bloqueio obtido por uma transação de leitura repetitiva garante que nenhuma outra transação que modifica a tabela ainda esteja em execução, mas se o instantâneo visto pela transação antecipar a obtenção do bloqueio, pode antecipar algumas mudanças agora comprometidas na tabela. O instantâneo de uma transação de leitura repetitiva é, na verdade, congelado no início de sua primeira consulta ou comando de modificação de dados (`SELECT`, `INSERT`, `UPDATE`, `DELETE` ou `MERGE`), portanto, é possível obter blocos explicitamente antes do instantâneo ser congelado.
