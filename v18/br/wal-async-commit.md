## 28.4. Compromisso Asíncrono [#](#WAL-ASYNC-COMMIT)

*Commit assíncrono* é uma opção que permite que as transações sejam concluídas mais rapidamente, com o custo de que as transações mais recentes podem ser perdidas, caso o banco de dados sofra um crash. Em muitas aplicações, esse é um compromisso aceitável.

Como descrito na seção anterior, o commit de transação é normalmente *síncrono*: o servidor espera que os registros do WAL da transação sejam descarregados para o armazenamento permanente antes de retornar uma indicação de sucesso ao cliente. Portanto, o cliente é garantido que uma transação relatada como comprometida será preservada, mesmo em caso de falha do servidor imediatamente após. No entanto, para transações curtas, esse atraso é um componente importante do tempo total da transação. Selecionar o modo de commit assíncrono significa que o servidor retorna sucesso assim que a transação é logicamente concluída, antes de os registros do WAL que ela gerou terem sido efetivamente descarregados no disco. Isso pode proporcionar um impulso significativo no desempenho para pequenas transações.

O commit assíncrono introduz o risco de perda de dados. Há uma janela de tempo curta entre o relatório da conclusão da transação ao cliente e o momento em que a transação é realmente comprometida (ou seja, é garantido que não será perdida se o servidor falhar). Assim, o commit assíncrono não deve ser usado se o cliente realizar ações externas que dependem da suposição de que a transação será lembrada. Como exemplo, um banco certamente não usaria commit assíncrono para uma transação que registra o dispensamento de dinheiro de um caixa automático. Mas, em muitos cenários, como o registro de eventos, não há necessidade de uma garantia forte desse tipo.

O risco que é assumido ao usar um commit assíncrono é a perda de dados, não a corrupção de dados. Se o banco de dados falhar, ele se recuperará replicando o WAL até o último registro que foi descarregado. Portanto, o banco de dados será restaurado a um estado autoconsistente, mas quaisquer transações que ainda não tenham sido descarregadas no disco não serão refletidas nesse estado. O efeito líquido, portanto, é a perda das últimas transações. Como as transações são replicadas na ordem do commit, não pode ser introduzida nenhuma inconsistência — por exemplo, se a transação B fez alterações baseadas nos efeitos de uma transação anterior A, não é possível que os efeitos de A sejam perdidos enquanto os efeitos de B são preservados.

O usuário pode selecionar o modo de compromisso de cada transação, de modo que seja possível ter transações de compromisso síncronas e assíncronas em execução simultaneamente. Isso permite opções flexíveis entre desempenho e certeza da durabilidade da transação. O modo de compromisso é controlado pelo parâmetro configurado pelo usuário [synchronous_commit][(runtime-config-wal.md#GUC-SYNCHRONOUS-COMMIT)], que pode ser alterado de qualquer maneira que um parâmetro de configuração possa ser configurado. O modo usado para qualquer uma transação depende do valor de `synchronous_commit` quando o compromisso da transação começa.

Certos comandos de utilidade, por exemplo `DROP TABLE`, são obrigados a se comprometerem de forma síncrona, independentemente da configuração de `synchronous_commit`. Isso é feito para garantir a consistência entre o sistema de arquivos do servidor e o estado lógico do banco de dados. Os comandos que suportam o compromisso de duas fases, como `PREPARE TRANSACTION`, também são sempre síncronos.

Se o banco de dados falhar durante a janela de risco entre um commit assíncrono e a escrita dos registros WAL da transação, as alterações feitas durante essa transação serão *perdidas*. A duração da janela de risco é limitada porque um processo em segundo plano (o "escritor WAL") esvazia os registros WAL não escritos no disco a cada [wal_writer_delay][(runtime-config-wal.md#GUC-WAL-WRITER-DELAY)] milissegundos. A duração máxima real da janela de risco é três vezes `wal_writer_delay`, pois o escritor WAL é projetado para favorecer a escrita de páginas inteiras de uma só vez durante períodos ocupados.

### Atenção

Um desligamento em modo imediato é equivalente a um crash do servidor e, portanto, causará a perda de quaisquer commits assíncronos não lavados.

O commit assíncrono fornece um comportamento diferente da configuração [fsync][(runtime-config-wal.md#GUC-FSYNC)]. `fsync` é um ajuste para todo o servidor que alterará o comportamento de todas as transações. Ele desativa toda a lógica dentro do PostgreSQL que tenta sincronizar os dados de diferentes partes do banco de dados, e, portanto, um travamento do sistema (ou seja, um travamento de hardware ou do sistema operacional, e não uma falha do PostgreSQL em si) poderia resultar em corrupção arbitrária do estado do banco de dados. Em muitos cenários, o commit assíncrono fornece a maioria das melhorias de desempenho que poderiam ser obtidas ao desativar `fsync`, mas sem o risco de corrupção de dados.

[commit_delay][(runtime-config-wal.md#GUC-COMMIT-DELAY) também soa muito semelhante ao commit assíncrono, mas na verdade é um método de commit síncrono (de fato, `commit_delay` é ignorado durante um commit assíncrono). `commit_delay` causa um atraso logo antes de uma transação limpar o WAL no disco, na esperança de que uma única limpeza executada por uma dessas transações também possa servir para outras transações que estão se comprometendo aproximadamente na mesma época. O ajuste pode ser visto como uma maneira de aumentar a janela de tempo em que as transações podem se juntar a um grupo prestes a participar de uma única limpeza, para amortizar o custo da limpeza entre várias transações.
