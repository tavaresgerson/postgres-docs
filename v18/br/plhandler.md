## Capítulo 57. Escrevendo um Manipulador de Linguagem Procedimental

Todas as chamadas a funções que são escritas em uma linguagem que não é a interface atual da "versão 1" para linguagens compiladas (isso inclui funções em linguagens procedimentais definidas pelo usuário e funções escritas em SQL) passam por uma função *manipulador de chamada* para a linguagem específica. É responsabilidade do manipulador de chamada executar a função de uma maneira significativa, como interpretando o texto de origem fornecido. Este capítulo descreve como um manipulador de chamada de uma nova linguagem procedural pode ser escrito.

O manipulador de chamadas para uma linguagem procedural é uma função “normal” que deve ser escrita em uma linguagem compilada, como C, usando a interface da versão 1, e registrada no PostgreSQL como não recebendo argumentos e retornando o tipo `language_handler`. Este pseudo-tipo especial identifica a função como um manipulador de chamadas e impede que ela seja chamada diretamente em comandos SQL. Para mais detalhes sobre as convenções de chamada em linguagem C e carregamento dinâmico, consulte [Seção 36.10][(xfunc-c.md "36.10. C-Language Functions")].

O manipulador de chamadas é chamado da mesma maneira que qualquer outra função: Ele recebe um ponteiro para um `FunctionCallInfoBaseData` `struct` contendo valores de argumento e informações sobre a função chamada, e espera-se que ele retorne um resultado `Datum` (e, possivelmente, defina o campo `isnull` da estrutura `FunctionCallInfoBaseData`, se desejar retornar um resultado nulo SQL). A diferença entre um manipulador de chamadas e uma função callee comum é que o campo `flinfo->fn_oid` da estrutura `FunctionCallInfoBaseData` conterá o OID da função real a ser chamada, não do próprio manipulador de chamadas. O manipulador de chamadas deve usar esse campo para determinar qual função executar. Além disso, a lista de argumentos passada foi configurada de acordo com a declaração da função alvo, não do manipulador de chamadas.

Cabe ao manipulador da chamada buscar a entrada da função do catálogo do sistema `pg_proc` e analisar os tipos de argumento e retorno da função chamada. A cláusula `AS` do comando `CREATE FUNCTION` para a função será encontrada na coluna `prosrc` da linha `pg_proc`. Este é um texto comum no idioma processual, mas, em teoria, poderia ser algo mais, como um nome de caminho para um arquivo, ou qualquer outra coisa que diga ao manipulador da chamada o que fazer em detalhes.

Muitas vezes, a mesma função é chamada várias vezes por declaração SQL. Um manipulador de chamada pode evitar buscas repetidas de informações sobre a função chamada usando o campo `flinfo->fn_extra`. Inicialmente, isso será `NULL`, mas pode ser configurado pelo manipulador de chamada para apontar para informações sobre a função chamada. Em chamadas subsequentes, se `flinfo->fn_extra` já não for `NULL`, ele pode ser usado e a etapa de busca de informações ignorada. O manipulador de chamada deve garantir que `flinfo->fn_extra` seja feito para apontar para memória que viverá pelo menos até o final da consulta atual, pois uma estrutura de dados `FmgrInfo` pode ser mantida por um longo período. Uma maneira de fazer isso é alocar os dados extras no contexto de memória especificado por `flinfo->fn_mcxt`; tais dados normalmente terão a mesma vida útil que o próprio `FmgrInfo`. Mas o manipulador também pode optar por usar um contexto de memória com vida útil mais longa para que possa cachear informações de definição de função em várias consultas.

Quando uma função de linguagem procedural é invocada como um gatilho, não são passados argumentos da maneira usual, mas o campo `context` do `FunctionCallInfoBaseData` aponta para uma estrutura `TriggerData`, em vez de ser `NULL`, como em uma chamada de função simples. Um manipulador de linguagem deve fornecer mecanismos para que as funções de linguagem procedural acessem as informações do gatilho.

Um modelo para um manipulador de linguagem procedural escrito como uma extensão em C é fornecido em `src/test/modules/plsample`. Esta é uma amostra funcional que demonstra uma maneira de criar um manipulador de linguagem procedural, processar parâmetros e retornar um valor.

Embora fornecer um manipulador de chamadas seja suficiente para criar uma linguagem procedural mínima, existem outras duas funções que podem ser fornecidas opcionalmente para tornar a linguagem mais conveniente de usar. Essas são um *validator* e um *manipulador inline*. Um validator pode ser fornecido para permitir que verificações específicas da linguagem sejam feitas durante [CREATE FUNCTION][(sql-createfunction.md "CREATE FUNCTION")]. Um manipulador inline pode ser fornecido para permitir que a linguagem suporte blocos de código anônimos executados via o comando [DO][(sql-do.md "DO")].

Se um validador for fornecido por uma linguagem procedural, ele deve ser declarado como uma função que recebe um único parâmetro do tipo `oid`. O resultado do validador é ignorado, portanto, ele é costumeiramente declarado para retornar `void`. O validador será chamado no final de um comando `CREATE FUNCTION` que criou ou atualizou uma função escrita na linguagem procedural. O OID passado é o OID da linha `pg_proc` da função. O validador deve buscar essa linha da maneira usual e fazer qualquer verificação apropriada. Primeiro, chame `CheckFunctionValidatorAccess()` para diagnosticar chamadas explícitas ao validador que o usuário não conseguiu realizar por meio de `CREATE FUNCTION`. Verificações típicas incluem então verificar que os tipos de argumento e resultado da função são suportados pela linguagem e que o corpo da função é sintaticamente correto na linguagem. Se o validador encontrar a função como correta, ele deve apenas retornar. Se encontrar um erro, deve relatar isso via o mecanismo normal de relatórios de erro `ereport()`. Lançar um erro forçará o rollback da transação e, assim, impedirá que a definição incorreta da função seja comprometida.

As funções de validação devem, normalmente, respeitar o parâmetro [check_function_bodies][(runtime-config-client.md#GUC-CHECK-FUNCTION-BODIES)]. Se estiver desativado, qualquer verificação cara ou sensível ao contexto deve ser ignorada. Se a linguagem permitir a execução do código no momento da compilação, o validador deve suprimir as verificações que induziriam tal execução. Em particular, este parâmetro é desativado pelo pg_dump para que ele possa carregar funções de linguagem procedural sem se preocupar com efeitos colaterais ou dependências dos corpos das funções em outros objetos do banco de dados. (Devido a essa exigência, o manipulador de chamadas deve evitar assumir que o validador verificou completamente a função. O ponto de ter um validador não é deixar que o manipulador de chamadas omita verificações, mas notificar o usuário imediatamente se houver erros óbvios em um comando `CREATE FUNCTION`. Embora a escolha exata do que verificar seja deixada, na maioria das vezes, à discrição da função de validação, observe que o código central `CREATE FUNCTION` executa apenas as cláusulas `SET` anexadas a uma função quando `check_function_bodies` está ativado. Portanto, as verificações cujos resultados possam ser afetados por parâmetros GUC definitivamente devem ser ignoradas quando `check_function_bodies` está desativado, para evitar falhas falsas ao restaurar um dump.

Se um manipulador inline for fornecido por uma linguagem procedural, ele deve ser declarado como uma função que recebe um único parâmetro do tipo `internal`. O resultado do manipulador inline é ignorado, portanto, ele é customariamente declarado para retornar `void`. O manipulador inline será chamado quando uma declaração `DO` for executada, especificando a linguagem procedural. O parâmetro realmente passado é um ponteiro para uma estrutura `InlineCodeBlock`, que contém informações sobre os parâmetros da declaração `DO`, em particular o texto do bloco de código anônimo a ser executado. O manipulador inline deve executar esse código e retornar.

Recomenda-se que você envolva todas essas declarações de função, bem como o próprio comando `CREATE LANGUAGE`, em uma *extensão* para que um simples comando `CREATE EXTENSION` seja suficiente para instalar a linguagem. Consulte a [Seção 36.17][(extend-extensions.md "36.17. Packaging Related Objects into an Extension")] para obter informações sobre a escrita de extensões.

Os idiomas processuais incluídos na distribuição padrão são boas referências ao tentar escrever seu próprio manipulador de linguagem. Consulte o subdiretório `src/pl` da árvore de origem. A página de referência [CREATE LANGUAGE](sql-createlanguage.md "CREATE LANGUAGE") também tem alguns detalhes úteis.
