Os bancos de dados PostgreSQL requerem manutenção periódica conhecida como *vazamento*. Para muitas instalações, é suficiente deixar que o *daemon de autovazamento* realize o vazamento, que é descrito em [Seção 24.1.6](routine-vacuuming.md#AUTOVACUUM). Você pode precisar ajustar os parâmetros de autovazamento descritos lá para obter os melhores resultados para sua situação. Alguns administradores de banco de dados desejam complementar ou substituir as atividades do daemon com comandos gerenciados manualmente `VACUUM`, que são executados tipicamente de acordo com um cronograma por scripts do cron ou do Gerenciador de Tarefas. Para configurar o vazamento gerenciado manualmente corretamente, é essencial entender os problemas discutidos nas próximas subseções. Administradores que dependem do autovazamento ainda podem querer revisar este material para ajudá-los a entender e ajustar o autovazamento.

### 24.1.1. Básico de Aspiração [#](#VACUUM-BASICS)

O comando `VACUUM` do PostgreSQL (sql-vacuum.md "VACUUM") precisa processar cada tabela regularmente por várias razões:

1. Para recuperar ou reutilizar o espaço em disco ocupado por linhas atualizadas ou excluídas.
2. Para atualizar as estatísticas de dados usadas pelo planejador de consultas do PostgreSQL.
3. Para atualizar o mapa de visibilidade, o que acelera a varredura apenas com índice (indexes-index-only-scans.md "11.9. Index-Only Scans and Covering Indexes").
4. Para proteger contra a perda de dados muito antigos devido ao *wraparound de ID de transação* ou *wraparound de ID multixact*.

Cada uma dessas razões determina a realização de operações `VACUUM` de frequência e abrangência variáveis, conforme explicado nas seções a seguir.

Existem duas variantes do `VACUUM`: a variante padrão `VACUUM` e a variante `VACUUM FULL`. O `VACUUM FULL` pode recuperar mais espaço em disco, mas funciona muito mais lentamente. Além disso, a forma padrão do `VACUUM` pode ser executada em paralelo com operações de banco de dados de produção. (Os comandos como `SELECT`, `INSERT`, `UPDATE` e `DELETE` continuarão a funcionar normalmente, embora você não possa modificar a definição de uma tabela com comandos como `ALTER TABLE` enquanto ela estiver sendo aspirada.) O `VACUUM FULL` requer um `ACCESS EXCLUSIVE` de bloqueio na tabela em que está trabalhando, e, portanto, não pode ser feito em paralelo com outro uso da tabela. Geralmente, portanto, os administradores devem se esforçar para usar a variante padrão `VACUUM` e evitar o `VACUUM FULL`.

`VACUUM` gera uma quantidade substancial de tráfego de E/S, o que pode causar um desempenho ruim para outras sessões ativas. Existem parâmetros de configuração que podem ser ajustados para reduzir o impacto no desempenho do rastreamento em segundo plano — consulte [Seção 19.10.2](runtime-config-vacuum.md#RUNTIME-CONFIG-RESOURCE-VACUUM-COST).

### 24.1.2. Recuperação de espaço em disco [#](#VACUUM-FOR-SPACE-RECOVERY)

Em PostgreSQL, uma `UPDATE` ou `DELETE` de uma linha não remove imediatamente a versão antiga da linha. Essa abordagem é necessária para obter os benefícios do controle de concorrência multiversão (MVCC, veja [Capítulo 13](mvcc.md)): a versão da linha não deve ser excluída enquanto ainda é potencialmente visível para outras transações. Mas, eventualmente, uma versão de linha desatualizada ou excluída não é mais de interesse para nenhuma transação. O espaço que ela ocupa deve então ser recuperado para uso por novas linhas, para evitar o crescimento ilimitado dos requisitos de espaço em disco. Isso é feito executando `VACUUM`.

O formulário padrão do `VACUUM` remove as versões de linha morta em tabelas e índices e marca o espaço disponível para uso futuro. No entanto, ele não retorna o espaço ao sistema operacional, exceto no caso especial em que uma ou mais páginas no final de uma tabela se tornam totalmente livres e uma bloqueio exclusivo de tabela pode ser facilmente obtido. Em contraste, o `VACUUM FULL` compacta ativamente as tabelas, escrevendo uma nova versão completa do arquivo da tabela sem espaço morto. Isso minimiza o tamanho da tabela, mas pode levar um longo tempo. Também requer espaço de disco extra para a nova cópia da tabela, até que a operação seja concluída.

O objetivo usual da aspiração rotineira é realizar `VACUUM`s padrão o suficiente para evitar a necessidade de `VACUUM FULL`. O daemon de autovacuagem tenta trabalhar dessa maneira e, de fato, nunca emitirá `VACUUM FULL`. Neste enfoque, a ideia não é manter as tabelas no seu tamanho mínimo, mas sim manter o uso estável do espaço em disco: cada tabela ocupa espaço equivalente ao seu tamanho mínimo mais o espaço que é usado entre as execuções de vácuo. Embora `VACUUM FULL` possa ser usado para reduzir uma tabela ao seu tamanho mínimo e devolver o espaço em disco ao sistema operacional, não há muito sentido nisso se a tabela simplesmente crescer novamente no futuro. Assim, execuções padrão de `VACUUM` moderadamente frequentes são uma abordagem melhor do que execuções de `VACUUM FULL` infrequentes para manter tabelas fortemente atualizadas.

Alguns administradores preferem agendar a limpeza de vácuo eles mesmos, por exemplo, fazendo todo o trabalho à noite, quando a carga é baixa. A dificuldade em fazer a limpeza de vácuo de acordo com um cronograma fixo é que, se uma tabela tiver um pico inesperado de atividade de atualização, ela pode ficar inchada ao ponto de que `VACUUM FULL` é realmente necessário para recuperar espaço. Usar o daemon de autovacuum alivia esse problema, uma vez que o daemon agende a limpeza de vácuo dinamicamente em resposta à atividade de atualização. Não é sábio desativar o daemon completamente, a menos que você tenha uma carga de trabalho extremamente previsível. Uma possível solução é definir os parâmetros do daemon para que ele só reaja a atividades de atualização incomumente pesadas, mantendo assim as coisas sob controle, enquanto os `VACUUM`s agendados são esperados para fazer a maior parte do trabalho quando a carga é típica.

Para aqueles que não usam autovacuum, uma abordagem típica é agendar um `VACUUM` em todo o banco de dados uma vez por dia durante um período de baixo uso, complementado por um rastreamento mais frequente de tabelas com muitas atualizações, conforme necessário. (Algumas instalações com taxas de atualização extremamente altas rastreamento suas tabelas mais movimentadas tão frequentemente quanto uma vez a cada poucos minutos.) Se você tem vários bancos de dados em um clúster, não se esqueça de `VACUUM` cada um; o programa [vacuumdb](app-vacuumdb.md) pode ser útil.

### DICA

A opção `VACUUM` simples pode não ser satisfatória quando uma tabela contém um grande número de versões de linha morta como resultado de uma atividade de atualização ou exclusão massiva. Se você tem uma tabela desse tipo e precisa recuperar o espaço em disco excedente que ela ocupa, você precisará usar `VACUUM FULL`, ou, como alternativa, `CLUSTER`(sql-cluster.md "CLUSTER") ou uma das variantes de reescrita de tabela de `ALTER TABLE`(sql-altertable.md "ALTER TABLE"). Esses comandos reescrevem uma nova cópia inteira da tabela e constroem novos índices para ela. Todas essas opções exigem um `ACCESS EXCLUSIVE` lock. Note que elas também usam temporariamente espaço em disco extra aproximadamente igual ao tamanho da tabela, uma vez que as cópias antigas da tabela e dos índices não podem ser liberadas até que as novas estejam completas.

### DICA

Se você tem uma tabela cujo conteúdo inteiro é excluído periodicamente, considere fazer isso com `TRUNCATE` em vez de usar (sql-truncate.md "TRUNCATE") seguido de `DELETE` e `VACUUM`. `TRUNCATE` remove o conteúdo inteiro da tabela imediatamente, sem exigir um `VACUUM` ou `VACUUM FULL` subsequente para recuperar o espaço de disco agora não utilizado. A desvantagem é que as restrições de semântica MVCC são violadas.

### 24.1.3. Atualização das estatísticas do Planner [#](#VACUUM-FOR-STATISTICS)

O planejador de consultas do PostgreSQL depende de informações estatísticas sobre o conteúdo das tabelas para gerar bons planos para consultas. Essas estatísticas são coletadas pelo comando `ANALYZE`(sql-analyze.md "ANALYZE"), que pode ser invocado por si só ou como uma etapa opcional em `VACUUM`. É importante ter estatísticas razoavelmente precisas, caso contrário, escolhas ruins de planos podem degradar o desempenho do banco de dados.

O daemon de autovacuum, se habilitado, emitirá automaticamente os comandos `ANALYZE` sempre que o conteúdo de uma tabela tiver sido alterado de forma suficiente. No entanto, os administradores podem preferir confiar em operações `ANALYZE` programadas manualmente, especialmente se for sabido que a atividade de atualização em uma tabela não afetará as estatísticas das colunas "interessantes". O daemon programa `ANALYZE` estritamente em função do número de linhas inseridas ou atualizadas; ele não tem conhecimento de se isso levará a mudanças estatísticas significativas.

Os tuplos alterados nas partições e as crianças de herança não desencadeiam o análise na tabela pai. Se a tabela pai estiver vazia ou raramente alterada, ela pode nunca ser processada pelo autovacuum, e as estatísticas para a árvore de herança como um todo não serão coletadas. É necessário executar `ANALYZE` na tabela pai manualmente para manter as estatísticas atualizadas.

Assim como a limpeza com aspirador de pó para recuperação de espaço, atualizações frequentes de estatísticas são mais úteis para tabelas fortemente atualizadas do que para as raramente atualizadas. Mas, mesmo para uma tabela fortemente atualizada, pode não haver necessidade de atualizações de estatísticas se a distribuição estatística dos dados não estiver mudando muito. Uma regra simples é pensar em quanto os valores mínimo e máximo das colunas da tabela mudam. Por exemplo, uma coluna `timestamp` que contém a hora da atualização da linha terá um valor máximo que aumenta constantemente à medida que as linhas são adicionadas e atualizadas; tal coluna provavelmente precisará de atualizações de estatísticas mais frequentes do que, por exemplo, uma coluna que contém URLs de páginas acessadas em um site. A coluna de URL pode receber mudanças com a mesma frequência, mas a distribuição estatística de seus valores provavelmente muda relativamente lentamente.

É possível executar `ANALYZE` em tabelas específicas e até mesmo em colunas específicas de uma tabela, portanto, a flexibilidade existe para atualizar algumas estatísticas com mais frequência do que outras, se sua aplicação o exigir. Na prática, no entanto, geralmente é melhor analisar todo o banco de dados, porque é uma operação rápida. `ANALYZE` usa uma amostragem aleatória estatística das linhas de uma tabela, em vez de ler cada linha individualmente.

### DICA

Embora o ajuste por coluna da frequência de `ANALYZE` possa não ser muito produtivo, você pode achar que vale a pena fazer o ajuste por coluna do nível de detalhe das estatísticas coletadas por `ANALYZE`. Colunas que são muito utilizadas em cláusulas de `WHERE` e que têm distribuições de dados altamente irregulares podem exigir um histograma de dados com grânulos mais finos do que outras colunas. Veja `ALTER TABLE SET STATISTICS`, ou altere o padrão padrão em toda a base de dados usando o parâmetro de configuração [default_statistics_target](runtime-config-query.md#GUC-DEFAULT-STATISTICS-TARGET)

Além disso, por padrão, há informações limitadas disponíveis sobre a seletividade das funções. No entanto, se você criar um objeto de estatísticas ou um índice de expressão que utilize uma chamada de função, estatísticas úteis serão coletadas sobre a função, o que pode melhorar significativamente os planos de consulta que utilizam o índice de expressão.

### DICA

O daemon autovacuum não emite comandos `ANALYZE` para tabelas estrangeiras, uma vez que não possui meios para determinar quantas vezes isso pode ser útil. Se suas consultas exigem estatísticas sobre tabelas estrangeiras para um planejamento adequado, é uma boa ideia executar comandos manualmente gerenciados `ANALYZE` nessas tabelas em um cronograma adequado.

### DICA

O daemon autovacuum não emite comandos `ANALYZE` para tabelas particionadas. Os pais de herança serão analisados apenas se o próprio pai for alterado — as alterações nas tabelas de filho não desencadeiam o autoanalyze na tabela pai. Se suas consultas exigem estatísticas sobre as tabelas pai para um planejamento adequado, é necessário executar periodicamente um `ANALYZE` manual nessas tabelas para manter as estatísticas atualizadas.

### 24.1.4. Atualização do Mapa de Visibilidade [#](#VACUUM-FOR-VISIBILITY-MAP)

O Vacuum mantém um mapa de [visibilidade](storage-vm.md) para cada tabela para acompanhar quais páginas contêm apenas tuplas que são conhecidas como visíveis para todas as transações ativas (e todas as futuras transações, até que a página seja novamente modificada). Isso tem dois propósitos. Primeiro, o próprio vacuum pode ignorar tais páginas na próxima execução, uma vez que não há nada a ser limpo.

Em segundo lugar, permite que o PostgreSQL responda a algumas consultas usando apenas o índice, sem referência à tabela subjacente. Como os índices do PostgreSQL não contêm informações de visibilidade de tupla, uma varredura normal de índice obtém a tupla do heap para cada entrada de índice correspondente, para verificar se ela deve ser vista pela transação atual. Um [*varredura apenas de índice](indexes-index-only-scans.md), por outro lado, verifica o mapa de visibilidade primeiro. Se for conhecido que todos os tuplos na página são visíveis, a busca do heap pode ser ignorada. Isso é mais útil em conjuntos de dados grandes onde o mapa de visibilidade pode impedir acessos ao disco. O mapa de visibilidade é muito menor que o heap, então ele pode ser facilmente armazenado em cache mesmo quando o heap é muito grande.

### 24.1.5. Prevenção de falhas de envolvimento do ID de transação [#](#VACUUM-FOR-WRAPAROUND)

A semântica de transação do PostgreSQL [MVCC](mvcc-intro.md) depende da capacidade de comparar números de ID de transação (XID): uma versão de linha com um XID de inserção maior que o XID da transação atual está “no futuro” e não deve ser visível para a transação atual. Mas, como os IDs de transação têm tamanho limitado (32 bits), um clúster que funciona por um longo período (mais de 4 bilhões de transações) sofreria *wraparound de ID de transação*: o contador de XID volta a zero, e de repente, as transações que estavam no passado parecem estar no futuro — o que significa que sua saída se torna invisível. Em resumo, perda catastrófica de dados. (Na verdade, os dados ainda estão lá, mas isso é um consolo frio se você não puder acessá-los.) Para evitar isso, é necessário realizar um vácuo em todas as tabelas de todos os bancos de dados pelo menos uma vez a cada dois bilhões de transações.

A razão pela qual a limpeza periódica do vácuo resolve o problema é que o `VACUUM` marcará as linhas como *congeladas*, indicando que foram inseridas por uma transação que se comprometeu suficientemente no passado, de modo que os efeitos da transação de inserção sejam certamente visíveis para todas as transações atuais e futuras. Os XIDs normais são comparados usando aritmética módulo-232. Isso significa que, para cada XID normal, existem dois bilhões de XIDs que são “mais antigos” e dois bilhões que são “mais novos”; outra maneira de dizer é que o espaço de XID normal é circular, sem extremidade. Portanto, uma vez que uma versão de linha tenha sido criada com um determinado XID normal, a versão da linha parecerá estar “no passado” para os próximos dois bilhões de transações, independentemente do XID normal com o qual estamos falando. Se a versão da linha ainda existir após mais de dois bilhões de transações, ela de repente parecerá estar no futuro. Para evitar isso, o PostgreSQL reserva um XID especial, `FrozenTransactionId`, que não segue as regras de comparação de XID normal e é sempre considerado mais antigo que todos os XIDs normais. As versões congeladas de linha são tratadas como se o XID de inserção fosse `FrozenTransactionId`, de modo que elas parecerão estar “no passado” para todas as transações normais, independentemente de questões de envolvimento, e, portanto, essas versões de linha serão válidas até serem excluídas, independentemente de quanto tempo isso seja.

### Nota

Nas versões do PostgreSQL anteriores à 9.4, o congelamento era implementado substituindo efetivamente o XID de inserção de uma linha com `FrozenTransactionId`, que era visível na coluna `xmin` do sistema da linha. As versões mais recentes apenas definem um bit de marca, preservando o `xmin` original da linha para uso forense possível. No entanto, linhas com `xmin` igual a `FrozenTransactionId` (2) ainda podem ser encontradas em bancos de dados que foram atualizados em versões anteriores à 9.4.

Além disso, os catálogos do sistema podem conter linhas com `xmin` igual a `BootstrapTransactionId` (1), indicando que foram inseridas durante a primeira fase do initdb. Assim como o `FrozenTransactionId`, este XID especial é tratado como mais antigo do que todos os XID normais.

[vacuum_freeze_min_age](runtime-config-vacuum.md#GUC-VACUUM-FREEZE-MIN-AGE) controla a idade de um valor XID antes de as linhas que possuem esse XID serem congeladas. Aumentar essa configuração pode evitar trabalho desnecessário se as linhas que seriam congeladas em breve forem modificadas novamente, mas diminuir essa configuração aumenta o número de transações que podem passar antes de a tabela precisar ser limpa novamente.

`VACUUM` utiliza o mapa de visibilidade (storage-vm.md "66.4. Visibility Map") para determinar quais páginas de uma tabela devem ser escaneadas. Normalmente, ele ignorará páginas que não possuem versões de linha morta, mesmo que essas páginas ainda possam ter versões de linha com valores de XID antigos. Portanto, os `VACUUM`s normais não congelarão sempre todas as versões de linha antiga na tabela. Quando isso acontece, o `VACUUM` precisará realizar um *vazamento agressivo*, que congelará todos os valores de XID e MXID elegíveis não congelados, incluindo aqueles de páginas visíveis, mas não congeladas.

Se uma tabela está acumulando uma série de páginas visíveis, mas não congeladas, um vácuo normal pode optar por analisar páginas ignoráveis, na tentativa de congelá-las. Isso diminui o número de páginas que o próximo vácuo agressivo deve analisar. Essas páginas são chamadas de *analisadas com urgência*. O mapeamento agressivo pode ser ajustado para tentar congelar mais páginas visíveis, aumentando [vacuum_max_eager_freeze_failure_rate](runtime-config-vacuum.md#GUC-VACUUM-MAX-EAGER-FREEZE-FAILURE-RATE). Mesmo que o mapeamento com urgência tenha mantido o número de páginas visíveis, mas não congeladas, a um mínimo, a maioria das tabelas ainda requer um mapeamento agressivo periódico. No entanto, quaisquer páginas congeladas com urgência podem ser ignoradas durante um vácuo agressivo, portanto, o congelamento com urgência pode minimizar o overhead dos vácuos agressivos.

[vacuum_freeze_table_age](runtime-config-vacuum.md#GUC-VACUUM-FREEZE-TABLE-AGE) controla quando uma tabela é agressivamente aspirada. Todas as páginas visíveis, mas não congeladas, são verificadas se o número de transações que passaram desde o último escaneamento desse tipo for maior que `vacuum_freeze_table_age` menos `vacuum_freeze_min_age`. Definir `vacuum_freeze_table_age` para 0 obriga `VACUUM` a usar sempre sua estratégia agressiva.

O tempo máximo que uma tabela pode ficar sem ser vacumada é de dois bilhões de transações menos o valor de `vacuum_freeze_min_age` no momento da última vacumação agressiva. Se ela ficar sem ser vacumada por mais tempo do que isso, a perda de dados pode ocorrer. Para garantir que isso não aconteça, o autovacuum é invocado em qualquer tabela que possa conter linhas não congeladas com XIDs mais antigos que a idade especificada pelo parâmetro de configuração [autovacuum_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-FREEZE-MAX-AGE). (Isso ocorrerá mesmo se o autovacuum estiver desativado.)

Isso implica que, se uma tabela não for aspirada de outra forma, o autovacuamento será invocado nela aproximadamente uma vez a cada `autovacuum_freeze_max_age` menos `vacuum_freeze_min_age` transações. Para tabelas que são regularmente aspiradas para fins de recuperação de espaço, isso é de pouca importância. No entanto, para tabelas estáticas (incluindo tabelas que recebem inserções, mas sem atualizações ou exclusões), não há necessidade de aspirar para recuperação de espaço, então pode ser útil tentar maximizar o intervalo entre autovacuums forçados em tabelas estáticas muito grandes. Obviamente, é possível fazer isso aumentando `autovacuum_freeze_max_age` ou diminuindo `vacuum_freeze_min_age`.

O valor máximo eficaz para `vacuum_freeze_table_age` é 0,95 * `autovacuum_freeze_max_age`; um ajuste superior a esse será limitado ao máximo. Um valor superior a `autovacuum_freeze_max_age` não faria sentido, pois um autovacuo anti-wraparound seria acionado nesse ponto de qualquer maneira, e o multiplicador de 0,95 deixa um pouco de espaço para executar um `VACUUM` manual antes que isso aconteça. Como regra geral, `vacuum_freeze_table_age` deve ser ajustado a um valor um pouco abaixo de `autovacuum_freeze_max_age`, deixando espaço suficiente para que um `VACUUM` regularmente agendado ou um autovacuo acionado por atividade normal de exclusão e atualização seja executado nessa janela. Definí-lo muito próximo pode levar a autovacuos anti-wraparound, mesmo que a tabela tenha sido recentemente vacumada para recuperar espaço, enquanto valores mais baixos levam a um vácuo mais frequente e agressivo.

A única desvantagem de aumentar `autovacuum_freeze_max_age` (e `vacuum_freeze_table_age` junto com ele) é que os subdiretórios `pg_xact` e `pg_commit_ts` do conjunto de nós de banco de dados ocuparão mais espaço, porque ele deve armazenar o status do commit e (se `track_commit_timestamp` estiver habilitado) o timestamp de todas as transações até o horizonte `autovacuum_freeze_max_age`. O status do commit usa dois bits por transação, então se `autovacuum_freeze_max_age` estiver definido no seu valor máximo permitido de dois bilhões, espera-se que `pg_xact` cresça para cerca de meio gigabyte e `pg_commit_ts` para cerca de 20GB. Se isso é trivial em comparação com o tamanho total do seu banco de dados, é recomendado definir `autovacuum_freeze_max_age` no seu valor máximo permitido. Caso contrário, defina-o dependendo do que está disposto a permitir para o armazenamento de `pg_xact` e `pg_commit_ts`. (O padrão, 200 milhões de transações, se traduz em cerca de 50MB de armazenamento `pg_xact` e cerca de 2GB de armazenamento `pg_commit_ts`.)

Uma desvantagem de diminuir `vacuum_freeze_min_age` é que isso pode fazer com que `VACUUM` realize um trabalho inútil: congelar uma versão de linha é um desperdício de tempo se a linha for modificada logo em seguida (fazendo com que ela adquira um novo XID). Portanto, o ajuste deve ser grande o suficiente para que as linhas não sejam congeladas até que seja improvável que elas mudem novamente.

Para rastrear a idade dos XIDs mais antigos não congelados em um banco de dados, o `VACUUM` armazena estatísticas de XID nas tabelas do sistema `pg_class` e `pg_database`. Em particular, a coluna `relfrozenxid` da linha `pg_class` de uma tabela contém o XID mais antigo restante no final do mais recente `VACUUM` que foi avançado com sucesso `relfrozenxid` (tipicamente o mais recente VACUUM agressivo). Da mesma forma, a coluna `datfrozenxid` da linha `pg_database` de um banco de dados é um limite inferior para os XIDs não congelados que aparecem nesse banco de dados — é apenas o mínimo dos valores `relfrozenxid` por tabela dentro do banco de dados. Uma maneira conveniente de examinar essas informações é executar consultas como:

```
SELECT c.oid::regclass as table_name,
       greatest(age(c.relfrozenxid),age(t.relfrozenxid)) as age
FROM pg_class c
LEFT JOIN pg_class t ON c.reltoastrelid = t.oid
WHERE c.relkind IN ('r', 'm');

SELECT datname, age(datfrozenxid) FROM pg_database;
```

A coluna `age` mede o número de transações do XID de corte até o XID da transação atual.

### DICA

Quando o parâmetro `VERBOSE` do comando `VACUUM` é especificado, o `VACUUM` exibe várias estatísticas sobre a tabela. Isso inclui informações sobre como os `relfrozenxid` e `relminmxid` avançaram e o número de páginas recém-congeladas. Os mesmos detalhes aparecem no log do servidor quando o registro do autovacuum (controlado por [log_autovacuum_min_duration](runtime-config-logging.md#GUC-LOG-AUTOVACUUM-MIN-DURATION)) relata uma operação `VACUUM` executada pelo autovacuum.

Enquanto os `VACUUM` varrem principalmente páginas que foram modificadas desde o último vácuo, também podem varrer ávidamente algumas páginas visíveis, mas não congeladas, na tentativa de congelá-las, mas o `relfrozenxid` só será avançado quando todas as páginas da tabela que podem conter XIDs não congelados forem varridas. Isso acontece quando o `relfrozenxid` tem mais de `vacuum_freeze_table_age` transações antigas, quando a opção `VACUUM` do `FREEZE` é usada, ou quando todas as páginas que não já estão totalmente congeladas acabam por exigir vácuo para remover versões de linha morta. Quando o `VACUUM` varre todas as páginas da tabela que não estão totalmente congeladas, deve definir o `age(relfrozenxid)` para um valor um pouco mais do que o ajuste `vacuum_freeze_min_age` que foi usado (mais pelo número de transações iniciadas desde que o `VACUUM` começou). O `VACUUM` definirá o `relfrozenxid` para o XID mais antigo que permanece na tabela, então é possível que o valor final seja muito mais recente do que estritamente necessário. Se não for emitida nenhuma `relfrozenxid` avançando `VACUUM` na tabela até que o `autovacuum_freeze_max_age` seja alcançado, um autovácuo será forçado em breve para a tabela.

Se, por algum motivo, o autovacuum não conseguir limpar os XIDs antigos de uma tabela, o sistema começará a emitir mensagens de alerta como esta quando os XIDs mais antigos do banco de dados atingem quarenta milhões de transações a partir do ponto de envolvimento:

```
WARNING:  database "mydb" must be vacuumed within 39985967 transactions
HINT:  To avoid XID assignment failures, execute a database-wide VACUUM in that database.
```

(Um manual `VACUUM` deve corrigir o problema, conforme sugerido pelo aviso; mas observe que o `VACUUM` deve ser realizado por um superusuário, caso contrário, ele não conseguirá processar os catálogos do sistema, o que o impedirá de avançar o `datfrozenxid` do banco de dados.) Se esses avisos forem ignorados, o sistema se negará a atribuir novos XIDs, uma vez que restarão menos de três milhões de transações até o wraparound:

```
ERROR:  database is not accepting commands that assign new XIDs to avoid wraparound data loss in database "mydb"
HINT:  Execute a database-wide VACUUM in that database.
```

Nessa condição, as transações já em andamento podem continuar, mas apenas transações de leitura podem ser iniciadas. As operações que modificam os registros do banco de dados ou truncam relações falharão. O comando `VACUUM` ainda pode ser executado normalmente. Observe que, ao contrário do que foi recomendado em algumas versões anteriores, não é necessário ou desejável parar o postmaster ou entrar no modo de usuário único para restaurar o funcionamento normal. Em vez disso, siga estes passos:

1. Resolva transações preparadas antigas. Você pode encontrá-las verificando [pg_prepared_xacts](view-pg-prepared-xacts.md) para linhas onde (monitoring-stats.md#MONITORING-PG-STAT-ACTIVITY-VIEW "27.2.3. pg_stat_activity") é grande. Essas transações devem ser comprometidas ou desfeitas.
2. Finalize transações abertas de longa duração. Você pode encontrá-las verificando [pg_stat_activity](monitoring-stats.md#MONITORING-PG-STAT-ACTIVITY-VIEW) para linhas onde (monitoring-stats.md#MONITORING-PG-STAT-REPLICATION-VIEW "27.2.4. pg_stat_replication") ou [[PH_LNK_184]] é grande. Essas transações devem ser comprometidas ou desfeitas, ou a sessão pode ser encerrada usando [[PH_LNK_185]].
3. Descarte quaisquer slots de replicação antigos. Use [pg_stat_replication](monitoring-stats.md#MONITORING-PG-STAT-REPLICATION-VIEW) para encontrar slots onde [[PH_LNK_186]] ou [[PH_LNK_187]] é grande. Em muitos casos, esses slots foram criados para replicação a servidores que não existem mais, ou que estão fora de uso há muito tempo. Se você descartar um slot para um servidor que ainda existe e que ainda pode tentar se conectar a esse slot, essa replica pode precisar ser reconstruída.
4. Execute [[PH_LNK_188]] no banco de dados alvo. Um [[PH_LNK_189]] de nível de banco é o mais simples; para reduzir o tempo necessário, também é possível emitir comandos manuais [[PH_LNK_190]] nas tabelas onde [[PH_LNK_191]] é a mais antiga. Não use [[PH_LNK_192]] neste cenário, porque ele requer um XID e, portanto, falhará, exceto no modo de superusuário, onde, em vez disso, consumirá um XID e, assim, aumentará o risco de reencaminhamentos de ID de transação. Não use [[PH_LNK_193]] também, porque fará mais do que a quantidade mínima de trabalho necessária para restaurar o funcionamento normal.
5. Uma vez que o funcionamento normal seja restaurado, garanta que o autovacuum esteja corretamente configurado no banco de dados alvo para evitar problemas futuros.

### Nota

Em versões anteriores, às vezes era necessário parar o postmaster e a base de dados em modo de único utilizador. Em cenários típicos, isso já não é necessário e deve ser evitado sempre que possível, uma vez que envolve a desativação do sistema. Também é mais arriscado, pois desativa as salvaguardas de envolvimento de ID de transação que são projetadas para prevenir a perda de dados. A única razão para usar o modo de único utilizador neste cenário é se você deseja `TRUNCATE` ou `DROP` tabelas desnecessárias para evitar a necessidade de `VACUUM`-las. A margem de segurança de três milhões de transações existe para permitir que o administrador faça isso. Consulte a página de referência [postgres](app-postgres.md) para obter detalhes sobre o uso do modo de único utilizador.

#### 24.1.5.1. Multixacts e Wraparound [#](#VACUUM-FOR-MULTIXACT-WRAPAROUND)

Os *IDs multixact* são usados para suportar o bloqueio de linhas por múltiplas transações. Como há apenas espaço limitado no cabeçalho de uma tupla para armazenar informações de bloqueio, essas informações são codificadas como um "ID de múltiplas transações", ou ID multixact, para abreviar, sempre que houver mais de uma transação bloqueando uma linha simultaneamente. As informações sobre quais IDs de transação estão incluídos em qualquer ID multixact específica são armazenadas separadamente no subdiretório `pg_multixact`, e apenas o ID multixact aparece no campo `xmax` no cabeçalho da tupla. Assim como os IDs de transação, os IDs multixact são implementados como um contador de 32 bits e armazenamento correspondente, o que exige gerenciamento cuidadoso de envelhecimento, limpeza de armazenamento e manipulação de envolvimento. Há uma área de armazenamento separada que contém a lista de membros em cada multixact, que também usa um contador de 32 bits e que também deve ser gerenciada. A função do sistema `pg_get_multixact_members()`, descrita na [Tabela 9.84](functions-info.md#FUNCTIONS-PG-SNAPSHOT "Table 9.84. Transaction ID and Snapshot Information Functions"), pode ser usada para examinar os IDs de transação associados a um ID multixact.

Sempre que o `VACUUM` analisa qualquer parte de uma tabela, ele substituirá qualquer ID multixact que encontrar e que seja mais antiga que [vacuum_multixact_freeze_min_age](runtime-config-vacuum.md#GUC-VACUUM-MULTIXACT-FREEZE-MIN-AGE) por um valor diferente, que pode ser o valor zero, um ID de transação única ou um ID multixact mais recente. Para cada tabela, o `pg_class`.`relminmxid` armazena o ID multixact mais antigo possível que ainda apareça em qualquer tupla dessa tabela. Se esse valor for mais antigo que [vacuum_multixact_freeze_table_age](runtime-config-vacuum.md#GUC-VACUUM-MULTIXACT-FREEZE-TABLE-AGE), um vácuo agressivo é forçado. Como discutido na seção anterior, um vácuo agressivo significa que apenas as páginas que são conhecidas como totalmente congeladas serão ignoradas. O `mxid_age()` pode ser usado no `pg_class`.`relminmxid` para encontrar sua idade.

`VACUUM`s agressivos, independentemente da causa, *garantidamente* poderão avançar a tabela `relminmxid`. Eventualmente, à medida que todas as tabelas em todos os bancos de dados são verificadas e seus valores multixact mais antigos são avançados, o armazenamento em disco para multixatos mais antigos pode ser removido.

Como um dispositivo de segurança, um varredura agressiva de vácuo ocorrerá para qualquer tabela cuja idade multixact é maior que [autovacuum_multixact_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-MULTIXACT-FREEZE-MAX-AGE). Além disso, se o armazenamento ocupado por membros multixact exceder cerca de 10 GB, varreduras agressivas de vácuo ocorrerão com mais frequência para todas as tabelas, começando com aquelas que têm a idade multixact mais antiga. Ambos esses tipos de varreduras agressivas ocorrerão mesmo se o autovacuum estiver nominalmente desativado. A área de armazenamento dos membros pode crescer até cerca de 20 GB antes de atingir o wraparound.

Se, assim como no caso do XID, o autovacuum não conseguir limpar os MXIDs antigos de uma tabela, o sistema começará a emitir mensagens de alerta quando os MXIDs mais antigos do banco de dados atingem quarenta milhões de transações a partir do ponto de reencaminhamento. E, assim como no caso do XID, se essas advertências forem ignoradas, o sistema se negará a gerar novos MXIDs assim que restarem menos de três milhões até o ponto de reencaminhamento.

O funcionamento normal quando os MXIDs são esgotados pode ser restaurado da mesma maneira que quando os XIDs são esgotados. Siga os mesmos passos na seção anterior, mas com as seguintes diferenças:

As transações em execução e as transações preparadas podem ser ignoradas se não houver chance de que elas possam aparecer em uma multixact. 2. As informações MXID não são diretamente visíveis em visualizações do sistema, como `pg_stat_activity`; no entanto, procurar por XIDs antigos ainda é uma boa maneira de determinar quais transações estão causando problemas de envolvimento MXID. 3. O esgotamento do XID bloqueará todas as transações de escrita, mas o esgotamento do MXID bloqueará apenas um subconjunto de transações de escrita, especificamente aquelas que envolvem bloqueios de linha que requerem um MXID.

### 24.1.6. O daemon de autovacuo [#](#AUTOVACUUM)

O PostgreSQL tem uma característica opcional, mas altamente recomendada, chamada *autovacuum*, cujo propósito é automatizar a execução dos comandos `VACUUM` e `ANALYZE`. Quando habilitada, o autovacuum verifica tabelas que tiveram um grande número de tuplas inseridas, atualizadas ou excluídas. Esses verificações utilizam a facilidade de coleta de estatísticas; portanto, o autovacuum não pode ser usado a menos que [track_counts](runtime-config-statistics.md#GUC-TRACK-COUNTS) esteja definido como `true`. Na configuração padrão, o autovacuum está habilitado e os parâmetros de configuração relacionados estão adequadamente definidos.

O "daemon de autovacuum" na verdade é composto por vários processos. Há um processo daemon persistente, chamado *lançador de autovacuum*, que é responsável por iniciar os processos *worker de autovacuum* para todos os bancos de dados. O lançador distribuirá o trabalho ao longo do tempo, tentando iniciar um *worker* em cada banco de dados a cada [autovacuum_naptime](runtime-config-vacuum.md#GUC-AUTOVACUUM-NAPTIME) segundos. (Portanto, se a instalação tiver *`N`* bancos de dados, um novo *worker* será lançado a cada [`autovacuum_naptime`]*/*`N`* segundos.) É permitido que até [autovacuum_max_workers](runtime-config-vacuum.md#GUC-AUTOVACUUM-MAX-WORKERS) processos de *worker* sejam executados ao mesmo tempo. Se houver mais de `autovacuum_max_workers` bancos de dados a serem processados, o próximo banco de dados será processado assim que o primeiro *worker* terminar. Cada processo de *worker* verificará cada tabela dentro de seu banco de dados e executará `VACUUM` e/ou `ANALYZE` conforme necessário. [log_autovacuum_min_duration](runtime-config-logging.md#GUC-LOG-AUTOVACUUM-MIN-DURATION) pode ser configurado para monitorar a atividade dos *worker* de autovacuum.

Se várias tabelas grandes todas se tornarem elegíveis para serem limpas em um curto período de tempo, todos os trabalhadores do autovacuum podem ficar ocupados limpando essas tabelas por um longo período. Isso resultaria em outras tabelas e bancos de dados não serem limpas até que um trabalhador esteja disponível. Não há limite sobre quantos trabalhadores podem estar em um único banco de dados, mas os trabalhadores tentam evitar repetir o trabalho que já foi feito por outros trabalhadores. Note que o número de trabalhadores em execução não conta para os limites [max_connections](runtime-config-connection.md#GUC-MAX-CONNECTIONS) ou [superuser_reserved_connections](runtime-config-connection.md#GUC-SUPERUSER-RESERVED-CONNECTIONS).

As tabelas cujo valor de `relfrozenxid` é superior a [autovacuum_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-FREEZE-MAX-AGE) de transações antigas são sempre limpas (isto também se aplica às tabelas cujo máximo de idade de congelamento foi modificado através de parâmetros de armazenamento; veja abaixo). Caso contrário, se o número de tuplas obsoletas desde a última `VACUUM` exceder o "limite de limpeza", a tabela é limpa. O limite de limpeza é definido como:

```
vacuum threshold = Minimum(vacuum max threshold, vacuum base threshold + vacuum scale factor * number of tuples)
```

onde o limite máximo de vácuo é [autovacuum_vacuum_max_threshold](runtime-config-vacuum.md#GUC-AUTOVACUUM-VACUUM-MAX-THRESHOLD), o limite de base de vácuo é [autovacuum_vacuum_threshold](runtime-config-vacuum.md#GUC-AUTOVACUUM-VACUUM-THRESHOLD), o fator de escala de vácuo é [autovacuum_vacuum_scale_factor](runtime-config-vacuum.md#GUC-AUTOVACUUM-VACUUM-SCALE-FACTOR), e o número de tuplas é `pg_class`.`reltuples`.

A tabela também é aspirada se o número de tuplas inseridas desde a última varredura tiver excedido o limite de inserção definido, que é definido como:

```
vacuum insert threshold = vacuum base insert threshold + vacuum insert scale factor * number of tuples * percent of table not frozen
```

onde o limite da base de inserção de vácuo é [autovacuum_vacuum_insert_threshold](runtime-config-vacuum.md#GUC-AUTOVACUUM-VACUUM-INSERT-THRESHOLD), o fator de escala da inserção de vácuo é [autovacuum_vacuum_insert_scale_factor](runtime-config-vacuum.md#GUC-AUTOVACUUM-VACUUM-INSERT-SCALE-FACTOR), o número de tuplas é `pg_class`.[[PH_LNK_246]], e o percentual da tabela que não está congelada é `1 - pg_class.relallfrozen / pg_class.relpages`. Tais vácuos podem permitir que partes da tabela sejam marcadas como *todas visíveis* e também permitir que as tuplas sejam congeladas, o que pode reduzir o trabalho necessário em vácuos subsequentes. Para tabelas que recebem operações de `INSERT`, mas sem ou quase sem operações de `UPDATE`/`DELETE`, pode ser benéfico diminuir a [autovacuum_freeze_min_age](sql-createtable.md#RELOPTION-AUTOVACUUM-FREEZE-MIN-AGE) da tabela, pois isso pode permitir que as tuplas sejam congeladas por vácuos anteriores. O número de tuplas obsoletas e o número de tuplas inseridas são obtidos do sistema de estatísticas cumulativas; é uma contagem eventualmente consistente atualizada por cada operação de `UPDATE`, `DELETE` e `INSERT`. Se o valor de `relfrozenxid` da tabela for mais de `vacuum_freeze_table_age` transações antigas, é realizada uma limpeza agressiva para congelar as tuplas antigas e avançar `relfrozenxid`.

Para análise, uma condição semelhante é usada: o limiar, definido como:

```
analyze threshold = analyze base threshold + analyze scale factor * number of tuples
```

é comparado ao número total de tuplas inseridas, atualizadas ou excluídas desde o último `ANALYZE`.

As tabelas particionadas não armazenam tuplas diretamente e, consequentemente, não são processadas pelo autovacuum. (O autovacuum processa as partições das tabelas da mesma forma que outras tabelas.) Infelizmente, isso significa que o autovacuum não executa `ANALYZE` em tabelas particionadas, o que pode causar planos subótimos para consultas que fazem referência às estatísticas de tabelas particionadas. Você pode contornar esse problema executando manualmente `ANALYZE` em tabelas particionadas quando elas são preenchidas pela primeira vez e novamente sempre que a distribuição dos dados em suas partições mudar significativamente.

As tabelas temporárias não podem ser acessadas pelo autovacuum. Portanto, as operações adequadas de vacúmen e análise devem ser realizadas por meio de comandos SQL de sessão.

Os limiais e fatores de escala padrão são retirados de `postgresql.conf`, mas é possível sobrepu-los (e muitos outros parâmetros de controle do autovacuo) em uma base por tabela; veja [Parâmetros de Armazenamento](sql-createtable.md#SQL-CREATETABLE-STORAGE-PARAMETERS "Storage Parameters") para mais informações. Se uma configuração tiver sido alterada através dos parâmetros de armazenamento de uma tabela, esse valor é usado ao processar essa tabela; caso contrário, os ajustes globais são usados. Veja [Seção 19.10.1](runtime-config-vacuum.md#RUNTIME-CONFIG-AUTOVACUUM "19.10.1. Automatic Vacuuming") para mais detalhes sobre os ajustes globais.

Quando vários trabalhadores estão em execução, os parâmetros de atraso do autovacuum (ver [Seção 19.10.2](runtime-config-vacuum.md#RUNTIME-CONFIG-RESOURCE-VACUUM-COST)) são "equilibrados" entre todos os trabalhadores em execução, de modo que o impacto total de I/O no sistema seja o mesmo, independentemente do número de trabalhadores que estão realmente em execução. No entanto, quaisquer trabalhadores que estão processando tabelas cujos parâmetros de armazenamento por tabela `autovacuum_vacuum_cost_delay` ou `autovacuum_vacuum_cost_limit` foram definidos não são considerados no algoritmo de equilíbrio.

Os trabalhadores do Autovacuum geralmente não bloqueiam outros comandos. Se um processo tentar adquirir um bloqueio que conflita com o bloqueio `SHARE UPDATE EXCLUSIVE` mantido pelo autovacuum, a aquisição do bloqueio interromperá o autovacuum. Para modos de bloqueio conflitantes, consulte [Tabela 13.2](explicit-locking.md#TABLE-LOCK-COMPATIBILITY). No entanto, se o autovacuum estiver em execução para prevenir o reenquadramento do ID de transação (ou seja, o nome da consulta do autovacuum na visão `pg_stat_activity` termina com `(to prevent wraparound)`, o autovacuum não será interrompido automaticamente.

### Aviso

Executar regularmente comandos que adquirirem bloqueios em conflito com um bloqueio `SHARE UPDATE EXCLUSIVE` (por exemplo, ANALYZE) pode efetivamente impedir que os autovacúolos sejam concluídos.
