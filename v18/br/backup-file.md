## 25.2. Backup em nível de sistema de arquivos [#](#BACKUP-FILE)

Uma estratégia alternativa de backup é copiar diretamente os arquivos que o PostgreSQL usa para armazenar os dados no banco de dados; [Seção 18.2] explica onde esses arquivos estão localizados. Você pode usar qualquer método que prefira para fazer backups do sistema de arquivos; por exemplo:

```
tar -cf backup.tar /usr/local/pgsql/data
```

Existem, no entanto, duas restrições que tornam esse método impraticável, ou pelo menos inferior ao método pg_dump:

O servidor de banco de dados *deve* ser desligado para obter um backup utilizável. Medidas intermediárias, como a proibição de todas as conexões, *não* funcionarão (em parte porque `tar` e ferramentas semelhantes não fazem um instantâneo atômico do estado do sistema de arquivos, mas também devido ao buffer interno dentro do servidor). Informações sobre como parar o servidor podem ser encontradas em [Seção 18.5][(server-shutdown.md "18.5. Shutting Down the Server")]. Desnecessário dizer que você também precisa desligar o servidor antes de restaurar os dados.
2. Se você investigou os detalhes do layout do sistema de arquivos do banco de dados, pode ser tentado tentar fazer backup ou restaurar apenas certas tabelas ou bancos de dados individuais a partir de seus respectivos arquivos ou diretórios. Isso *não* funcionará porque as informações contidas nesses arquivos não são utilizáveis sem os arquivos de log de commit, `pg_xact/*`, que contêm o status de commit de todas as transações. Um arquivo de tabela só é utilizável com essas informações. Claro que também é impossível restaurar apenas uma tabela e os dados associados `pg_xact` porque isso tornaria todas as outras tabelas no clúster do banco de dados inúteis. Portanto, os backups do sistema de arquivos só funcionam para backup e restauração completa de um clúster de banco de dados completo.

Uma abordagem alternativa de backup de sistema de arquivos é fazer um "instantâneo consistente" do diretório de dados, se o sistema de arquivos suportar essa funcionalidade (e você estiver disposto a confiar que ela foi implementada corretamente). O procedimento típico é fazer um "instantâneo congelado" do volume que contém o banco de dados, depois copiar todo o diretório de dados (não apenas partes, veja acima) do instantâneo para um dispositivo de backup, depois liberar o instantâneo congelado. Isso funcionará mesmo enquanto o servidor de banco de dados estiver em execução. No entanto, um backup criado dessa maneira salva os arquivos do banco de dados em um estado como se o servidor de banco de dados não tivesse sido desligado corretamente; portanto, quando você iniciar o servidor de banco de dados nos dados de backup, ele pensará que a instância anterior do servidor quebrou e irá reproduzir o log WAL. Isso não é um problema; apenas esteja ciente disso (e certifique-se de incluir os arquivos WAL em seu backup). Você pode realizar um `CHECKPOINT` antes de fazer o instantâneo para reduzir o tempo de recuperação.

Se o seu banco de dados estiver espalhado por vários sistemas de arquivos, talvez não haja nenhuma maneira de obter instantâneos congelados exatamente simultâneos de todos os volumes. Por exemplo, se seus arquivos de dados e o log WAL estiverem em discos diferentes, ou se os espaços de tabelas estiverem em diferentes sistemas de arquivos, talvez não seja possível usar o backup de instantâneo porque os instantâneos *devem* ser simultâneos. Leia a documentação do seu sistema de arquivos com muito cuidado antes de confiar na técnica de instantâneo consistente nessas situações.

Se instantâneos simultâneos não forem possíveis, uma opção é desligar o servidor de banco de dados por um período suficiente para estabelecer todos os instantâneos congelados. Outra opção é realizar um backup de arquivamento contínuo ([Seção 25.3.2][(continuous-archiving.md#BACKUP-BASE-BACKUP "25.3.2. Making a Base Backup")]) porque tais backups são imunes a alterações no sistema de arquivos durante o backup. Isso requer a habilitação do arquivamento contínuo apenas durante o processo de backup; o restabelecimento é feito usando a recuperação de arquivo contínuo ([Seção 25.3.5][(continuous-archiving.md#BACKUP-PITR-RECOVERY "25.3.5. Recovering Using a Continuous Archive Backup")]).

Outra opção é usar o rsync para realizar um backup do sistema de arquivos. Isso é feito executando o rsync primeiro enquanto o servidor de banco de dados está em execução, e depois desligando o servidor de banco de dados o tempo suficiente para realizar um `rsync --checksum`. (O `--checksum` é necessário porque o `rsync` tem apenas granularidade de tempo de modificação de arquivo de um segundo.) O segundo rsync será mais rápido que o primeiro, porque tem relativamente pouca dados para transferir, e o resultado final será consistente porque o servidor estava fora de operação. Esse método permite que um backup do sistema de arquivos seja realizado com o menor tempo de inatividade possível.

Observe que um backup do sistema de arquivos geralmente será maior do que um dump SQL. (pg_dump, por exemplo, não precisa fazer o dump do conteúdo dos índices, apenas os comandos para recriá-los.) No entanto, fazer um backup do sistema de arquivos pode ser mais rápido.
