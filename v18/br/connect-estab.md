## 51.2. Como as Conexões São Estabelecidas [#](#CONNECT-ESTAB)

O PostgreSQL implementa um modelo de cliente/servidor "processo por usuário". Nesse modelo, cada [*[processo de cliente](glossary.md#GLOSSARY-CLIENT "Client (process)")*](glossary.md#GLOSSARY-CLIENT) se conecta exatamente a um [*[processo de backend](glossary.md#GLOSSARY-BACKEND "Backend (process)")*](glossary.md#GLOSSARY-BACKEND). Como não sabemos antecipadamente quantas conexões serão feitas, temos que usar um "processo supervisor" que gera um novo processo de backend a cada vez que uma conexão é solicitada. Esse processo supervisor é chamado [*[postmaster](glossary.md#GLOSSARY-POSTMASTER "Postmaster (process)")*](glossary.md#GLOSSARY-POSTMASTER) e escuta em uma porta TCP/IP especificada para conexões recebidas. Sempre que detecta uma solicitação de conexão, gera um novo processo de backend. Esses processos de backend se comunicam entre si e com outros processos da [*[instância](glossary.md#GLOSSARY-INSTANCE "Instance")*](glossary.md#GLOSSARY-INSTANCE) usando *semaforos* e [*[memória compartilhada](glossary.md#GLOSSARY-SHARED-MEMORY "Shared memory")*](glossary.md#GLOSSARY-SHARED-MEMORY) para garantir a integridade dos dados durante o acesso concorrente aos dados.

O processo do cliente pode ser qualquer programa que entenda o protocolo PostgreSQL descrito em [Capítulo 54][(protocol.md "Chapter 54. Frontend/Backend Protocol")]. Muitos clientes são baseados na biblioteca de linguagem C libpq, mas existem várias implementações independentes do protocolo, como o driver JDBC do Java.

Uma vez que uma conexão é estabelecida, o processo cliente pode enviar uma consulta ao processo de backend com o qual está conectado. A consulta é transmitida usando texto simples, ou seja, não há análise realizada no cliente. O processo de backend analisa a consulta, cria um *plano de execução*, executa o plano e retorna as linhas recuperadas ao cliente, transmitindo-as pela conexão estabelecida.
