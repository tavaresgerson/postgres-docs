## 28.1. Confiabilidade [#](#WAL-RELIABILITY)

A confiabilidade é uma propriedade importante de qualquer sistema de banco de dados sério, e o PostgreSQL faz tudo o que é possível para garantir o funcionamento confiável. Um aspecto do funcionamento confiável é que todos os dados registrados por uma transação comprometida devem ser armazenados em uma área não volátil que esteja segura contra perda de energia, falha do sistema operacional e falha de hardware (exceto, claro, a falha da própria área não volátil). Escrever com sucesso os dados no armazenamento permanente do computador (unidade de disco ou equivalente) normalmente atende a esse requisito. De fato, mesmo que um computador seja fatalmente danificado, se as unidades de disco sobreviverem, elas podem ser movidas para outro computador com hardware semelhante e todas as transações comprometidas permanecerão intactas.

Embora forçar dados nos pratos do disco periodicamente possa parecer uma operação simples, não é. Como as unidades de disco são significativamente mais lentas que a memória principal e as CPUs, existem várias camadas de cache entre a memória principal do computador e os pratos do disco. Primeiro, há o cache de buffer do sistema operacional, que cache blocos de disco frequentemente solicitados e combina escritas no disco. Felizmente, todos os sistemas operacionais fornecem às aplicações uma maneira de forçar escritas do cache de buffer para o disco, e o PostgreSQL usa essas funcionalidades. (Veja o parâmetro [wal_sync_method][(runtime-config-wal.md#GUC-WAL-SYNC-METHOD)] para ajustar como isso é feito.)

Em seguida, pode haver um cache no controlador da unidade de disco; isso é particularmente comum em cartões de controlador RAID. Algumas dessas caches são *de passagem*, o que significa que as escritas são enviadas para a unidade assim que chegam. Outras são *de retorno*, o que significa que os dados são enviados para a unidade em algum momento posterior. Tais caches podem ser um risco de confiabilidade, pois a memória do cache do controlador de disco é volátil e perderá seu conteúdo em caso de falha de energia. Cartões de controlador melhores têm *unidades de backup de bateria* (BBUs), o que significa que o cartão tem uma bateria que mantém a energia para o cache em caso de perda de energia do sistema. Após a restauração da energia, os dados serão escritos nas unidades de disco.

E, finalmente, a maioria das unidades de disco tem caches. Algumas são de escrita direta, enquanto outras são de escrita indireta, e as mesmas preocupações sobre perda de dados existem para caches de drives de escrita indireta, assim como para caches de controladores de disco. Unidades IDE e SATA de nível de consumo têm uma probabilidade particularmente alta de ter caches de escrita indireta que não sobreviverão a uma falha de energia. Muitas unidades de estado sólido (SSD) também têm caches de escrita voláteis de escrita indireta.

Normalmente, esses caches podem ser desativados; no entanto, o método para fazer isso varia de acordo com o sistema operacional e o tipo de unidade:

* Em Linux, as unidades IDE e SATA podem ser consultadas usando `hdparm -I`; o cache de escrita é habilitado se houver um `*` ao lado de `Write cache`. `hdparm -W 0` pode ser usado para desativar o cache de escrita. Unidades SCSI podem ser consultadas usando [sdparm](http://sg.danny.cz/sg/sdparm.html). Use `sdparm --get=WCE` para verificar se o cache de escrita está habilitado e `sdparm --clear=WCE` para desabilitá-lo.
* Em FreeBSD, as unidades IDE podem ser consultadas usando `camcontrol identify` e o cache de escrita desativado usando `hw.ata.wc=0` em `/boot/loader.conf`; unidades SCSI podem ser consultadas usando `camcontrol identify`, e o cache de escrita pode ser tanto consultado quanto alterado usando `sdparm` quando disponível.
* Em Solaris, o cache de escrita do disco é controlado por `format -e`. (O sistema de arquivos ZFS de Solaris é seguro com cache de escrita do disco habilitado, pois emite seus próprios comandos de limpeza de cache de disco.)
* Em Windows, se `wal_sync_method` for `open_datasync` (o padrão), o cache de escrita pode ser desativado desmarcando `My Computer\Open\disk drive\Properties\Hardware\Properties\Policies\Enable write caching on the disk`. Alternativamente, defina `wal_sync_method` para `fdatasync` (apenas NTFS) ou `fsync`, que impedem o cache de escrita.
* Em macOS, o cache de escrita pode ser impedido definindo `wal_sync_method` para `fsync_writethrough`.

As unidades SATA recentes (as que seguem o ATAPI-6 ou posteriores) oferecem um comando de esvaziamento de cache de unidade (`FLUSH CACHE EXT`), enquanto as unidades SCSI têm suporte há muito tempo para um comando semelhante `SYNCHRONIZE CACHE`. Esses comandos não são diretamente acessíveis ao PostgreSQL, mas alguns sistemas de arquivos (por exemplo, ZFS, ext4) podem usá-los para esvaziar dados para os pratos em unidades com suporte de retorno de escrita. Infelizmente, esses sistemas de arquivos se comportam de forma subótima quando combinados com controladores de disco de unidade de backup por bateria (BBU). Em tais configurações, o comando de sincronização força todos os dados da cache do controlador para os discos, eliminando grande parte do benefício da BBU. Você pode executar o programa [pg_test_fsync](pgtestfsync.md "pg_test_fsync") para verificar se você está afetado. Se estiver afetado, os benefícios de desempenho da BBU podem ser recuperados desligando as barreiras de escrita no sistema de arquivos ou reconfigurando o controlador de disco, se essa for uma opção. Se as barreiras de escrita forem desligadas, certifique-se de que a bateria permaneça funcional; uma bateria defeituosa pode potencialmente levar à perda de dados. Esperamos que os projetistas de sistemas de arquivos e controladores de disco resolvam eventualmente esse comportamento subótimo.

Quando o sistema operacional envia uma solicitação de escrita para o hardware de armazenamento, há pouco que ele pode fazer para garantir que os dados tenham chegado a uma área de armazenamento verdadeiramente não volátil. Em vez disso, é responsabilidade do administrador garantir que todos os componentes de armazenamento garantam a integridade tanto dos dados quanto dos metadados do sistema de arquivos. Evite controladores de disco que tenham caches de escrita não suportados por bateria. No nível do disco, desative o cache de escrita-de-retorno se o disco não puder garantir que os dados serão escritos antes do desligamento. Se você usa SSDs, esteja ciente de que muitos desses não respeitam comandos de esvaziamento de cache por padrão. Você pode testar o comportamento confiável do subsistema de E/S usando [[`diskchecker.pl`][(https://brad.livejournal.com/2116715.html)]].

Outro risco de perda de dados é representado pelas próprias operações de gravação em discos. Os discos são divididos em setores, geralmente de 512 bytes cada. Cada operação de leitura ou escrita física processa um setor inteiro. Quando uma solicitação de escrita chega à unidade, pode ser para algum múltiplo de 512 bytes (o PostgreSQL normalmente escreve 8192 bytes, ou 16 setores, de uma vez), e o processo de escrita pode falhar a qualquer momento, o que significa que alguns dos setores de 512 bytes foram escritos enquanto outros não foram. Para se proteger contra tais falhas, o PostgreSQL escreve periodicamente imagens de página inteira no armazenamento permanente WAL * antes * de modificar a página real no disco. Ao fazer isso, durante a recuperação em caso de falha, o PostgreSQL pode restaurar páginas parcialmente escritas do WAL. Se você tem um software de sistema de arquivos que impede escritas de página parcial (por exemplo, ZFS), pode desativar essa imagem de página ao desativar o parâmetro [full_page_writes][(runtime-config-wal.md#GUC-FULL-PAGE-WRITES)]. Os controladores de disco de unidade alimentada por bateria (BBU) não impedem escritas de página parcial, a menos que garantam que os dados sejam escritos no BBU como páginas completas (8 kB).

O PostgreSQL também protege contra alguns tipos de corrupção de dados em dispositivos de armazenamento que podem ocorrer devido a erros de hardware ou falha de mídia ao longo do tempo, como leitura/escrita de dados lixo.

* Cada registro individual em um arquivo WAL é protegido por um CRC-32C (32 bits) que nos permite dizer se o conteúdo do registro está correto. O valor CRC é definido quando escrevemos cada registro WAL e verificado durante a recuperação em caso de falha, recuperação de arquivo e replicação.
* As páginas de dados são verificadas por padrão, e as imagens de página completa gravadas em registros WAL são sempre protegidas por CRC.
* As estruturas de dados internas, como `pg_xact`, `pg_subtrans`, `pg_multixact`, `pg_serial`, `pg_notify`, `pg_stat`, `pg_snapshots` não são verificadas diretamente, nem as páginas protegidas por gravações de página completa. No entanto, onde essas estruturas de dados são persistentes, os registros WAL são escritos para permitir que as mudanças recentes sejam reconstruídas com precisão na recuperação em caso de falha e esses registros WAL são protegidos como discutido acima.
* Os arquivos de estado individuais em `pg_twophase` são protegidos por CRC-32C.
* Os arquivos de dados temporários usados em consultas SQL maiores para ordenamentos, materializações e resultados intermediários atualmente não são verificados por CRC, nem os registros WAL serão escritos para mudanças nesses arquivos.

O PostgreSQL não protege contra erros de memória corrigíveis e se assume que você operará usando RAM que utiliza Códigos de Correção de Erros (ECC) padrão da indústria ou uma proteção melhor.
