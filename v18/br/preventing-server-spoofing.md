## 18.7. Prevenção de falsificação de servidor [#](#PREVENTING-SERVER-SPOOFING)

Enquanto o servidor estiver em funcionamento, não é possível que um usuário malicioso substitua o servidor normal de banco de dados. No entanto, quando o servidor estiver fora de operação, é possível que um usuário local se aproprie do servidor normal, iniciando seu próprio servidor. O servidor falsificado pode ler senhas e consultas enviadas pelos clientes, mas não pode retornar nenhum dado, porque o diretório `PGDATA` ainda estará seguro devido às permissões de diretório. O falsificado é possível porque qualquer usuário pode iniciar um servidor de banco de dados; um cliente não pode identificar um servidor inválido, a menos que seja especialmente configurado.

Uma maneira de prevenir a falsificação das conexões de `local` é usar um diretório de socket de domínio Unix ([unix_socket_directories](runtime-config-connection.md#GUC-UNIX-SOCKET-DIRECTORIES)) que tenha permissão de escrita apenas para um usuário local de confiança. Isso impede que um usuário malicioso crie seu próprio arquivo de socket nesse diretório. Se você está preocupado que algumas aplicações ainda possam fazer referência a `/tmp` para o arquivo de socket e, portanto, serem vulneráveis à falsificação, durante o arranque do sistema operacional, crie um link simbólico `/tmp/.s.PGSQL.5432` que aponte para o arquivo de socket realocado. Você também pode precisar modificar seu script de limpeza de `/tmp` para evitar a remoção do link simbólico.

Outra opção para conexões `local` é que os clientes usem `requirepeer`(libpq-connect.md#LIBPQ-CONNECT-REQUIREPEER) para especificar o proprietário necessário do processo do servidor conectado ao soquete.

Para evitar o spoofing em conexões TCP, use certificados SSL e certifique-se de que os clientes verificam o certificado do servidor, ou use criptografia GSSAPI (ou ambos, se estiverem em conexões separadas).

Para evitar spoofing com SSL, o servidor deve ser configurado para aceitar apenas conexões `hostssl` ([Seção 20.1](auth-pg-hba-conf.md "20.1. The pg_hba.conf File")) e ter arquivos de chave e certificado SSL ([Seção 18.9](ssl-tcp.md "18.9. Secure TCP/IP Connections with SSL")). O cliente TCP deve se conectar usando `sslmode=verify-ca` ou `verify-full` e ter o arquivo de certificado raiz apropriado instalado ([Seção 32.19.1](libpq-ssl.md#LIBQ-SSL-CERTIFICATES "32.19.1. Client Verification of Server Certificates")). Alternativamente, o [pool de CA do sistema](libpq-connect.md#LIBPQ-CONNECT-SSLROOTCERT), conforme definido pela implementação SSL, pode ser usado usando `sslrootcert=system`; nesse caso, `sslmode=verify-full` é forçado por segurança, uma vez que é geralmente trivial obter certificados que são assinados por uma CA pública.

Para evitar que o spoofing de servidor ocorra ao usar autenticação com senha [scram-sha-256][(auth-password.md "20.5. Password Authentication")] em uma rede, você deve garantir que se conecte ao servidor usando SSL e com um dos métodos anti-spoofing descritos no parágrafo anterior. Além disso, a implementação SCRAM em libpq não pode proteger toda a troca de autenticação, mas usar o parâmetro de conexão `channel_binding=require` fornece uma mitigação contra o spoofing de servidor. Um atacante que usa um servidor falso para interceptar uma troca SCRAM pode usar análise off-line para potencialmente determinar a senha criptografada do cliente.

Para evitar o spoofing com GSSAPI, o servidor deve ser configurado para aceitar apenas conexões `hostgssenc` ([Seção 20.1](auth-pg-hba-conf.md "20.1. The pg_hba.conf File")) e usar autenticação `gss` com elas. O cliente TCP deve se conectar usando `gssencmode=require`.
