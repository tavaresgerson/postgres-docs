## 68.1. Regras de Declaração do Catálogo do Sistema [#](#SYSTEM-CATALOG-DECLARATIONS)

A parte principal de um arquivo de cabeçalho de catálogo é uma definição de estrutura C que descreve o layout de cada linha do catálogo. Isso começa com uma macro `CATALOG`, que, no que diz respeito ao compilador C, é apenas uma abreviação para `typedef struct FormData_catalogname`. Cada campo na estrutura dá origem a uma coluna do catálogo. Os campos podem ser anotados usando as macros de propriedade BKI descritas em `genbki.h`, por exemplo, para definir um valor padrão para um campo ou marcá-lo como nulo ou não nulo. A linha `CATALOG` também pode ser anotada, com algumas outras macros de propriedade BKI descritas em `genbki.h`, para definir outras propriedades do catálogo como um todo, como se ele é uma relação compartilhada.

O código do cache do catálogo do sistema (e a maioria dos códigos de manipulação de catálogo em geral) assume que as porções de comprimento fixo de todos os tuplos do catálogo do sistema estão de fato presentes, porque mapeia essa declaração de estrutura C neles. Assim, todos os campos de comprimento variável e campos nulos devem ser colocados no final, e não podem ser acessados como campos de estrutura. Por exemplo, se você tentasse definir `pg_type`.`typrelid` para ser NULL, falharia quando algum pedaço de código tentasse referenciar `typetup->typrelid` (ou pior, `typetup->typelem`, porque isso segue `typrelid`). Isso resultaria em erros aleatórios ou até mesmo violações de segmentação.

Como uma proteção parcial contra esse tipo de erro, campos de comprimento variável ou nulos não devem ser tornados diretamente visíveis para o compilador C. Isso é feito envolvendo-os em `#ifdef CATALOG_VARLEN`...`#endif` (onde `CATALOG_VARLEN` é um símbolo que nunca é definido). Isso impede que o código C tente descuidadamente acessar campos que talvez não estejam lá ou que estejam em algum outro deslocamento. Como uma proteção independente contra a criação de linhas incorretas, exigimos que todas as colunas que devem ser não nulos sejam marcadas assim em `pg_attribute`. O código de bootstrap marcará automaticamente as colunas do catálogo como `NOT NULL` se elas tiverem largura fixa e não forem precedidas por nenhuma coluna nulo ou de largura variável. Onde essa regra é inadequada, você pode forçar a marcação correta usando as anotações `BKI_FORCE_NOT_NULL` e `BKI_FORCE_NULL` conforme necessário.

O código do frontend não deve incluir nenhum arquivo de cabeçalho do catálogo `pg_xxx.h`, pois esses arquivos podem conter código C que não será compilado fora do backend. (Tipicamente, isso acontece porque esses arquivos também contêm declarações para funções nos arquivos `src/backend/catalog/`. Em vez disso, o código do frontend pode incluir o cabeçalho correspondente gerado `pg_xxx_d.h`, que conterá OID `#define` e qualquer outro dado que possa ser útil do lado do cliente. Se você deseja que macros ou outro código em um cabeçalho de catálogo seja visível para o código do frontend, escreva `#ifdef EXPOSE_TO_CLIENT_CODE`...`#endif` ao redor daquela seção para instruir o `genbki.pl` a copiar essa seção para o cabeçalho `pg_xxx_d.h`.

Alguns dos catálogos são tão fundamentais que nem mesmo o comando BKI `create` pode criá-los, que é o utilizado para a maioria dos catálogos, porque esse comando precisa escrever informações nesses catálogos para descrever o novo catálogo. Esses são chamados de catálogos *bootstrap*, e definir um deles exige muito trabalho extra: você precisa preparar manualmente as entradas apropriadas para eles nos conteúdos pré-carregados de `pg_class` e `pg_type`, e essas entradas precisarão ser atualizadas para mudanças subsequentes na estrutura do catálogo. (Os catálogos bootstrap também precisam de entradas pré-carregadas em `pg_attribute`, mas, felizmente, `genbki.pl` lida com essa tarefa hoje em dia.) Evite criar novos catálogos como bootstrap, se possível.
