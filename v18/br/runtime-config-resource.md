### 19.4.1. Memória [#](#RUNTIME-CONFIG-RESOURCE-MEMORY)

`shared_buffers` (`integer`) [#](#GUC-SHARED-BUFFERS): Define a quantidade de memória que o servidor de banco de dados utiliza para buffers de memória compartilhada. O padrão é tipicamente 128 megabytes (`128MB`, mas pode ser menor se as configurações do seu kernel não o suportarão (determinado durante o initdb). Este ajuste deve ser pelo menos 128 kilobytes. No entanto, configurações significativamente mais altas do que o mínimo geralmente são necessárias para um bom desempenho. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. (Valores não padrão de `BLCKSZ` alteram o valor mínimo.) Este parâmetro só pode ser definido no início do servidor.

Se você tiver um servidor de banco de dados dedicado com 1 GB ou mais de RAM, um valor inicial razoável para `shared_buffers` é de 25% da memória do seu sistema. Há algumas cargas de trabalho onde configurações ainda maiores para `shared_buffers` são eficazes, mas, como o PostgreSQL também depende do cache do sistema operacional, é improvável que uma alocação de mais de 40% de RAM para `shared_buffers` funcione melhor do que uma quantidade menor. Configurações maiores para `shared_buffers` geralmente exigem um aumento correspondente em `max_wal_size`, a fim de espalhar o processo de escrita de grandes quantidades de dados novos ou alterados em um período de tempo mais longo.

Em sistemas com menos de 1 GB de RAM, uma porcentagem menor de RAM é apropriada, para deixar espaço adequado para o sistema operacional.

`huge_pages` (`enum`) [#](#GUC-HUGE-PAGES): Controla se páginas enormes são solicitadas para a área principal de memória compartilhada. Os valores válidos são `try` (o padrão), `on` e `off`. Este parâmetro só pode ser definido no início do servidor. Com `huge_pages` definido como `try`, o servidor tentará solicitar páginas enormes, mas voltará ao padrão se isso falhar. Com `on`, a falha em solicitar páginas enormes impedirá o início do servidor. Com `off`, páginas enormes não serão solicitadas. O estado real das páginas enormes é indicado pela variável do servidor [huge_pages_status](runtime-config-preset.md#GUC-HUGE-PAGES-STATUS).

Atualmente, essa configuração é suportada apenas no Linux e no Windows. A configuração é ignorada em outros sistemas quando definida como `try`. No Linux, ela só é suportada quando `shared_memory_type` é definida como `mmap` (o padrão).

O uso de páginas enormes resulta em tabelas de página menores e menos tempo de CPU gasto na gestão de memória, aumentando o desempenho. Para mais detalhes sobre o uso de páginas enormes no Linux, consulte [Seção 18.4.5][(kernel-resources.md#LINUX-HUGE-PAGES "18.4.5. Linux Huge Pages")].

Páginas enormes são conhecidas como páginas grandes no Windows. Para usá-las, você precisa atribuir o direito do usuário “Bloquear páginas em memória” à conta de usuário do Windows que executa o PostgreSQL. Você pode usar a ferramenta de Diretiva de Grupo do Windows (gpedit.msc) para atribuir o direito do usuário “Bloquear páginas em memória”. Para iniciar o servidor de banco de dados no prompt de comando como um processo autônomo, e não como um serviço do Windows, o prompt de comando deve ser executado como administrador ou o Controle de Acesso do Usuário (UAC) deve ser desativado. Quando o UAC está habilitado, o prompt de comando normal revoga o direito do usuário “Bloquear páginas em memória” quando iniciado.

Observe que essa configuração afeta apenas a área principal da memória compartilhada. Sistemas operacionais como Linux, FreeBSD e Illumos também podem usar páginas enormes (também conhecidas como páginas "super" ou páginas "grandes") automaticamente para alocação de memória normal, sem uma solicitação explícita do PostgreSQL. Em Linux, isso é chamado de "páginas enormes transparentes" (THP). Esse recurso é conhecido por causar degradação de desempenho com o PostgreSQL para alguns usuários em algumas versões do Linux, portanto, seu uso é atualmente desencorajado (ao contrário do uso explícito de `huge_pages`).

`huge_page_size` (`integer`) [#](#GUC-HUGE-PAGE-SIZE): Controla o tamanho das páginas enormes, quando elas são habilitadas com [huge_pages](runtime-config-resource.md#GUC-HUGE-PAGES). O padrão é zero (`0`). Quando definido como `0`, o tamanho padrão da página enorme do sistema será usado. Este parâmetro só pode ser definido no início do servidor.

Alguns tamanhos de página comumente disponíveis em arquiteturas modernas de servidores de 64 bits incluem: `2MB` e `1GB` (Intel e AMD), `16MB` e `16GB` (IBM POWER) e `64kB`, `2MB`, `32MB` e `1GB` (ARM). Para mais informações sobre uso e suporte, consulte [Seção 18.4.5][(kernel-resources.md#LINUX-HUGE-PAGES "18.4.5. Linux Huge Pages")].

As configurações não padrão são atualmente suportadas apenas no Linux.

`temp_buffers` (`integer`) [#](#GUC-TEMP-BUFFERS): Define a quantidade máxima de memória usada para buffers temporários dentro de cada sessão do banco de dados. Esses são buffers locais da sessão usados apenas para acesso a tabelas temporárias. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. O padrão é oito megabytes (`8MB`). (Se `BLCKSZ` não for 8kB, o valor padrão escala proporcionalmente a ele.) Este ajuste pode ser alterado dentro de sessões individuais, mas apenas antes do primeiro uso de tabelas temporárias dentro da sessão; tentativas subsequentes de alterar o valor não terão efeito nessa sessão.

Uma sessão alocará tampões temporários conforme necessário, até o limite dado por `temp_buffers`. O custo de definir um valor grande em sessões que não precisam realmente de muitos tampões temporários é apenas um descriptor de buffer, ou cerca de 64 bytes, por incremento em `temp_buffers`. No entanto, se um buffer é realmente usado, serão consumidos 8192 bytes adicionais para ele (ou, em geral, `BLCKSZ` bytes).

`max_prepared_transactions` (`integer`) [#](#GUC-MAX-PREPARED-TRANSACTIONS): Define o número máximo de transações que podem estar no estado "preparado" simultaneamente (ver [PREPARAR TRANSAÇÃO](sql-prepare-transaction.md "PREPARE TRANSACTION")). Definir este parâmetro como zero (que é o padrão) desativa o recurso de transação preparada. Este parâmetro só pode ser definido no início do servidor.

Se você não planeja usar transações preparadas, este parâmetro deve ser definido como zero para evitar a criação acidental de transações preparadas. Se você estiver usando transações preparadas, provavelmente você deseja que `max_prepared_transactions` seja pelo menos tão grande quanto [max_connections][(runtime-config-connection.md#GUC-MAX-CONNECTIONS)], para que cada sessão possa ter uma transação preparada pendente.

Ao executar um servidor de espera, você deve definir esse parâmetro no mesmo valor ou superior ao do servidor primário. Caso contrário, as consultas não serão permitidas no servidor de espera.

`work_mem` (`integer`) [#](#GUC-WORK-MEM): Define o valor máximo de memória base a ser utilizado por uma operação de consulta (como uma ordenação ou tabela hash) antes de gravar em arquivos de disco temporários. Se este valor for especificado sem unidades, ele é considerado em kilobytes. O valor padrão é quatro megabytes (`4MB`). Observe que uma consulta complexa pode realizar várias operações de ordenação e hash ao mesmo tempo, com cada operação geralmente sendo permitida para utilizar tanto espaço de memória quanto este valor especifica antes de começar a gravar dados em arquivos temporários. Além disso, várias sessões em execução podem estar realizando tais operações de forma concorrente. Portanto, a memória total utilizada pode ser muitas vezes o valor de `work_mem`; é necessário ter isso em mente ao escolher o valor. As operações de ordenação são utilizadas para `ORDER BY`, `DISTINCT` e junções de fusão. Tabelas hash são utilizadas em junções hash, agregação baseada em hash, memorizar nós e processamento baseado em hash de subconsultas `IN`.

As operações baseadas em hash são, em geral, mais sensíveis à disponibilidade de memória do que as operações equivalentes baseadas em classificação. O limite de memória para uma tabela de hash é calculado multiplicando `work_mem` por `hash_mem_multiplier`. Isso permite que as operações baseadas em hash utilizem uma quantidade de memória que excede o valor usual da base `work_mem`.

`hash_mem_multiplier` (`floating point`) [#](#GUC-HASH-MEM-MULTIPLIER): Usado para calcular o máximo de memória que operações baseadas em hash podem usar. O limite final é determinado multiplicando `work_mem` por `hash_mem_multiplier`. O valor padrão é 2,0, o que faz com que as operações baseadas em hash usem o dobro do valor base usual de `work_mem`.

Considere aumentar `hash_mem_multiplier` em ambientes onde a derrapagem por operações de consulta é uma ocorrência regular, especialmente quando simplesmente aumentar `work_mem` resulta em pressão de memória (a pressão de memória geralmente assume a forma de erros intermitentes de falta de memória). O ajuste padrão de 2,0 é frequentemente eficaz com cargas de trabalho mistas. Configurações mais altas na faixa de 2,0 - 8,0 ou mais podem ser eficazes em ambientes onde `work_mem` já foi aumentado para 40 MB ou mais.

`maintenance_work_mem` (`integer`) [#](#GUC-MAINTENANCE-WORK-MEM): Especifica a quantidade máxima de memória a ser usada por operações de manutenção, como `VACUUM`, `CREATE INDEX` e `ALTER TABLE ADD FOREIGN KEY`. Se este valor for especificado sem unidades, ele é considerado em kilobytes. O padrão é de 64 megabytes (`64MB`). Como apenas uma dessas operações pode ser executada por sessão de banco de dados de cada vez, e uma instalação normalmente não tem muitas delas em execução simultaneamente, é seguro definir este valor significativamente maior que `work_mem`. Configurações maiores podem melhorar o desempenho para a limpeza e para a restauração de dumps de banco de dados.

Observe que, quando o autovacuum é executado, até [autovacuum_max_workers][(runtime-config-vacuum.md#GUC-AUTOVACUUM-MAX-WORKERS)] vezes essa memória pode ser alocada, então tenha cuidado para não definir o valor padrão muito alto. Pode ser útil controlar isso definindo separadamente [autovacuum_work_mem][(runtime-config-resource.md#GUC-AUTOVACUUM-WORK-MEM)].

`autovacuum_work_mem` (`integer`) [#](#GUC-AUTOVACUUM-WORK-MEM): Especifica o valor máximo de memória a ser utilizado por cada processo de trabalho do autovacuum. Se este valor for especificado sem unidades, ele é considerado em kilobytes. O valor padrão é -1, indicando que o valor de [maintenance_work_mem](runtime-config-resource.md#GUC-MAINTENANCE-WORK-MEM) deve ser utilizado em vez disso. O ajuste não tem efeito no comportamento de `VACUUM` quando executado em outros contextos. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`vacuum_buffer_usage_limit` (`integer`) [#](#GUC-VACUUM-BUFFER-USAGE-LIMIT): Especifica o tamanho do [*[Buffer Access Strategy](glossary.md#GLOSSARY-BUFFER-ACCESS-STRATEGY "Buffer Access Strategy")*](glossário.md#GLOSSARY-BUFFER-ACCESS-STRATEGY) utilizado pelos comandos `VACUUM` e `ANALYZE`. Uma configuração de `0` permitirá que a operação use qualquer número de `shared_buffers`. Caso contrário, os tamanhos válidos variam de `128 kB` a `16 GB`. Se o tamanho especificado exceder 1/8 do tamanho de `shared_buffers`, o tamanho é silenciosamente limitado a esse valor. O valor padrão é `2MB`. Se este valor for especificado sem unidades, ele é considerado em kilobytes. Este parâmetro pode ser definido a qualquer momento. Pode ser sobrescrito para [VACUUM](sql-vacuum.md "VACUUM") e [ANALYZE](sql-analyze.md "ANALYZE") ao passar a opção `BUFFER_USAGE_LIMIT`. Configurações mais altas podem permitir que `VACUUM` e `ANALYZE` sejam executados mais rapidamente, mas ter uma configuração muito grande pode causar que muitas outras páginas úteis sejam expulsas dos buffers compartilhados.

`logical_decoding_work_mem` (`integer`) [#](#GUC-LOGICAL-DECODING-WORK-MEM): Especifica a quantidade máxima de memória a ser usada pelo decodificamento lógico, antes que algumas das alterações decodificadas sejam escritas no disco local. Isso limita a quantidade de memória usada pelas conexões de replicação de fluxo lógico. O valor padrão é de 64 megabytes (`64MB`). Como cada conexão de replicação usa apenas um único buffer desse tamanho, e uma instalação normalmente não tem muitas conexões desse tipo simultaneamente (como limitado por `max_wal_senders`), é seguro definir esse valor significativamente maior que `work_mem`, reduzindo a quantidade de alterações decodificadas escritas no disco.

`commit_timestamp_buffers` (`integer`) [#](#GUC-COMMIT-TIMESTAMP-BUFFERS): Especifica a quantidade de memória a ser usada para cachear o conteúdo de `pg_commit_ts` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `0`, que solicita [[`shared_buffers`]/512]] até 1024 blocos, mas não menos de 16 blocos. Este parâmetro só pode ser definido no início do servidor.

`multixact_member_buffers` (`integer`) [#](#GUC-MULTIXACT-MEMBER-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_multixact/members` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `32`. Este parâmetro só pode ser definido no início do servidor.

`multixact_offset_buffers` (`integer`) [#](#GUC-MULTIXACT-OFFSET-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_multixact/offsets` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `16`. Este parâmetro só pode ser definido no início do servidor.

`notify_buffers` (`integer`) [#](#GUC-NOTIFY-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_notify` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `16`. Este parâmetro só pode ser definido no início do servidor.

`serializable_buffers` (`integer`) [#](#GUC-SERIALIZABLE-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_serial` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `32`. Este parâmetro só pode ser definido no início do servidor.

`subtransaction_buffers` (`integer`) [#](#GUC-SUBTRANSACTION-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_subtrans` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `0`, que solicita `shared_buffers`/512 até 1024 blocos, mas não menos de 16 blocos. Este parâmetro só pode ser definido no início do servidor.

`transaction_buffers` (`integer`) [#](#GUC-TRANSACTION-BUFFERS): Especifica a quantidade de memória compartilhada a ser usada para cachear o conteúdo de `pg_xact` (ver [Tabela 66.1] (storage-file-layout.md#PGDATA-CONTENTS-TABLE "Table 66.1. Contents of PGDATA")). Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8 kB. O valor padrão é `0`, que solicita [[`shared_buffers`]/512]] até 1024 blocos, mas não menos de 16 blocos. Este parâmetro só pode ser definido na inicialização do servidor.

`max_stack_depth` (`integer`) [#](#GUC-MAX-STACK-DEPTH): Especifica a profundidade máxima segura do pilha de execução do servidor. O ajuste ideal para este parâmetro é o limite real do tamanho da pilha imposto pelo kernel (conforme definido por `ulimit -s` ou equivalente local), menos uma margem de segurança de aproximadamente um megabyte. A margem de segurança é necessária porque a profundidade da pilha não é verificada em todas as rotinas do servidor, mas apenas em rotinas potencialmente recursivas. Se este valor for especificado sem unidades, ele é considerado em kilobytes. O ajuste padrão é de dois megabytes (`2MB`), que é conservadoramente pequeno e improvável que cause falhas. No entanto, pode ser muito pequeno para permitir a execução de funções complexas. Somente usuários superusuários e usuários com o privilégio apropriado `SET` podem alterar este ajuste.

Definir `max_stack_depth` mais alto do que o limite real do kernel significará que uma função recursiva descontrolada pode fazer com que um processo de backend individual falhe. Em plataformas onde o PostgreSQL pode determinar o limite do kernel, o servidor não permitirá que essa variável seja definida para um valor inseguro. No entanto, nem todas as plataformas fornecem as informações, então é recomendado ter cautela ao selecionar um valor.

`shared_memory_type` (`enum`) [#](#GUC-SHARED-MEMORY-TYPE): Especifica a implementação de memória compartilhada que o servidor deve usar para a região principal de memória compartilhada que contém os buffers compartilhados do PostgreSQL e outros dados compartilhados. Os valores possíveis são `mmap` (para memória compartilhada anônima alocada usando `mmap`), `sysv` (para memória compartilhada do Sistema V alocada via `shmget`) e `windows` (para memória compartilhada do Windows). Nem todos os valores são suportados em todas as plataformas; a primeira opção suportada é a padrão para aquela plataforma. O uso da opção `sysv`, que não é padrão em nenhuma plataforma, é geralmente desencorajado, pois geralmente requer configurações de kernel não padrão para permitir grandes alocações (ver [Seção 18.4.1][(kernel-resources.md#SYSVIPC "18.4.1. Shared Memory and Semaphores")]). Este parâmetro só pode ser definido no início do servidor.

`dynamic_shared_memory_type` (`enum`) [#](#GUC-DYNAMIC-SHARED-MEMORY-TYPE): Especifica a implementação de memória compartilhada dinâmica que o servidor deve usar. Os valores possíveis são `posix` (para memória compartilhada POSIX alocada usando `shm_open`), `sysv` (para memória compartilhada System V alocada via `shmget`), `windows` (para memória compartilhada Windows), e `mmap` (para simular memória compartilhada usando arquivos mapeados de memória armazenados no diretório de dados). Nem todos os valores são suportados em todas as plataformas; a primeira opção suportada geralmente é a padrão para aquela plataforma. O uso da opção `mmap`, que não é padrão em nenhuma plataforma, geralmente é desencorajado, pois o sistema operacional pode escrever páginas modificadas de volta ao disco repetidamente, aumentando a carga de E/S do sistema; no entanto, pode ser útil para depuração, quando o diretório `pg_dynshmem` é armazenado em um disco RAM, ou quando outras facilidades de memória compartilhada não estão disponíveis. Este parâmetro só pode ser definido no início do servidor.

`min_dynamic_shared_memory` (`integer`) [#](#GUC-MIN-DYNAMIC-SHARED-MEMORY): Especifica a quantidade de memória que deve ser alocada na inicialização do servidor para uso por consultas paralelas. Quando essa região de memória é insuficiente ou esgotada por consultas concorrentes, novas consultas paralelas tentam alocar memória compartilhada extra temporariamente do sistema operacional usando o método configurado com `dynamic_shared_memory_type`, o que pode ser mais lento devido aos encargos de gerenciamento de memória. A memória que é alocada na inicialização com `min_dynamic_shared_memory` é afetada pelo ajuste `huge_pages` em sistemas operacionais onde isso é suportado, e pode ser mais provável que beneficie de páginas maiores em sistemas operacionais onde isso é gerenciado automaticamente. O valor padrão é `0` (nenhum). Este parâmetro só pode ser definido na inicialização do servidor.

### 19.4.2. Disco [#](#RUNTIME-CONFIG-RESOURCE-DISK)

`temp_file_limit` (`integer`) [#](#GUC-TEMP-FILE-LIMIT): Especifica a quantidade máxima de espaço em disco que um processo pode usar para arquivos temporários, como arquivos temporários de ordenação e hash, ou o arquivo de armazenamento para um cursor mantido. Uma transação que tente exceder esse limite será cancelada. Se esse valor for especificado sem unidades, ele é considerado em kilobytes. `-1` (o padrão) significa sem limite. Somente usuários superusuários e usuários com o privilégio apropriado `SET` podem alterar essa configuração.

Essa configuração limita o espaço total usado em qualquer instante por todos os arquivos temporários usados por um processo específico do PostgreSQL. Deve-se notar que o espaço em disco usado para tabelas temporárias explícitas, em oposição aos arquivos temporários usados nos bastidores na execução de consultas, *não* conta para esse limite.

`file_copy_method` (`enum`) [#](#GUC-FILE-COPY-METHOD): Especifica o método usado para copiar arquivos. Os valores possíveis são `COPY` (padrão) e `CLONE` (se o suporte operacional estiver disponível).

Este parâmetro afeta:

* `CREATE DATABASE ... STRATEGY=FILE_COPY` * `ALTER DATABASE ... SET TABLESPACE ...`

`CLONE` utiliza as chamadas de sistema `copy_file_range()` (Linux, FreeBSD) ou `copyfile` (macOS), dando ao kernel a oportunidade de compartilhar blocos de disco ou empurrar o trabalho para camadas mais baixas em alguns sistemas de arquivos.

`file_extend_method` (`enum`) [#](#GUC-FILE-EXTEND-METHOD): Especifica o método utilizado para estender arquivos de dados durante operações em massa, como `COPY`. A primeira opção disponível é usada como padrão, dependendo do sistema operacional:

* `posix_fallocate` (Unix) utiliza a interface padrão POSIX para alocar espaço em disco, mas está ausente em alguns sistemas. Se estiver presente, mas o sistema de arquivos subjacente não o suportar, essa opção silenciosamente retorna para `write_zeros`. As versões atuais do BTRFS são conhecidas por desativar a compressão quando essa opção é usada. Isso é o padrão em sistemas que possuem a função. * `write_zeros` estende os arquivos escrevendo blocos de bytes nulos. Isso é o padrão em sistemas que não possuem a função `posix_fallocate`.

O método `write_zeros` é sempre utilizado quando os arquivos de dados são estendidos por 8 blocos ou menos.

`max_notify_queue_pages` (`integer`) [#](#GUC-MAX-NOTIFY-QUEUE-PAGES): Especifica o valor máximo de páginas alocadas para a fila de espera [NOTIFY](sql-notify.md "NOTIFY") / [LISTEN](sql-listen.md "LISTEN"). O valor padrão é 1048576. Para páginas de 8 KB, permite consumir até 8 GB de espaço em disco. Este parâmetro só pode ser definido no início do servidor.

### 19.4.3. Uso de recursos do kernel [#](#RUNTIME-CONFIG-RESOURCE-KERNEL)

`max_files_per_process` (`integer`) [#](#GUC-MAX-FILES-PER-PROCESS): Define o número máximo de arquivos abertos que cada subprocesso do servidor pode abrir simultaneamente; os arquivos já abertos no postmaster não são contabilizados para este limite. O padrão é mil arquivos.

Se o kernel estiver aplicando um limite seguro por processo, você não precisa se preocupar com essa configuração. Mas em algumas plataformas (especialmente, a maioria dos sistemas BSD), o kernel permitirá que processos individuais abram muitos mais arquivos do que o sistema realmente pode suportar, se muitos processos tentarem abrir tantos arquivos. Se você encontrar falhas de "Existem muitos arquivos abertos", tente reduzir essa configuração. Esse parâmetro só pode ser definido no início do servidor.

### 19.4.4. Escritor de fundo [#](#RUNTIME-CONFIG-RESOURCE-BACKGROUND-WRITER)

Existe um processo de servidor separado chamado *escritor de segundo plano*, cuja função é emitir escritas de tampões compartilhados "sujos" (novos ou modificados). Quando o número de tampões compartilhados limpos parece ser insuficiente, o escritor de segundo plano escreve alguns tampões sujos no sistema de arquivos e os marca como limpos. Isso reduz a probabilidade de que os processos de servidor que lidam com consultas do usuário não consigam encontrar tampões limpos e precisem escrever tampões sujos eles mesmos. No entanto, o escritor de segundo plano causa um aumento líquido na carga de E/S, porque, embora uma página repetidamente suja possa ser escrita de outra forma apenas uma vez por intervalo de verificação, o escritor de segundo plano pode escrevê-la várias vezes, pois é suja no mesmo intervalo. Os parâmetros discutidos nesta subseção podem ser usados para ajustar o comportamento para necessidades locais.

`bgwriter_delay` (`integer`) [#](#GUC-BGWRITER-DELAY): Especifica o atraso entre os ciclos de atividade para o escritor de fundo. Em cada ciclo, o escritor emite leituras para um número de buffers sujos (controlável pelos seguintes parâmetros). Em seguida, dorme por o tempo de `bgwriter_delay`, e repete. Quando não há buffers sujos na pilha de buffers, no entanto, ele entra em um sono mais longo, independentemente de `bgwriter_delay`. Se este valor for especificado sem unidades, ele é considerado em milissegundos. O valor padrão é de 200 milissegundos (`200ms`). Note que, em alguns sistemas, a resolução efetiva dos atrasos de sono é de 10 milissegundos; definir `bgwriter_delay` para um valor que não seja um múltiplo de 10 pode ter os mesmos resultados que definí-lo para o próximo múltiplo superior de 10. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`bgwriter_lru_maxpages` (`integer`) [#](#GUC-BGWRITER-LRU-MAXPAGES): Em cada rodada, não mais do que este número de buffers serão escritos pelo escritor de fundo. Definir este valor como zero desativa a escrita de fundo. (Observe que os pontos de verificação, que são gerenciados por um processo auxiliar separado e dedicado, não são afetados.) O valor padrão é de 100 buffers. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`bgwriter_lru_multiplier` (`floating point`) [#](#GUC-BGWRITER-LRU-MULTIPLIER): O número de buffers sujos escritos em cada rodada é baseado no número de novos buffers que foram necessários pelos processos do servidor durante as rodadas recentes. A necessidade recente média é multiplicada por `bgwriter_lru_multiplier` para chegar a uma estimativa do número de buffers que serão necessários na próxima rodada. Buffers sujos são escritos até que haja tantos buffers limpos e reutilizáveis disponíveis. (No entanto, não mais de `bgwriter_lru_maxpages` buffers serão escritos por rodada.) Assim, um ajuste de 1,0 representa uma política de “just in time” de escrita exatamente o número de buffers previstos como necessários. Valores maiores fornecem algum alívio contra picos na demanda, enquanto valores menores intencionalmente deixam as escritas para serem feitas pelos processos do servidor. O padrão é 2,0. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`bgwriter_flush_after` (`integer`) [#](#GUC-BGWRITER-FLUSH-AFTER): Sempre que mais de essa quantidade de dados tiver sido escrita pelo escritor em segundo plano, tente forçar o SO a emitir esses escritos no armazenamento subjacente. Fazer isso limitará a quantidade de dados sujos no cache de páginas do kernel, reduzindo a probabilidade de travamentos quando um `fsync` é emitido no final de um ponto de verificação, ou quando o SO escreve dados de volta em lotes maiores em segundo plano. Muitas vezes, isso resultará em uma latência de transação muito reduzida, mas também há alguns casos, especialmente com cargas de trabalho que são maiores que [shared_buffers](runtime-config-resource.md#GUC-SHARED-BUFFERS), mas menores que o cache de páginas do SO, onde o desempenho pode degradar. Este ajuste pode não ter efeito em algumas plataformas. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. A faixa válida está entre `0`, que desativa a escrita forçada de volta, e `2MB`. O padrão é `512kB` no Linux, `0` em outros lugares. (Se `BLCKSZ` não for 8kB, os valores padrão e máximos escalam proporcionalmente a ele.) Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

Valores menores de `bgwriter_lru_maxpages` e `bgwriter_lru_multiplier` reduzem a carga extra de I/O causada pelo escritor de segundo plano, mas tornam mais provável que os processos do servidor precisem emitir escritas por si mesmos, atrasando consultas interativas.

### 19.4.5. I/O [#](#RUNTIME-CONFIG-RESOURCE-IO)

`backend_flush_after` (`integer`) [#](#GUC-BACKEND-FLUSH-AFTER): Sempre que mais de essa quantidade de dados tiver sido escrita por um único backend, tente forçar o sistema operacional a emitir esses escritos no armazenamento subjacente. Fazer isso limitará a quantidade de dados sujos no cache de páginas do kernel, reduzindo a probabilidade de travamentos quando um `fsync` é emitido no final de um ponto de verificação, ou quando o sistema operacional escreve dados de volta em lotes maiores no segundo plano. Muitas vezes, isso resultará em uma latência de transação muito reduzida, mas também há alguns casos, especialmente com cargas de trabalho maiores que [shared_buffers](runtime-config-resource.md#GUC-SHARED-BUFFERS), mas menores que o cache de páginas do sistema operacional, onde o desempenho pode degradar. Este ajuste pode não ter efeito em algumas plataformas. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. A faixa válida está entre `0`, que desativa a escrita forçada de volta, e `2MB`. O padrão é `0`, ou seja, sem escrita forçada de volta. (Se `BLCKSZ` não for 8kB, o valor máximo escala proporcionalmente a ele.)

`effective_io_concurrency` (`integer`) [#](#GUC-EFFECTIVE-IO-CONCURRENCY): Define o número de operações de E/S de armazenamento concorrentes que o PostgreSQL espera que possam ser executadas simultaneamente. Aumentar esse valor aumentará o número de operações de E/S que qualquer sessão do PostgreSQL tenta iniciar em paralelo. O intervalo permitido é `1` a `1000`, ou `0` para desabilitar a emissão de solicitações de E/S assíncronas. O padrão é `16`.

Valores mais altos terão o maior impacto em armazenamento de maior latência, onde as consultas que, de outra forma, experimentam paradas de I/O perceptíveis e em dispositivos com IOPS elevados. Valores excessivamente altos podem aumentar a latência de I/O para todas as consultas no sistema.

Nos sistemas com suporte a conselhos de pré-pesquisa, `effective_io_concurrency` também controla a distância de pré-pesquisa.

Esse valor pode ser sobrescrito para tabelas em um espaço de tabelas específico, definindo o parâmetro do mesmo nome do espaço de tabelas (consulte [ALTER TABLESPACE](sql-altertablespace.md "ALTER TABLESPACE")).

`maintenance_io_concurrency` (`integer`) [#](#GUC-MAINTENANCE-IO-CONCURRENCY): Semelhante a `effective_io_concurrency`, mas utilizado para trabalhos de manutenção que são realizados em nome de muitas sessões de clientes.

O padrão é `16`. Esse valor pode ser sobrescrito para tabelas em um espaço de tabelas específico, definindo o parâmetro do mesmo nome do espaço de tabelas (consulte ALTER TABLESPACE (sql-altertablespace.md "ALTER TABLESPACE")).

`io_max_combine_limit` (`integer`) [#](#GUC-IO-MAX-COMBINE-LIMIT): Controla o maior tamanho de I/O em operações que combinam I/O, e silenciosamente limita o parâmetro configurável pelo usuário `io_combine_limit`. Este parâmetro só pode ser configurado no início do servidor. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. O tamanho máximo possível depende do sistema operacional e do tamanho do bloco, mas é tipicamente 1MB no Unix e 128kB no Windows. O padrão é 128kB.

`io_combine_limit` (`integer`) [#](#GUC-IO-COMBINE-LIMIT): Controla o maior tamanho de I/O nas operações que combinam I/O. Se definido mais alto que o parâmetro `io_max_combine_limit`, o valor mais baixo será usado silenciosamente, então ambos podem precisar ser aumentados para aumentar o tamanho de I/O. Se este valor for especificado sem unidades, ele é considerado em blocos, ou seja, `BLCKSZ` bytes, tipicamente 8kB. O tamanho máximo possível depende do sistema operacional e do tamanho do bloco, mas é tipicamente 1MB no Unix e 128kB no Windows. O padrão é 128kB.

`io_max_concurrency` (`integer`) [#](#GUC-IO-MAX-CONCURRENCY): Controla o número máximo de operações de E/S que um processo pode executar simultaneamente.

A configuração padrão de `-1` seleciona um número com base em [shared_buffers][(runtime-config-resource.md#GUC-SHARED-BUFFERS)] e no número máximo de processos ([max_connections][(runtime-config-connection.md#GUC-MAX-CONNECTIONS)], [autovacuum_worker_slots][(runtime-config-vacuum.md#GUC-AUTOVACUUM-WORKER-SLOTS)], [max_worker_processes][(runtime-config-resource.md#GUC-MAX-WORKER-PROCESSES)] e [max_wal_senders][(runtime-config-replication.md#GUC-MAX-WAL-SENDERS)]), mas não mais do que `64`.

Este parâmetro só pode ser definido no início do servidor.

`io_method` (`enum`) [#](#GUC-IO-METHOD): Seleciona o método para executar I/O assíncrono. Os valores possíveis são:

* `worker` (execute I/O assíncrono usando processos de trabalho) * `io_uring` (execute I/O assíncrono usando io_uring, requer uma compilação com [`--with-liburing`](install-make.md#CONFIGURE-OPTION-WITH-LIBURING) / [`-Dliburing`](install-meson.md#CONFIGURE-WITH-LIBURING-MESON)) * `sync` (execute I/O elegível para sincronização assíncrona)

O padrão é `worker`.

Este parâmetro só pode ser definido no início do servidor.

`io_workers` (`integer`) [#](#GUC-IO-WORKERS): Seleciona o número de processos de trabalhador de E/S a serem utilizados. O padrão é 3. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

Só tem efeito se [io_method][(runtime-config-resource.md#GUC-IO-METHOD)] estiver definido como `worker`.

### 19.4.6. Processos de trabalhadores [#](#RUNTIME-CONFIG-RESOURCE-WORKER-PROCESSES)

`max_worker_processes` (`integer`) [#](#GUC-MAX-WORKER-PROCESSES): Define o número máximo de processos de fundo que o clúster pode suportar. Este parâmetro só pode ser definido no início do servidor. O padrão é 8.

Ao executar um servidor de espera, você deve definir esse parâmetro no mesmo valor ou superior ao do servidor primário. Caso contrário, as consultas não serão permitidas no servidor de espera.

Ao alterar esse valor, considere também ajustar [max_parallel_workers][(runtime-config-resource.md#GUC-MAX-PARALLEL-WORKERS)], [max_parallel_maintenance_workers][(runtime-config-resource.md#GUC-MAX-PARALLEL-MAINTENANCE-WORKERS)] e [max_parallel_workers_per_gather][(runtime-config-resource.md#GUC-MAX-PARALLEL-WORKERS-PER-GATHER)].

`max_parallel_workers_per_gather` (`integer`) [#](#GUC-MAX-PARALLEL-WORKERS-PER-GATHER): Define o número máximo de trabalhadores que podem ser iniciados por um único nó `Gather` ou `Gather Merge`. Os trabalhadores paralelos são retirados do conjunto de processos estabelecido por [max_worker_processes](runtime-config-resource.md#GUC-MAX-WORKER-PROCESSES), limitado por [max_parallel_workers](runtime-config-resource.md#GUC-MAX-PARALLEL-WORKERS). Observe que o número de trabalhadores solicitado pode não estar disponível na execução em tempo real. Se isso ocorrer, o plano será executado com menos trabalhadores do que o esperado, o que pode ser ineficiente. O valor padrão é 2. Definir esse valor para 0 desativa a execução de consultas paralelas.

Observe que consultas paralelas podem consumir muito mais recursos do que consultas não paralelas, porque cada processo de trabalhador é um processo completamente separado que tem aproximadamente o mesmo impacto no sistema que uma sessão de usuário adicional. Isso deve ser levado em consideração ao escolher um valor para esta configuração, bem como ao configurar outras configurações que controlam a utilização de recursos, como [work_mem][(runtime-config-resource.md#GUC-WORK-MEM)]. Limites de recursos, como `work_mem`, são aplicados individualmente a cada trabalhador, o que significa que a utilização total pode ser muito maior em todos os processos do que normalmente seria para qualquer processo individual. Por exemplo, uma consulta paralela usando 4 trabalhadores pode consumir até 5 vezes mais tempo de CPU, memória, largura de banda de E/S, etc., do que uma consulta que não usa nenhum trabalhador.

Para mais informações sobre consulta paralela, consulte o [Capítulo 15][(parallel-query.md "Chapter 15. Parallel Query")].

`max_parallel_maintenance_workers` (`integer`) [#](#GUC-MAX-PARALLEL-MAINTENANCE-WORKERS): Define o número máximo de trabalhadores paralelos que podem ser iniciados por um único comando de utilitário. Atualmente, os comandos de utilitário paralelos que suportam o uso de trabalhadores paralelos são `CREATE INDEX` ao construir um índice B-tree, GIN ou BRIN, e `VACUUM` sem a opção `FULL`. Os trabalhadores paralelos são retirados do conjunto de processos estabelecido por [max_worker_processes](runtime-config-resource.md#GUC-MAX-WORKER-PROCESSES), limitado por [max_parallel_workers](runtime-config-resource.md#GUC-MAX-PARALLEL-WORKERS). Note que o número de trabalhadores solicitado pode não estar disponível na hora do funcionamento. Se isso ocorrer, a operação de utilitário será executada com menos trabalhadores do que o esperado. O valor padrão é 2. Definir esse valor em 0 desativa o uso de trabalhadores paralelos por comandos de utilitário.

Observe que comandos de utilitário paralelos não devem consumir substancialmente mais memória do que operações equivalentes não paralelas. Essa estratégia difere daquela de consultas paralelas, onde os limites de recursos geralmente se aplicam por processo de trabalhador. Os comandos de utilitário paralelos tratam o limite de recursos `maintenance_work_mem` como um limite a ser aplicado ao comando de utilitário inteiro, independentemente do número de processos de trabalhador paralelos. No entanto, os comandos de utilitário paralelos ainda podem consumir substancialmente mais recursos de CPU e largura de banda de E/S.

`max_parallel_workers` (`integer`) [#](#GUC-MAX-PARALLEL-WORKERS): Define o número máximo de trabalhadores que o clúster pode suportar para operações paralelas. O valor padrão é 8. Ao aumentar ou diminuir esse valor, considere ajustar também [max_parallel_maintenance_workers](runtime-config-resource.md#GUC-MAX-PARALLEL-MAINTENANCE-WORKERS) e [max_parallel_workers_per_gather](runtime-config-resource.md#GUC-MAX-PARALLEL-WORKERS-PER-GATHER). Além disso, observe que uma configuração desse valor que é maior que [max_worker_processes](runtime-config-resource.md#GUC-MAX-WORKER-PROCESSES) não terá efeito, uma vez que os trabalhadores paralelos são retirados do conjunto de processos de trabalhadores estabelecido por essa configuração.

`parallel_leader_participation` (`boolean`) [#](#GUC-PARALLEL-LEADER-PARTICIPATION): Permite que o processo líder execute o plano de consulta nos nós `Gather` e `Gather Merge` em vez de esperar pelos processos de trabalho. O padrão é `on`. Definir esse valor para `off` reduz a probabilidade de os trabalhadores ficarem bloqueados porque o líder não está lendo tuplas rápido o suficiente, mas exige que o processo líder espere pelos processos de trabalho para começar antes que os primeiros tuplos possam ser produzidos. O grau em que o líder pode ajudar ou prejudicar o desempenho depende do tipo de plano, número de trabalhadores e duração da consulta.
