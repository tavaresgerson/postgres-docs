## 28.6. Internos da WAL [#](#WAL-INTERNALS)

O WAL é habilitado automaticamente; não é necessária nenhuma ação do administrador, exceto garantir que os requisitos de espaço em disco para os arquivos WAL sejam atendidos e que qualquer ajuste necessário seja feito (consulte [Seção 28.5](wal-configuration.md)).

Os registros WAL são anexados aos arquivos WAL à medida que cada novo registro é escrito. A posição de inserção é descrita por um Número de Sequência de Log (LSN), que é um deslocamento de byte no WAL, aumentando de forma monótona com cada novo registro. Os valores LSN são retornados como o tipo de dado `pg_lsn`(datatype-pg-lsn.md "8.20. pg_lsn Type"). Os valores podem ser comparados para calcular o volume de dados do WAL que os separa, portanto, são usados para medir o progresso da replicação e recuperação.

Os arquivos WAL são armazenados no diretório `pg_wal` sob o diretório de dados, como um conjunto de arquivos de segmento, normalmente cada um com 16 MB de tamanho (mas o tamanho pode ser alterado alterando a opção `--wal-segsize` initdb). Cada segmento é dividido em páginas, normalmente 8 kB cada (este tamanho pode ser alterado via opção `--with-wal-blocksize` configure). Os cabeçalhos dos registros WAL são descritos em `access/xlogrecord.h`; o conteúdo do registro depende do tipo de evento que está sendo registrado. Os arquivos de segmento recebem números cada vez maiores como nomes, começando em `000000010000000000000001`. Os números não se esgotam, mas levará um tempo muito, muito longo para esgotar o estoque de números disponíveis.

É vantajoso se o WAL estiver localizado em um disco diferente dos arquivos do banco de dados principal. Isso pode ser feito movendo o diretório `pg_wal` para outro local (claro que enquanto o servidor estiver desligado) e criando um link simbólico a partir da localização original no diretório de dados principal para o novo local.

O objetivo do WAL é garantir que o log seja escrito antes de os registros do banco de dados serem alterados, mas isso pode ser contornado por unidades de disco que relatam falsamente um registro de sucesso ao kernel, quando, na verdade, elas apenas armazenaram os dados em cache e ainda não os armazenaram no disco. Uma falha de energia, em tal situação, pode levar à corrupção dos dados irrecuperável. Os administradores devem tentar garantir que os discos que contêm os arquivos WAL do PostgreSQL não façam tais relatórios falsos. (Veja [Seção 28.1](wal-reliability.md)).

Após a criação de um ponto de verificação e o WAL ser esvaziado, a posição do ponto de verificação é salva no arquivo `pg_control`. Portanto, no início da recuperação, o servidor primeiro lê `pg_control` e, em seguida, o registro do ponto de verificação; então, ele realiza a operação REDO, fazendo uma varredura para a frente a partir da localização do WAL indicada no registro do ponto de verificação. Como todo o conteúdo das páginas de dados é salvo no WAL na primeira modificação de página após um ponto de verificação (assumindo que [full_page_writes](runtime-config-wal.md#GUC-FULL-PAGE-WRITES) não está desativado), todas as páginas alteradas desde o ponto de verificação serão restauradas a um estado consistente.

Para lidar com o caso em que o `pg_control` está corrompido, devemos apoiar a possibilidade de digitalizar os segmentos WAL existentes em ordem inversa — do mais recente ao mais antigo — para encontrar o ponto de verificação mais recente. Isso ainda não foi implementado. O `pg_control` é pequeno o suficiente (menos de uma página de disco) para não ser afetado por problemas de escrita parcial, e, até a data desta redação, não houve relatos de falhas no banco de dados devido exclusivamente à incapacidade de ler o próprio `pg_control`. Portanto, embora seja teoricamente um ponto fraco, o `pg_control` não parece ser um problema na prática.
