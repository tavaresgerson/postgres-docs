## 26.3. Falha no failover [#](#WARM-STANDBY-FAILOVER)

Se o servidor principal falhar, o servidor de reserva deve iniciar os procedimentos de failover.

Se o servidor de espera falhar, não será necessário realizar uma falha de transição. Se o servidor de espera puder ser reiniciado, mesmo algum tempo depois, o processo de recuperação também pode ser reiniciado imediatamente, aproveitando a recuperação reativa. Se o servidor de espera não puder ser reiniciado, então uma nova instância de servidor de espera completa deve ser criada.

Se o servidor principal falhar e o servidor de espera se tornar o novo principal, e depois o antigo principal se reiniciar, você deve ter um mecanismo para informar ao antigo principal que ele não é mais o principal. Isso é às vezes conhecido como STONITH (Shoot The Other Node In The Head), o que é necessário para evitar situações em que ambos os sistemas pensam que são os principais, o que levará a confusão e, em última análise, à perda de dados.

Muitos sistemas de falha utilizam apenas dois sistemas, o primário e o de espera, conectados por algum tipo de mecanismo de batimento cardíaco para verificar continuamente a conectividade entre os dois e a viabilidade do primário. Também é possível utilizar um terceiro sistema (conhecido como servidor testemunha) para prevenir alguns casos de falha inadequada, mas a complexidade adicional pode não valer a pena, a menos que seja configurado com o devido cuidado e testes rigorosos.

O PostgreSQL não fornece o software do sistema necessário para identificar uma falha no banco de dados primário e notificar o servidor de banco de dados de reserva. Muitas dessas ferramentas existem e estão bem integradas com as facilidades do sistema operacional necessárias para o sucesso do failover, como a migração de endereço IP.

Uma vez que ocorre a falha para o modo de espera, há apenas um único servidor em operação. Isso é conhecido como um estado degenerado. O antigo servidor de espera agora é o principal, mas o antigo servidor principal está fora de operação e pode permanecer fora de operação. Para retornar ao funcionamento normal, um servidor de espera deve ser recriado, seja no antigo sistema principal quando ele é ativado, ou em um terceiro, possivelmente novo, sistema. A utilidade [pg_rewind](app-pgrewind.md) pode ser usada para acelerar esse processo em grandes clústeres. Uma vez concluído, o principal e o servidor de espera podem ser considerados como tendo trocado de papéis. Algumas pessoas optam por usar um terceiro servidor para fornecer backup para o novo principal até que o novo servidor de espera seja recriado, embora, claramente, isso complica a configuração do sistema e os processos operacionais.

Portanto, mudar de servidor primário para de reserva pode ser rápido, mas exige algum tempo para se preparar novamente o cluster de falha. A troca regular de primário para de reserva é útil, pois permite o tempo de inatividade regular em cada sistema para manutenção. Isso também serve como um teste do mecanismo de falha para garantir que ele realmente funcione quando você precisar. Procedimentos de administração escritos são recomendados.

Se optou pela sincronização de slots de replicação lógica (consulte [Seção 47.2.3](logicaldecoding-explanation.md#LOGICALDECODING-REPLICATION-SLOTS-SYNCHRONIZATION)), então, antes de mudar para o servidor de espera, é recomendável verificar se os slots lógicos sincronizados no servidor de espera estão prontos para o failover. Isso pode ser feito seguindo os passos descritos em [Seção 29.3](logical-replication-failover.md).

Para ativar o failover de um servidor de standby de log-shipping, execute `pg_ctl promote` ou chame `pg_promote()`. Se você está configurando servidores de relatórios que são usados apenas para descarregar consultas somente de leitura do primário, e não para fins de alta disponibilidade, você não precisa promover.
