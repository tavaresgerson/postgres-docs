## 69.3. Estatísticas e Segurança do Planejador [#](#PLANNER-STATS-SECURITY)

O acesso à tabela `pg_statistic` é restrito a superusuários, para que os usuários comuns não possam aprender sobre o conteúdo das tabelas de outros usuários a partir dela. Algumas funções de estimativa de seletividade usarão um operador fornecido pelo usuário (seja o operador que aparece na consulta ou um operador relacionado) para analisar as estatísticas armazenadas. Por exemplo, para determinar se um valor mais comum armazenado é aplicável, o estimulador de seletividade terá que executar o operador apropriado `=` para comparar a constante na consulta ao valor armazenado. Assim, os dados em `pg_statistic` são potencialmente passados para operadores definidos pelo usuário. Um operador adequadamente elaborado pode intencionalmente vazar os operandos passados (por exemplo, registrá-los ou escrevê-los em uma tabela diferente), ou vazá-los acidentalmente, mostrando seus valores em mensagens de erro, em qualquer caso, possivelmente expondo dados de `pg_statistic` a um usuário que não deveria ser capaz de vê-lo.

Para evitar isso, o seguinte se aplica a todas as funções de estimativa de seletividade embutidas. Ao planejar uma consulta, para poder usar estatísticas armazenadas, o usuário atual deve ter privilégio `SELECT` na tabela ou nas colunas envolvidas, ou o operador usado deve ser `LEAKPROOF` (mais precisamente, a função com a qual o operador é baseado). Caso contrário, o estimador de seletividade se comportará como se não houvesse estatísticas disponíveis, e o planejador prosseguirá com suposições padrão ou de fallback. O comando meta `\do+` do programa [psql][(app-psql.md "psql")] é útil para determinar quais operadores são marcados como à prova de vazamento.

Se um usuário não tiver o privilégio necessário na tabela ou nas colunas, então, na maioria dos casos, a consulta receberá um erro de negado de permissão, nesse caso, esse mecanismo é invisível na prática. Mas se o usuário estiver lendo de uma visão com barreira de segurança, então o planejador pode querer verificar as estatísticas de uma tabela subjacente que, de outra forma, é inacessível ao usuário. Nesse caso, o operador deve ser à prova de vazamento ou as estatísticas não serão usadas. Não há feedback direto sobre isso, exceto que o plano pode ser subótimo. Se suspeitar que este seja o caso, pode tentar executar a consulta como um usuário mais privilegiado, para ver se um plano diferente resulta.

Essa restrição se aplica apenas aos casos em que o planejador precisaria executar um operador definido pelo usuário em um ou mais valores de `pg_statistic`. Assim, o planejador é autorizado a usar informações estatísticas genéricas, como a fração de valores nulos ou o número de valores distintos em uma coluna, independentemente dos privilégios de acesso.

As funções de estimativa de seletividade contidas em extensões de terceiros que potencialmente operam em estatísticas com operadores definidos pelo usuário devem seguir as mesmas regras de segurança. Consulte o código-fonte do PostgreSQL para obter orientações.
