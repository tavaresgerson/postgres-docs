## 13.1. Introdução [#](#MVCC-INTRO)

O PostgreSQL oferece um conjunto rico de ferramentas para que os desenvolvedores gerenciem o acesso concorrente aos dados. Internamente, a consistência dos dados é mantida usando um modelo multiversão (Controle de Concorrência Multiversão, MVCC). Isso significa que cada declaração SQL vê um instantâneo dos dados (uma *versão do banco de dados*) como estava há algum tempo, independentemente do estado atual dos dados subjacentes. Isso impede que as declarações vejam dados inconsistentes produzidos por transações concorrentes que realizam atualizações nas mesmas linhas de dados, fornecendo *isolamento de transação* para cada sessão do banco de dados. O MVCC, ao evitar as metodologias de bloqueio dos sistemas de banco de dados tradicionais, minimiza a disputa de bloqueio para permitir um desempenho razoável em ambientes multiusuário.

A principal vantagem de usar o modelo MVCC de controle de concorrência em vez de bloqueio é que, no MVCC, as chaves adquiridas para consulta (leitura) de dados não entram em conflito com as chaves adquiridas para escrita de dados, e, portanto, a leitura nunca bloqueia a escrita e a escrita nunca bloqueia a leitura. O PostgreSQL mantém essa garantia mesmo quando fornece o nível mais estrito de isolamento de transação através do uso de um nível inovador de *Isolamento de Escaneamento Serializável* (SSI).

As facilidades de bloqueio de nível de tabela e de linha também estão disponíveis no PostgreSQL para aplicações que geralmente não precisam de isolamento total de transação e preferem gerenciar explicitamente pontos específicos de conflito. No entanto, o uso adequado do MVCC geralmente fornecerá um desempenho melhor do que os bloqueios. Além disso, os bloqueios consultivos definidos pela aplicação fornecem um mecanismo para adquirir bloqueios que não estão vinculados a uma única transação.
