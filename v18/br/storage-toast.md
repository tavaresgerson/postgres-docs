## 66.2. TOAST [#](#STORAGE-TOAST)

* [66.2.1. Armazenamento TOAST fora de linha, em disco](storage-toast.md#STORAGE-TOAST-ONDISK)
* [66.2.2. Armazenamento TOAST fora de linha, em memória](storage-toast.md#STORAGE-TOAST-INMEMORY)

Esta seção fornece uma visão geral da TOAST (The Oversized-Attribute Storage Technique).

O PostgreSQL utiliza um tamanho de página fixo (comumente 8 kB) e não permite que os tuplos se espalhem por múltiplas páginas. Portanto, não é possível armazenar valores de campo muito grandes diretamente. Para superar essa limitação, valores de campo grandes são comprimidos e/ou divididos em várias linhas físicas. Isso acontece de forma transparente para o usuário, com apenas um pequeno impacto na maioria do código do backend. A técnica é carinhosamente conhecida como TOAST (ou “a melhor coisa desde o pão fatiado”). A infraestrutura TOAST também é usada para melhorar o manuseio de grandes valores de dados em memória.

Apenas certos tipos de dados suportam TOAST — não há necessidade de impor o overhead em tipos de dados que não podem produzir valores de campo grandes. Para suportar TOAST, um tipo de dados deve ter uma representação de comprimento variável (*varlena*), na qual, normalmente, a primeira palavra de quatro bytes de qualquer valor armazenado contém o comprimento total do valor em bytes (incluindo a si mesmo). TOAST não restringe o resto da representação do tipo de dados. As representações especiais coletivamente chamadas de *valores TOASTados* funcionam modificando ou reinterpretando essa palavra de comprimento inicial. Portanto, as funções de nível C que suportam um tipo de dados TOAST-ável devem ter cuidado com a forma como lidam com valores de entrada potencialmente TOASTados: uma entrada pode não consistir realmente em uma palavra de comprimento de quatro bytes e conteúdo até depois de ter sido *desTOASTada*. (Isso normalmente é feito invocando `PG_DETOAST_DATUM` antes de fazer qualquer coisa com um valor de entrada, mas, em alguns casos, abordagens mais eficientes são possíveis. Veja [Seção 36.13.1][(xtypes.md#XTYPES-TOAST "36.13.1. TOAST Considerations")] para mais detalhes.)

O TOAST usurpa dois bits da palavra de comprimento varlena (os bits de alta ordem em máquinas big-endian, os bits de baixa ordem em máquinas little-endian), limitando assim o tamanho lógico de qualquer valor de um tipo de dados TOAST-ável a 1 GB (230 - 1 bytes). Quando ambos os bits são zero, o valor é um valor comum não TOAST-ado do tipo de dados, e os bits restantes da palavra de comprimento indicam o tamanho total do dado (incluindo a palavra de comprimento) em bytes. Quando o bit de alta ordem ou de baixa ordem é definido, o valor tem apenas um cabeçalho de um byte em vez do cabeçalho normal de quatro bytes, e os bits restantes desse byte indicam o tamanho total do dado (incluindo o byte de comprimento) em bytes. Esta alternativa suporta o armazenamento eficiente de espaço de valores mais curtos que 127 bytes, permitindo ainda que o tipo de dados cresça conforme necessário. Valores com cabeçalhos de um byte não estão alinhados em qualquer limite particular, enquanto valores com cabeçalhos de quatro bytes estão alinhados em pelo menos um limite de quatro bytes; esta omissão de alinhamento de preenchimento fornece economias de espaço adicionais que são significativas em comparação com valores curtos. Como um caso especial, se os bits restantes de um cabeçalho de um byte são todos zero (o que seria impossível para uma extensão auto-inclusiva), o valor é um ponteiro para dados fora da linha, com várias alternativas possíveis como descrito abaixo. O tipo e o tamanho de tal *ponteiro TOAST* são determinados por um código armazenado no segundo byte do dado. Por último, quando o bit de alta ordem ou de baixa ordem está claro, mas o bit adjacente está definido, o conteúdo do dado foi comprimido e deve ser descomprimido antes do uso. Neste caso, os bits restantes da palavra de comprimento de quatro bytes indicam o tamanho total do dado comprimido, não os dados originais. Note que a compressão também é possível para dados fora da linha, mas o cabeçalho varlena não indica se isso ocorreu — o conteúdo do ponteiro TOAST indica isso, em vez disso.

A técnica de compressão utilizada para dados comprimidos em linha ou fora da linha pode ser selecionada para cada coluna definindo a opção da coluna `COMPRESSION` em `CREATE TABLE` ou `ALTER TABLE`. O padrão para colunas sem configuração explícita é consultar o parâmetro [default_toast_compression](runtime-config-client.md#GUC-DEFAULT-TOAST-COMPRESSION) no momento em que os dados são inseridos.

Como mencionado, existem vários tipos de datums de ponteiros TOAST. O tipo mais antigo e mais comum é um ponteiro para dados fora da linha armazenados em uma *tabela TOAST* que é separada, mas associada, à tabela que contém o próprio datum do TOAST. Esses datums de ponteiro *em disco* são criados pelo código de gerenciamento TOAST (em `access/common/toast_internals.c`) quando uma tupla que será armazenada em disco é grande demais para ser armazenada como está. Mais detalhes aparecem em [Seção 66.2.1] (storage-toast.md#STORAGE-TOAST-ONDISK "66.2.1. Out-of-Line, On-Disk TOAST Storage"). Alternativamente, um datum de ponteiro TOAST pode conter um ponteiro para dados fora da linha que aparecem em outro lugar na memória. Esses datums são necessariamente de curta duração e nunca aparecerão em disco, mas são muito úteis para evitar a cópia e o processamento redundante de grandes valores de dados. Mais detalhes aparecem em [Seção 66.2.2] (storage-toast.md#STORAGE-TOAST-INMEMORY "66.2.2. Out-of-Line, In-Memory TOAST Storage").

### 66.2.1. Armazenamento TOAST fora de linha, em disco [#](#STORAGE-TOAST-ONDISK)

Se alguma das colunas de uma tabela for TOAST-ável, a tabela terá uma tabela associada TOAST, cujo OID é armazenado na entrada `pg_class`.`reltoastrelid` da tabela. Os valores TOAST armazenados em disco são mantidos na tabela TOAST, conforme descrito em detalhes abaixo.

Os valores fora da linha são divididos (após a compressão, se utilizada) em partes de no máximo `TOAST_MAX_CHUNK_SIZE` bytes (por padrão, esse valor é escolhido para que quatro linhas de partes se encaixem em uma página, resultando em cerca de 2000 bytes). Cada parte é armazenada como uma linha separada na tabela TOAST pertencente à tabela proprietária. Cada tabela TOAST tem as colunas `chunk_id` (um OID que identifica o valor específico TOASTado), `chunk_seq` (um número de sequência para a parte dentro de seu valor) e `chunk_data` (os dados reais da parte). Um índice único em `chunk_id` e `chunk_seq` fornece a recuperação rápida dos valores. Portanto, um dado de ponteiro que representa um valor TOASTado fora da linha em disco precisa armazenar o OID da tabela TOAST na qual procurar e o OID do valor específico (seu `chunk_id`). Por conveniência, os dados de ponteiro também armazenam o tamanho lógico do dado (comprimento original dos dados não comprimidos), o tamanho armazenado físico (diferente se a compressão foi aplicada) e o método de compressão usado, se houver. Permitindo os bytes de cabeçalho varlena, o tamanho total de um dado de ponteiro TOAST em disco é, portanto, de 18 bytes, independentemente do tamanho real do valor representado.

O código de gerenciamento TOAST é acionado apenas quando um valor de linha a ser armazenado em uma tabela é mais largo que `TOAST_TUPLE_THRESHOLD` bytes (normalmente 2 kB). O código TOAST comprime e/ou move os valores dos campos fora da linha até que o valor da linha seja mais curto que `TOAST_TUPLE_TARGET` bytes (também normalmente 2 kB, ajustável) ou não mais ganhos possam ser obtidos. Durante uma operação de UPDATE, os valores dos campos não alterados são normalmente preservados como estão; portanto, uma UPDATE de uma linha com valores fora da linha não incorre em custos TOAST se nenhum dos valores fora da linha mudar.

O código de gerenciamento TOAST reconhece quatro estratégias diferentes para armazenar colunas TOASTáveis em disco:

* `PLAIN` impede a compressão ou o armazenamento fora da linha. Esta é a única estratégia possível para colunas de tipos de dados que não são TOASTáveis.
* `EXTENDED` permite tanto a compressão quanto o armazenamento fora da linha. Esta é a opção padrão para a maioria dos tipos de dados TOASTáveis. A compressão será tentada primeiro, e depois o armazenamento fora da linha se a linha ainda for grande demais.
* `EXTERNAL` permite o armazenamento fora da linha, mas não a compressão. O uso de `EXTERNAL` tornará as operações de substring em colunas largas de `text` e `bytea` mais rápidas (com a penalidade de aumento do espaço de armazenamento), pois essas operações são otimizadas para buscar apenas as partes necessárias do valor fora da linha quando ele não está comprimido.
* `MAIN` permite a compressão, mas não o armazenamento fora da linha. (Na verdade, o armazenamento fora da linha ainda será realizado para tais colunas, mas apenas como último recurso quando não há outra maneira de tornar a linha pequena o suficiente para caber em uma página.)

Cada tipo de dado TOAST-ável especifica uma estratégia padrão para as colunas desse tipo de dado, mas a estratégia para uma coluna de tabela específica pode ser alterada com `ALTER TABLE ... SET STORAGE`(sql-altertable.md "ALTER TABLE").

`TOAST_TUPLE_TARGET` pode ser ajustado para cada tabela usando [`ALTER TABLE ... SET (toast_tuple_target = N)`](sql-altertable.md "ALTER TABLE")

Este esquema tem várias vantagens em comparação com uma abordagem mais direta, como permitir que os valores das linhas se estendam por páginas. Supondo que as consultas geralmente sejam qualificadas por comparações contra valores de chave relativamente pequenos, a maior parte do trabalho do executor será realizada usando a entrada principal da linha. Os grandes valores dos atributos TOAST serão extraídos (se forem selecionados) apenas no momento em que o conjunto de resultados é enviado ao cliente. Assim, a tabela principal é muito menor e mais de suas linhas cabem no cache de buffer compartilhado do que o caso sem armazenamento fora da linha. Os conjuntos de classificação também diminuem, e as classificações serão mais frequentemente realizadas inteiramente na memória. Um pequeno teste mostrou que uma tabela contendo páginas HTML típicas e seus URLs foi armazenada em cerca de metade do tamanho dos dados brutos, incluindo a tabela TOAST, e que a tabela principal continha apenas cerca de 10% de todos os dados (os URLs e algumas pequenas páginas HTML). Não houve diferença de tempo de execução em comparação com uma tabela de comparação não TOAST, na qual todas as páginas HTML foram reduzidas a 7 kB para caber.

### 66.2.2. Armazenamento de TOAST fora de linha, em memória [#](#STORAGE-TOAST-INMEMORY)

Os ponteiros de TOAST podem apontar para dados que não estão no disco, mas que estão em outro lugar na memória do processo atual do servidor. Esses ponteiros, obviamente, não podem ser de longa duração, mas ainda assim são úteis. Atualmente, existem dois subcasos: ponteiros para dados *indiretos* e ponteiros para dados *expandidos*.

Os ponteiros indiretos de TOAST simplesmente apontam para um valor de varlena não indireto armazenado em algum lugar da memória. Este caso foi originalmente criado apenas como uma prova de conceito, mas atualmente é usado durante a decodificação lógica para evitar que seja necessário criar tuplas físicas que excedam 1 GB (como o ato de extrair todos os valores de campo fora da linha para a tupla pode fazer). O caso tem uso limitado, uma vez que o criador do dado do ponteiro é totalmente responsável por garantir que os dados referenciados sobrevivam enquanto o ponteiro puder existir, e não há infraestrutura para ajudar nisso.

Os ponteiros TOAST expandidos são úteis para tipos de dados complexos cuja representação em disco não é especialmente adequada para fins computacionais. Como exemplo, a representação padrão de varlena de um array do PostgreSQL inclui informações sobre dimensionalidade, uma bitmap de nulos se houver algum elemento nulo, e depois os valores de todos os elementos em ordem. Quando o próprio tipo de elemento é de comprimento variável, a única maneira de encontrar o *`N`'º elemento é percorrer todos os elementos anteriores. Essa representação é apropriada para armazenamento em disco devido à sua compactação, mas para cálculos com o array é muito melhor ter uma representação “expandida” ou “desconstruída” na qual todas as localizações de início dos elementos tenham sido identificadas. O mecanismo de ponteiro TOAST suporta essa necessidade, permitindo que um Datum por referência aponte para um valor de varlena padrão (a representação em disco) ou para um ponteiro TOAST que aponta para uma representação expandida em algum lugar na memória. Os detalhes dessa representação expandida dependem do tipo de dados, embora deva ter um cabeçalho padrão e atender aos outros requisitos da API dados em `src/include/utils/expandeddatum.h`. As funções de nível C que trabalham com o tipo de dados podem optar por lidar com qualquer uma das representações. As funções que não conhecem a representação expandida, mas simplesmente aplicam `PG_DETOAST_DATUM` aos seus inputs, receberão automaticamente a representação tradicional de varlena; assim, o suporte para uma representação expandida pode ser introduzido gradualmente, uma função de cada vez.

Os ponteiros para valores expandidos são ainda divididos em ponteiros *leitura-escrita* e *somente leitura*. A representação apontada é a mesma em ambos os casos, mas uma função que recebe um ponteiro de leitura-escrita é permitida para modificar o valor referenciado no local, enquanto uma que recebe um ponteiro de apenas leitura não deve; ela deve primeiro criar uma cópia se quiser fazer uma versão modificada do valor. Essa distinção e algumas convenções associadas tornam possível evitar a cópia desnecessária de valores expandidos durante a execução da consulta.

Para todos os tipos de ponteiros TOAST de memória, o código de gerenciamento de TOAST garante que nenhum dado desse tipo possa ser acidentalmente armazenado em disco. Os ponteiros TOAST de memória são automaticamente expandidos para valores normais de varlena em linha antes do armazenamento — e, em seguida, possivelmente convertidos em ponteiros TOAST em disco, se o tuplo que os contém, de outra forma, fosse muito grande.
