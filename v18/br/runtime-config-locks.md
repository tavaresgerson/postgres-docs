## 19.12. Gerenciamento de bloqueio [#](#RUNTIME-CONFIG-LOCKS)

`deadlock_timeout` (`integer`) [#](#GUC-DEADLOCK-TIMEOUT): Este é o tempo que o servidor deve esperar em um bloqueio antes de verificar se há uma condição de deadlock. O check para deadlock é relativamente caro, então o servidor não o executa toda vez que espera por um bloqueio. Aprofundadamente, assumimos que deadlocks não são comuns em aplicações de produção e apenas esperamos no bloqueio por um tempo antes de verificar se há um deadlock. Aumentar esse valor reduz o tempo desperdiçado em verificações desnecessárias de deadlock, mas desacelera o relatório de erros reais de deadlock. Se esse valor for especificado sem unidades, ele é considerado em milissegundos. O padrão é um segundo (`1s`), que provavelmente é o menor valor que você deseja na prática. Em um servidor com alta carga, você pode querer aumentá-lo. Idealmente, a configuração deve exceder o tempo típico da transação, para melhorar as chances de um bloqueio ser liberado antes de o garçom decidir verificar o deadlock. Somente usuários superusuários e usuários com o privilégio apropriado `SET` podem alterar essa configuração.

Quando [log_lock_waits](runtime-config-logging.md#GUC-LOG-LOCK-WAITS) é definido, este parâmetro também determina o tempo que deve ser esperado antes de uma mensagem de log ser emitida sobre a espera do bloqueio. Se você está tentando investigar os atrasos de bloqueio, talvez queira definir um valor menor que o normal [`deadlock_timeout`].

`max_locks_per_transaction` (`integer`) [#](#GUC-MAX-LOCKS-PER-TRANSACTION): A tabela de bloqueio compartilhado tem espaço para objetos `max_locks_per_transaction` (por exemplo, tabelas) por processo do servidor ou transação preparada; portanto, não mais do que esse número de objetos distintos podem ser bloqueados em qualquer momento. Este parâmetro limita o número médio de bloqueios de objetos usados por cada transação; as transações individuais podem bloquear mais objetos, desde que os bloqueios de todas as transações se encaixem na tabela de bloqueio. Isso *não* é o número de linhas que podem ser bloqueadas; esse valor é ilimitado. O padrão, 64, historicamente provou ser suficiente, mas você pode precisar aumentar esse valor se tiver consultas que tocam muitas tabelas diferentes em uma única transação, por exemplo, consulta de uma tabela pai com muitas crianças. Este parâmetro só pode ser definido no início do servidor.

Ao executar um servidor de espera, você deve definir esse parâmetro para ter o mesmo valor ou um valor maior que o do servidor primário. Caso contrário, as consultas não serão permitidas no servidor de espera.

`max_pred_locks_per_transaction` (`integer`) [#](#GUC-MAX-PRED-LOCKS-PER-TRANSACTION): A tabela de bloqueio de predicado compartilhado tem espaço para objetos `max_pred_locks_per_transaction` (por exemplo, tabelas) por processo do servidor ou transação preparada; portanto, não mais do que esse número de objetos distintos podem ser bloqueados em qualquer momento. Este parâmetro limita o número médio de bloqueios de objetos usados por cada transação; as transações individuais podem bloquear mais objetos, desde que os bloqueios de todas as transações se encaixem na tabela de bloqueio. Isso *não* é o número de linhas que podem ser bloqueadas; esse valor é ilimitado. O padrão, 64, historicamente provou ser suficiente, mas você pode precisar aumentar esse valor se tiver clientes que tocam muitas tabelas diferentes em uma única transação serializável. Este parâmetro só pode ser definido no início do servidor.

`max_pred_locks_per_relation` (`integer`) [#](#GUC-MAX-PRED-LOCKS-PER-RELATION): Isso controla quantos páginas ou tuplas de uma única relação podem ser bloqueadas por predicado antes que o bloqueio seja promovido para cobrir toda a relação. Valores maiores ou iguais a zero significam um limite absoluto, enquanto valores negativos significam [max_pred_locks_per_transaction](runtime-config-locks.md#GUC-MAX-PRED-LOCKS-PER-TRANSACTION) dividido pelo valor absoluto deste ajuste. O padrão é -2, que mantém o comportamento das versões anteriores do PostgreSQL. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`max_pred_locks_per_page` (`integer`) [#](#GUC-MAX-PRED-LOCKS-PER-PAGE): Isso controla quantas linhas em uma única página podem ser bloqueadas por predicado antes que o bloqueio seja promovido para cobrir toda a página. O padrão é 2. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.
