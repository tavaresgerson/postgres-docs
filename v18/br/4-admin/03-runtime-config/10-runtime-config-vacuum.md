### 19.10.1. Aspiração Automática [#](#RUNTIME-CONFIG-AUTOVACUUM)

Esses ajustes controlam o comportamento do recurso *autovacuum*. Consulte [Seção 24.1.6](routine-vacuuming.md#AUTOVACUUM) para obter mais informações. Observe que muitos desses ajustes podem ser ignorados por tabela; consulte [Parâmetros de armazenamento](sql-createtable.md#SQL-CREATETABLE-STORAGE-PARAMETERS).

`autovacuum` (`boolean`) [#](#GUC-AUTOVACUUM): Controla se o servidor deve executar o daemon de inicialização do autovacuum. Isso está ativado por padrão; no entanto, [track_counts](runtime-config-statistics.md#GUC-TRACK-COUNTS) também deve ser habilitado para que o autovacuum funcione. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; no entanto, o autovacuum pode ser desativado para tabelas individuais alterando os parâmetros de armazenamento de tabela.

Observe que, mesmo quando esse parâmetro é desativado, o sistema iniciará processos de autovacuum, se necessário, para evitar o enrolamento do ID de transação. Consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND) para obter mais informações.

`autovacuum_worker_slots` (`integer`) [#](#GUC-AUTOVACUUM-WORKER-SLOTS): Especifica o número de slots de backend a ser reservado para os processos de trabalho do autovacuum. O padrão é normalmente 16 slots, mas pode ser menos se as configurações do seu kernel não o suportar (determinado durante o initdb). Este parâmetro só pode ser definido no início do servidor.

Ao alterar esse valor, considere ajustar também [autovacuum_max_workers](runtime-config-vacuum.md#GUC-AUTOVACUUM-MAX-WORKERS).

`autovacuum_max_workers` (`integer`) [#](#GUC-AUTOVACUUM-MAX-WORKERS): Especifica o número máximo de processos de autovacuum (diferente do iniciador de autovacuum) que podem estar em execução ao mesmo tempo. O padrão é `3`. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

Observe que uma configuração para esse valor que é maior que [autovacuum_worker_slots](runtime-config-vacuum.md#GUC-AUTOVACUUM-WORKER-SLOTS) não terá efeito, uma vez que os trabalhadores do autovacuum são retirados do conjunto de slots estabelecido por essa configuração.

`autovacuum_naptime` (`integer`) [#](#GUC-AUTOVACUUM-NAPTIME): Especifica o atraso mínimo entre as execuções do autovacuum em qualquer banco de dados. Em cada rodada, o daemon examina o banco de dados e emite os comandos `VACUUM` e `ANALYZE` conforme necessário para as tabelas desse banco de dados. Se este valor for especificado sem unidades, ele é considerado em segundos. O padrão é um minuto (`1min`). Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor.

`autovacuum_vacuum_threshold` (`integer`) [#](#GUC-AUTOVACUUM-VACUUM-THRESHOLD): Especifica o número mínimo de tuplas atualizadas ou excluídas necessárias para desencadear um `VACUUM` em qualquer uma tabela. O padrão é de 50 tuplas. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_vacuum_insert_threshold` (`integer`) [#](#GUC-AUTOVACUUM-VACUUM-INSERT-THRESHOLD): Especifica o número de tuplas inseridas necessárias para desencadear uma `VACUUM` em qualquer uma tabela. O padrão é 1000 tuplas. Se for especificado -1, o autovacuum não desencadeará uma operação de `VACUUM` em quaisquer tabelas com base no número de inserções. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_analyze_threshold` (`integer`) [#](#GUC-AUTOVACUUM-ANALYZE-THRESHOLD): Especifica o número mínimo de tuplas inseridas, atualizadas ou excluídas necessárias para desencadear um `ANALYZE` em qualquer uma tabela. O padrão é de 50 tuplas. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_vacuum_scale_factor` (`floating point`) [#](#GUC-AUTOVACUUM-VACUUM-SCALE-FACTOR): Especifica uma fração do tamanho da tabela a ser adicionada ao `autovacuum_vacuum_threshold` ao decidir se deve ser acionado um `VACUUM`. O padrão é `0.2` (20% do tamanho da tabela). Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_vacuum_insert_scale_factor` (`floating point`) [#](#GUC-AUTOVACUUM-VACUUM-INSERT-SCALE-FACTOR): Especifica uma fração das páginas não congeladas na tabela para adicionar ao `autovacuum_vacuum_insert_threshold` ao decidir se deve ser acionado um `VACUUM`. O padrão é `0.2` (20% das páginas não congeladas na tabela). Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_analyze_scale_factor` (`floating point`) [#](#GUC-AUTOVACUUM-ANALYZE-SCALE-FACTOR): Especifica uma fração do tamanho da tabela a ser adicionada ao `autovacuum_analyze_threshold` ao decidir se deve ser acionado um `ANALYZE`. O padrão é `0.1` (10% do tamanho da tabela). Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento da tabela.

`autovacuum_vacuum_max_threshold` (`integer`) [#](#GUC-AUTOVACUUM-VACUUM-MAX-THRESHOLD): Especifica o número máximo de tuplas atualizadas ou excluídas necessárias para desencadear um `VACUUM` em qualquer uma tabela, ou seja, um limite para o valor calculado com `autovacuum_vacuum_threshold` e `autovacuum_vacuum_scale_factor`. O padrão é 100.000.000 de tuplas. Se for especificado -1, o autovacuum não aplicará um número máximo de tuplas atualizadas ou excluídas que desencadearão uma operação `VACUUM`. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento.

`autovacuum_freeze_max_age` (`integer`) [#](#GUC-AUTOVACUUM-FREEZE-MAX-AGE): Especifica a idade máxima (em transações) que o campo `pg_class`.`relfrozenxid` de uma tabela pode atingir antes que uma operação `VACUUM` seja forçada para evitar o enrolamento de ID de transação dentro da tabela. Observe que o sistema iniciará processos de autovacuum para evitar o enrolamento mesmo quando o autovacuum esteja desativado.

O vácuo também permite a remoção de arquivos antigos do subdiretório `pg_xact`, e é por isso que o padrão é uma quantidade relativamente baixa de 200 milhões de transações. Este parâmetro só pode ser definido no início do servidor, mas o ajuste pode ser reduzido para tabelas individuais alterando os parâmetros de armazenamento de tabela. Para mais informações, consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND).

`autovacuum_multixact_freeze_max_age` (`integer`) [#](#GUC-AUTOVACUUM-MULTIXACT-FREEZE-MAX-AGE): Especifica a idade máxima (em multixatos) que o campo `pg_class`.`relminmxid` de uma tabela pode atingir antes que uma operação `VACUUM` seja forçada para evitar o envolvimento de ID multixato dentro da tabela. Observe que o sistema iniciará processos de autovacuum para evitar o envolvimento mesmo quando o autovacuum esteja desativado.

A limpeza de multixactos também permite a remoção de arquivos antigos dos subdiretórios `pg_multixact/members` e `pg_multixact/offsets`, e é por isso que o valor padrão é de 400 milhões de multixactos, relativamente baixo. Este parâmetro só pode ser definido no início do servidor, mas o ajuste pode ser reduzido para tabelas individuais alterando os parâmetros de armazenamento de tabela. Para mais informações, consulte [Seção 24.1.5.1](routine-vacuuming.md#VACUUM-FOR-MULTIXACT-WRAPAROUND).

`autovacuum_vacuum_cost_delay` (`floating point`) [#](#GUC-AUTOVACUUM-VACUUM-COST-DELAY): Especifica o valor de atraso de custo que será utilizado em operações automáticas de `VACUUM`. Se -1 for especificado, o valor regular de [vacuum_cost_delay](runtime-config-vacuum.md#GUC-VACUUM-COST-DELAY) será utilizado. Se este valor for especificado sem unidades, ele será considerado em milissegundos. O valor padrão é de 2 milissegundos. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento de tabela.

`autovacuum_vacuum_cost_limit` (`integer`) [#](#GUC-AUTOVACUUM-VACUUM-COST-LIMIT): Especifica o valor limite de custo que será utilizado em operações automáticas de `VACUUM`. Se `-1` for especificado (o que é o padrão), o valor regular de [vacuum_cost_limit](runtime-config-vacuum.md#GUC-VACUUM-COST-LIMIT) será utilizado. Observe que o valor é distribuído proporcionalmente entre os trabalhadores de autovacuum em execução, se houver mais de um, de modo que a soma dos limites para cada trabalhador não exceda o valor desta variável. Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando os parâmetros de armazenamento de tabela.

### 19.10.2. Retardo por vácuo baseado em custo [#](#RUNTIME-CONFIG-RESOURCE-VACUUM-COST)

Durante a execução dos comandos [VACUUM](sql-vacuum.md) e [ANALYZE](sql-analyze.md), o sistema mantém um contador interno que acompanha o custo estimado das várias operações de E/S que são realizadas. Quando o custo acumulado atinge um limite (especificado por `vacuum_cost_limit`, o processo que está realizando a operação dormirá por um curto período de tempo, conforme especificado por `vacuum_cost_delay`. Em seguida, ele redefinirá o contador e continuará a execução.

A intenção deste recurso é permitir que os administradores reduzam o impacto de I/O desses comandos na atividade de banco de dados concorrente. Há muitas situações em que não é importante que comandos de manutenção como `VACUUM` e `ANALYZE` sejam concluídos rapidamente; no entanto, geralmente é muito importante que esses comandos não interfiram significativamente na capacidade do sistema de realizar outras operações de banco de dados. O atraso de vácuo baseado em custo oferece uma maneira para os administradores alcançarem isso.

Esse recurso é desativado por padrão para comandos `VACUUM` emitidos manualmente. Para ativá-lo, defina a variável `vacuum_cost_delay` para um valor não nulo.

`vacuum_cost_delay` (`floating point`) [#](#GUC-VACUUM-COST-DELAY): O tempo que o processo permanecerá em espera quando o limite de custo tiver sido excedido. Se esse valor for especificado sem unidades, ele será considerado em milissegundos. O valor padrão é `0`, que desativa o recurso de atraso de vácuo baseado em custo. Valores positivos permitem a limpeza de vácuo com base no custo.

Ao usar a aspiração baseada em custo, os valores apropriados para `vacuum_cost_delay` são geralmente bastante pequenos, talvez menos de 1 milissegundo. Embora `vacuum_cost_delay` possa ser ajustado para valores de milisegundo fracionário, tais atrasos podem não ser medidos com precisão em plataformas mais antigas. Em tais plataformas, aumentar o consumo de recursos com restrição de `VACUUM` acima do que você obtém em 1 ms exigirá a alteração dos outros parâmetros de custo de aspiração. No entanto, você deve manter `vacuum_cost_delay` tão pequeno quanto a sua plataforma medir consistentemente; grandes atrasos não são úteis.

`vacuum_cost_page_hit` (`integer`) [#](#GUC-VACUUM-COST-PAGE-HIT): O custo estimado para varrer um buffer encontrado no cache de buffer compartilhado. Representa o custo para bloquear o pool de buffer, procurar a tabela hash compartilhada e digitalizar o conteúdo da página. O valor padrão é `1`.

`vacuum_cost_page_miss` (`integer`) [#](#GUC-VACUUM-COST-PAGE-MISS): O custo estimado para varrer um buffer que precisa ser lido do disco. Isso representa o esforço para bloquear o pool de buffers, procurar a tabela hash compartilhada, ler o bloco desejado a partir do disco e digitalizar seu conteúdo. O valor padrão é `2`.

`vacuum_cost_page_dirty` (`integer`) [#](#GUC-VACUUM-COST-PAGE-DIRTY): O custo estimado cobrado quando o vácuo modifica um bloco que estava previamente limpo. Representa o I/O adicional necessário para esvaziar o bloco sujo novamente no disco. O valor padrão é `20`.

`vacuum_cost_limit` (`integer`) [#](#GUC-VACUUM-COST-LIMIT): Este é o custo acumulado que fará com que o processo de aspiração durma por `vacuum_cost_delay`. O padrão é `200`.

Nota

Existem certas operações que possuem bloqueios críticos e, portanto, devem ser concluídas o mais rapidamente possível. Os atrasos de vácuo baseados no custo não ocorrem durante tais operações. Portanto, é possível que o custo se acumule muito mais alto do que o limite especificado. Para evitar atrasos desnecessariamente longos, em tais casos, o atraso real é calculado como `vacuum_cost_delay` * `accumulated_balance` / `vacuum_cost_limit` com um máximo de `vacuum_cost_delay` * 4.

### 19.10.3. Comportamento padrão [#](#RUNTIME-CONFIG-VACUUM-DEFAULT)

`vacuum_truncate` (`boolean`) [#](#GUC-VACUUM-TRUNCATE): Habilita ou desabilita o vácuo para tentar truncar quaisquer páginas vazias no final da tabela. O valor padrão é `true`. Se `true`, `VACUUM` e autovacuum realizam a truncagem e o espaço em disco para as páginas truncadas é devolvido ao sistema operacional. Observe que a truncagem requer um `ACCESS EXCLUSIVE` bloqueio na tabela. O parâmetro `TRUNCATE` de [`VACUUM`](sql-vacuum.md), se especificado, substitui o valor deste parâmetro. O ajuste também pode ser substituído para tabelas individuais, alterando os parâmetros de armazenamento da tabela.

### 19.10.4. Congelamento [#](#RUNTIME-CONFIG-VACUUM-FREEZING)

Para manter a correção mesmo após os IDs de transação voltarem a se repetir, o PostgreSQL marca as linhas que são suficientemente antigas como *congeladas*. Essas linhas são visíveis para todos; outras transações não precisam examinar seu XID de inserção para determinar a visibilidade. `VACUUM` é responsável por marcar as linhas como congeladas. Os seguintes ajustes controlam o comportamento de congelamento de `VACUUM` e devem ser ajustados com base na taxa de consumo de XID do sistema e nos padrões de acesso aos dados das cargas de trabalho dominantes. Consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND) para obter mais informações sobre o reenchimento de IDs de transação e o ajuste desses parâmetros.

`vacuum_freeze_table_age` (`integer`) [#](#GUC-VACUUM-FREEZE-TABLE-AGE): `VACUUM` realiza uma varredura agressiva se o campo `pg_class`.`relfrozenxid` da tabela tiver atingido a idade especificada por esta configuração. Uma varredura agressiva difere de uma `VACUUM` regular na medida em que visita todas as páginas que podem conter XIDs ou MXIDs não congelados, e não apenas aquelas que podem conter tuplas mortas. O valor padrão é de 150 milhões de transações. Embora os usuários possam definir este valor em qualquer lugar de zero a dois bilhões, `VACUUM` limitará silenciosamente o valor efetivo a 95% de [autovacuum_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-FREEZE-MAX-AGE), para que um `VACUUM` manual periódico tenha a chance de ser executado antes de um autovacuum anti-wraparound ser lançado para a tabela. Para mais informações, consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND "24.1.5. Preventing Transaction ID Wraparound Failures").

`vacuum_freeze_min_age` (`integer`) [#](#GUC-VACUUM-FREEZE-MIN-AGE): Especifica a idade de corte (em transações) que `VACUUM` deve usar para decidir se deve desencadear o congelamento de páginas que têm um XID mais antigo. O valor padrão é de 50 milhões de transações. Embora os usuários possam definir esse valor em qualquer lugar de zero a um bilhão, `VACUUM` limitará silenciosamente o valor efetivo para metade do valor de [autovacuum_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-FREEZE-MAX-AGE), para que não haja um período de tempo irrazonavelmente curto entre os autovacuos forçados. Para mais informações, consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND "24.1.5. Preventing Transaction ID Wraparound Failures").

`vacuum_failsafe_age` (`integer`) [#](#GUC-VACUUM-FAILSAFE-AGE): Especifica a idade máxima (em transações) que o campo `pg_class`.`relfrozenxid` de uma tabela pode atingir antes de `VACUUM` tomar medidas extraordinárias para evitar a falha de envolvimento de ID de transação em todo o sistema. Essa é a estratégia de último recurso do `VACUUM`. O sistema de segurança geralmente é acionado quando um autovacuum para prevenir o envolvimento de ID de transação já está em execução há algum tempo, embora seja possível que o sistema de segurança seja acionado durante qualquer `VACUUM`.

Quando o sistema de segurança é acionado, qualquer atraso baseado em custos que estiver em vigor não será mais aplicado, as tarefas de manutenção não essenciais (como a limpeza do índice) são ignoradas e qualquer [*[Estratégia de Acesso ao Buffer](glossary.md#GLOSSARY-BUFFER-ACCESS-STRATEGY)]](glossário.md#GLOSSARY-BUFFER-ACCESS-STRATEGY) em uso será desativado, resultando em `VACUUM` ser livre para utilizar todos os [*[buffers compartilhados](glossary.md#GLOSSARY-SHARED-MEMORY)]](glossário.md#GLOSSARY-SHARED-MEMORY).

O valor padrão é de 1,6 bilhão de transações. Embora os usuários possam definir esse valor em qualquer lugar, de zero a 2,1 bilhão, `VACUUM` ajustará silenciosamente o valor efetivo para não menos de 105% de [autovacuum_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-FREEZE-MAX-AGE).

`vacuum_multixact_freeze_table_age` (`integer`) [#](#GUC-VACUUM-MULTIXACT-FREEZE-TABLE-AGE): `VACUUM` realiza uma varredura agressiva se o campo `pg_class`.`relminmxid` da tabela tiver atingido a idade especificada por esta configuração. Uma varredura agressiva difere de uma `VACUUM` regular na medida em que visita todas as páginas que podem conter XIDs ou MXIDs não congelados, e não apenas aquelas que podem conter tuplas mortas. O valor padrão é de 150 milhões de multixatos. Embora os usuários possam definir este valor em qualquer lugar, de zero a dois bilhões, `VACUUM` limitará silenciosamente o valor efetivo a 95% de [autovacuum_multixact_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-MULTIXACT-FREEZE-MAX-AGE), para que um `VACUUM` manual periódico tenha a chance de ser executado antes de ser lançada uma proteção contra o reenchimento da tabela. Para mais informações, consulte [Seção 24.1.5.1](routine-vacuuming.md#VACUUM-FOR-MULTIXACT-WRAPAROUND "24.1.5.1. Multixacts and Wraparound").

`vacuum_multixact_freeze_min_age` (`integer`) [#](#GUC-VACUUM-MULTIXACT-FREEZE-MIN-AGE): Especifica a idade de corte (em multixatos) que `VACUUM` deve usar para decidir se deve desencadear o congelamento de páginas com um ID de multixato mais antigo. O valor padrão é de 5 milhões de multixatos. Embora os usuários possam definir esse valor em qualquer lugar de zero a um bilhão, `VACUUM` limitará silenciosamente o valor efetivo para metade do valor de [autovacuum_multixact_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-MULTIXACT-FREEZE-MAX-AGE), para que não haja um tempo indevidamente curto entre os autovacúmenos forçados. Para mais informações, consulte [Seção 24.1.5.1](routine-vacuuming.md#VACUUM-FOR-MULTIXACT-WRAPAROUND "24.1.5.1. Multixacts and Wraparound").

`vacuum_multixact_failsafe_age` (`integer`) [#](#GUC-VACUUM-MULTIXACT-FAILSAFE-AGE): Especifica a idade máxima (em multixatos) que o campo `pg_class`.`relminmxid` de uma tabela pode atingir antes que o `VACUUM` tome medidas extraordinárias para evitar a falha de envolvimento de ID multixato em todo o sistema. Essa é a estratégia de último recurso da `VACUUM`. O sistema de segurança geralmente é acionado quando um autovacuum para evitar o envolvimento de ID de transação já está em execução há algum tempo, embora seja possível que o sistema de segurança seja acionado durante qualquer `VACUUM`.

Quando o sistema de segurança é acionado, qualquer atraso baseado em custos que estiver em vigor deixará de ser aplicado, e outras tarefas de manutenção não essenciais (como a limpeza do índice) serão ignoradas.

O valor padrão é de 1,6 bilhão de multixatos. Embora os usuários possam definir esse valor em qualquer lugar, de zero a 2,1 bilhões, `VACUUM` ajustará silenciosamente o valor efetivo para não menos de 105% de [autovacuum_multixact_freeze_max_age](runtime-config-vacuum.md#GUC-AUTOVACUUM-MULTIXACT-FREEZE-MAX-AGE).

`vacuum_max_eager_freeze_failure_rate` (`floating point`) [#](#GUC-VACUUM-MAX-EAGER-FREEZE-FAILURE-RATE): Especifica o número máximo de páginas (como uma fração das páginas totais na relação) que `VACUUM` pode digitalizar e *falhar* em definir tudo congelado no mapa de visibilidade antes de desabilitar a digitalização ávida. Um valor de `0` desativa a digitalização ávida completamente. O padrão é `0.03` (3%).

Observe que, quando o varredura ansiosa está habilitada, apenas as falhas de congelamento contam para o limite, não o congelamento bem-sucedido. Os congelamentos bem-sucedidos de página são limitados internamente a 20% de todas as páginas visíveis, mas não congeladas na relação. Limitar os congelamentos bem-sucedidos de página ajuda a amortizar o overhead em vários vazios normais e limita o potencial prejuízo dos congelamentos ansiosos desperdiçados de páginas que são modificadas novamente antes do próximo vácuo agressivo.

Este parâmetro só pode ser definido no arquivo `postgresql.conf` ou na linha de comando do servidor; mas o ajuste pode ser sobrescrito para tabelas individuais alterando o [parâmetro de armazenamento da tabela correspondente](sql-createtable.md#RELOPTION-VACUUM-MAX-EAGER-FREEZE-FAILURE-RATE). Para mais informações sobre o comportamento de congelamento do vacuum, consulte [Seção 24.1.5](routine-vacuuming.md#VACUUM-FOR-WRAPAROUND "24.1.5. Preventing Transaction ID Wraparound Failures").
