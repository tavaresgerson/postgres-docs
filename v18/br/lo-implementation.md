## 33.2. Características de implementação [#](#LO-IMPLEMENTATION)

A implementação de objetos grandes quebra os objetos grandes em "pedaços" e armazena esses pedaços em linhas no banco de dados. Um índice de árvore B garante pesquisas rápidas para o número correto do pedaço ao realizar leituras e escritas de acesso aleatório.

Os pedaços armazenados para um objeto grande não precisam ser contínuos. Por exemplo, se um aplicativo abre um novo objeto grande, busca um deslocamento de 1.000.000 e escreve alguns bytes lá, isso não resulta na alocação de 1.000.000 de bytes de armazenamento; apenas de pedaços que cobrem o intervalo de bytes de dados realmente escritos. Uma operação de leitura, no entanto, lerá zeros para quaisquer locais não alocados que precedem o último pedaço existente. Isso corresponde ao comportamento comum de arquivos "alocionados esparsos" em sistemas de arquivos Unix.

A partir do PostgreSQL 9.0, os objetos grandes têm um proprietário e um conjunto de permissões de acesso, que podem ser gerenciados usando [GRANT](sql-grant.md) e [REVOKE](sql-revoke.md). Os privilégios `SELECT` são necessários para ler um objeto grande, e os privilégios `UPDATE` são necessários para escrevê-lo ou truncá-lo. Apenas o proprietário do objeto grande (ou um superusuário do banco de dados) pode excluir, comentar ou alterar o proprietário de um objeto grande. Para ajustar esse comportamento para compatibilidade com versões anteriores, consulte o parâmetro de tempo de execução [lo_compat_privileges](runtime-config-compatible.md#GUC-LO-COMPAT-PRIVILEGES).
