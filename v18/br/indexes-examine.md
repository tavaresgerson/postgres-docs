## 11.12. Análise do uso do índice [#](#INDEXES-EXAMINE)

Embora os índices no PostgreSQL não precisem de manutenção ou ajuste, ainda é importante verificar quais índices são realmente utilizados pelo volume de consultas do mundo real. Examinar o uso de índices para uma consulta individual é feito com o comando [EXPLAIN][(sql-explain.md "EXPLAIN")]; sua aplicação para esse propósito é ilustrada em [Seção 14.1][(using-explain.md "14.1. Using EXPLAIN")]. Também é possível coletar estatísticas gerais sobre o uso de índices em um servidor em execução, conforme descrito em [Seção 27.2][(monitoring-stats.md "27.2. The Cumulative Statistics System")].

É difícil formular um procedimento geral para determinar quais índices criar. Há vários casos típicos que foram mostrados nos exemplos ao longo das seções anteriores. Muita experimentação é frequentemente necessária. O restante desta seção oferece algumas dicas para isso:

* Sempre execute [ANALYSE][(sql-analyze.md "ANALYZE")]. Esse comando coleta estatísticas sobre a distribuição dos valores na tabela. Essas informações são necessárias para estimar o número de linhas devolvidas por uma consulta, o que é necessário para o planejador atribuir custos realistas a cada plano de consulta possível. Na ausência de estatísticas reais, alguns valores padrão são assumidos, que quase certamente serão imprecisos. Examinar o uso do índice de uma aplicação sem executar `ANALYZE` é, portanto, uma causa perdida. Consulte [Seção 24.1.3][(routine-vacuuming.md#VACUUM-FOR-STATISTICS "24.1.3. Updating Planner Statistics") e [Seção 24.1.6][(routine-vacuuming.md#AUTOVACUUM "24.1.6. The Autovacuum Daemon") para obter mais informações.
* Use dados reais para experimentação. Usar dados de teste para configurar índices lhe dirá quais índices você precisa para os dados de teste, mas isso é tudo.

É especialmente fatal usar conjuntos de dados de teste muito pequenos. Embora selecionar 1000 de 100000 linhas possa ser um candidato para um índice, selecionar 1 de 100 linhas dificilmente será, porque as 100 linhas provavelmente cabem em uma única página de disco, e não há um plano que possa superar a obtenção sequencial de 1 página de disco.

Além disso, tenha cuidado ao criar dados de teste, que muitas vezes é inevitável quando o aplicativo ainda não está em produção. Valores que são muito semelhantes, completamente aleatórios ou inseridos em ordem ordenada distorcerão as estatísticas, afastando-as da distribuição que os dados reais teriam.
* Quando os índices não são usados, pode ser útil para testar a força de sua utilização. Existem parâmetros de tempo de execução que podem desativar vários tipos de planos (consulte [Seção 19.7.1] [(runtime-config-query.md#RUNTIME-CONFIG-QUERY-ENABLE "19.7.1. Planner Method Configuration")]). Por exemplo, desativar varreduras sequenciais (`enable_seqscan`) e junções de laço aninhado (`enable_nestloop`), que são os planos mais básicos, fará com que o sistema use um plano diferente. Se o sistema ainda escolher uma varredura sequencial ou uma junção de laço aninhado, provavelmente há um motivo mais fundamental por que o índice não está sendo usado; por exemplo, a condição da consulta não corresponde ao índice. (O tipo de consulta que pode usar o tipo de índice é explicado nas seções anteriores.)
* Se o uso do índice resultar em estimativas incorretas, novamente, há duas possibilidades. O custo total é calculado a partir dos custos por linha de cada nó do plano multiplicado pela estimativa de seletividade do nó do plano. Os custos estimados para os nós do plano podem ser ajustados via parâmetros de tempo de execução (descritos em [Seção 19.7.2] [(runtime-config-query.md#RUNTIME-CONFIG-QUERY-CONSTANTS "19.7.2. Planner Cost Constants")]). Uma estimativa de seletividade inexata é devido a estatísticas insuficientes. Pode ser possível melhorar isso ajustando os parâmetros de coleta de estatísticas (consulte [ALTER TABLE] [(sql-altertable.md "ALTER TABLE")]).

Se você não conseguir ajustar os custos para serem mais apropriados, talvez precise recorrer à força explícita do uso do índice. Você também pode querer entrar em contato com os desenvolvedores do PostgreSQL para examinar o problema.
