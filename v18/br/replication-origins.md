## Capítulo 48. Rastreamento do progresso da replicação

As origens de replicação são destinadas a facilitar a implementação de soluções de replicação lógica em cima de [decodificação lógica][(logicaldecoding.md "Chapter 47. Logical Decoding")]. Elas fornecem uma solução para dois problemas comuns:

* Como acompanhar o progresso da replicação com segurança * Como alterar o comportamento da replicação com base na origem de uma linha; por exemplo, para evitar laços em configurações de replicação bidirecional

As origens de replicação têm apenas duas propriedades, um nome e uma ID. O nome, que é o que deve ser usado para se referir à origem em diferentes sistemas, é livre de formato `text`. Deve ser usado de uma maneira que torne improvável conflitos entre origens de replicação criadas por diferentes soluções de replicação; por exemplo, prefixando o nome da solução de replicação a ele. A ID é usada apenas para evitar a necessidade de armazenar a versão longa em situações em que a eficiência de espaço é importante. Nunca deve ser compartilhada em diferentes sistemas.

As origens de replicação podem ser criadas usando a função `pg_replication_origin_create()`(functions-admin.md#PG-REPLICATION-ORIGIN-CREATE); eliminadas usando `pg_replication_origin_drop()`(functions-admin.md#PG-REPLICATION-ORIGIN-DROP); e vistas no catálogo do sistema `pg_replication_origin`(catalog-pg-replication-origin.md "52.44. pg_replication_origin").

Uma parte importante na construção de uma solução de replicação é manter o progresso do replay de forma segura. Quando o processo de aplicação ou todo o clúster morre, é necessário ser possível descobrir até onde os dados foram replicados com sucesso. Soluções ingênuas para isso, como atualizar uma linha em uma tabela para cada transação re-replicata, têm problemas como sobrecarga de tempo de execução e bloat do banco de dados.

Usando a infraestrutura de origem de replicação, uma sessão pode ser marcada como que replicando a partir de um nó remoto (usando a função `pg_replication_origin_session_setup()`(functions-admin.md#PG-REPLICATION-ORIGIN-SESSION-SETUP)). Além disso, o LSN e o selo de tempo de compromisso de cada transação de origem podem ser configurados em uma base por transação usando `pg_replication_origin_xact_setup()`(functions-admin.md#PG-REPLICATION-ORIGIN-XACT-SETUP). Se isso for feito, o progresso da replicação persistirá de forma segura em caso de falha. O progresso da replicação para todas as origens de replicação pode ser visto na vista `pg_replication_origin_status`(view-pg-replication-origin-status.md "53.19. pg_replication_origin_status"). O progresso de uma origem individual, por exemplo, ao retomar a replicação, pode ser obtido usando `pg_replication_origin_progress()`(functions-admin.md#PG-REPLICATION-ORIGIN-PROGRESS) para qualquer origem ou `pg_replication_origin_session_progress()`(functions-admin.md#PG-REPLICATION-ORIGIN-SESSION-PROGRESS) para a origem configurada na sessão atual.

Em topologias de replicação mais complexas do que a replicação de exatamente um sistema para outro, outro problema pode ser que é difícil evitar a replicação de linhas repetidas novamente. Isso pode levar tanto a ciclos na replicação quanto a ineficiências. As origens de replicação fornecem um mecanismo opcional para reconhecer e prevenir isso. Quando configuradas usando as funções referenciadas no parágrafo anterior, todas as alterações e transações passadas para os callbacks dos plugins de saída (ver [Seção 47.6] [(logicaldecoding-output-plugin.md "47.6. Logical Decoding Output Plugins")]) geradas pela sessão são marcadas com a origem de replicação da sessão geradora. Isso permite tratá-las de forma diferente no plugin de saída, por exemplo, ignorando todas as linhas que não são de origem local. Além disso, o callback [`filter_by_origin_cb`](logicaldecoding-output-plugin.md#LOGICALDECODING-OUTPUT-PLUGIN-FILTER-ORIGIN "47.6.4.7. Origin Filter Callback") pode ser usado para filtrar o fluxo de mudança de decodificação lógica com base na fonte. Embora menos flexível, filtrar via esse callback é consideravelmente mais eficiente do que fazer isso no plugin de saída.
