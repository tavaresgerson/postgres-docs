## 51.6. Executor [#](#EXECUTOR)

O *executor* pega o plano criado pelo planejador/optimizador e processa-o recursivamente para extrair o conjunto de linhas necessário. Isso é essencialmente um mecanismo de pipeline de demanda-atração. Toda vez que um nó do plano é chamado, ele deve entregar mais uma linha, ou informar que já entregou as linhas.

Para fornecer um exemplo concreto, suponha que o nó superior seja um nó `MergeJoin`. Antes que qualquer fusão possa ser feita, duas linhas têm que ser obtidas (uma de cada subplano). Então, o executor chama recursivamente a si mesmo para processar os subplanos (ele começa com o subplano anexado ao `lefttree`). O novo nó superior (o nó superior do subplano esquerdo) é, digamos, um nó `Sort` e, novamente, é necessária a recursão para obter uma linha de entrada. O nó filho do `Sort` pode ser um nó `SeqScan`, representando a leitura real de uma tabela. A execução deste nó faz com que o executor obtenha uma linha da tabela e a retorne até o nó que a solicitou. O nó `Sort` repetidamente chamará seu filho para obter todas as linhas a serem ordenadas. Quando a entrada é esgotada (como indicado pelo nó filho que retorna um NULL em vez de uma linha), o código `Sort` realiza a ordenação e, finalmente, é capaz de retornar sua primeira linha de saída, ou seja, a primeira em ordem ordenada. Ele mantém as linhas restantes armazenadas para que possa entregá-las em ordem ordenada em resposta a demandas posteriores.

O nó `MergeJoin` exige, de forma semelhante, a primeira linha de seu subplano direito. Em seguida, compara as duas linhas para verificar se elas podem ser unidas; se sim, retorna uma linha de junção para seu chamador. Na próxima chamada, ou imediatamente, se não puder unir o par atual de entradas, avança para a próxima linha de uma tabela ou da outra (dependendo de como a comparação saiu), e novamente verifica uma correspondência. Eventualmente, um subplano ou o outro é esgotado, e o nó `MergeJoin` retorna NULL para indicar que mais linhas de junção não podem ser formadas.

As consultas complexas podem envolver muitos níveis de nós do plano, mas a abordagem geral é a mesma: cada nó calcula e retorna sua próxima linha de saída cada vez que é chamado. Cada nó também é responsável por aplicar quaisquer expressões de seleção ou projeção que lhe foram atribuídas pelo planejador.

O mecanismo executor é usado para avaliar os cinco tipos básicos de consulta SQL: `SELECT`, `INSERT`, `UPDATE`, `DELETE` e `MERGE`. Para `SELECT`, o código executor de nível superior só precisa enviar cada linha devolvida pelo plano de consulta para o cliente. `INSERT ... SELECT`, `UPDATE`, `DELETE` e `MERGE` são efetivamente `SELECT`s sob um nó especial de nível superior chamado `ModifyTable`.

`INSERT ... SELECT` alimenta as linhas até `ModifyTable` para inserção. Para `UPDATE`, o planejador garante que cada linha calculada inclua todos os valores das colunas atualizados, além do *TID* (ID do tupla, ou ID da linha) da linha alvo original; esses dados são alimentados até o nó `ModifyTable`, que usa as informações para criar uma nova linha atualizada e marcar a linha antiga como excluída. Para `DELETE`, a única coluna que é realmente devolvida pelo plano é o TID, e o nó `ModifyTable` simplesmente usa o TID para visitar cada linha alvo e marcar como excluída. Para `MERGE`, o planejador une as relações de origem e alvo e inclui todos os valores das colunas exigidos por qualquer uma das cláusulas `WHEN`, além do TID da linha alvo; esses dados são alimentados até o nó `ModifyTable`, que usa as informações para determinar qual cláusula `WHEN` executar e, em seguida, insere, atualiza ou exclui a linha alvo, conforme necessário.

Um comando simples `INSERT ... VALUES` cria uma árvore de plano trivial, composta por um único nó `Result`, que calcula apenas uma linha de resultado, alimentando-a até `ModifyTable` para realizar a inserção.
