## 51.5. Planejador/Otimizador [#](#PLANNER-OPTIMIZER)

* [51.5.1. Gerando Planos Possíveis](planner-optimizer.md#PLANNER-OPTIMIZER-GENERATING-POSSIBLE-PLANS)

A tarefa do *planificador/optimizador* é criar um plano de execução ótimo. Uma consulta SQL dada (e, portanto, uma árvore de consulta) pode ser executada de várias maneiras diferentes, cada uma das quais produzirá o mesmo conjunto de resultados. Se for computacionalmente viável, o otimizador de consulta examinará cada um desses planos de execução possíveis, selecionando, em última análise, o plano de execução que se espera que seja o mais rápido.

### Nota

Em algumas situações, examinar cada maneira possível de executar uma consulta pode levar um tempo e memória excessivos. Isso ocorre, em particular, quando se executam consultas que envolvem um grande número de operações de junção. Para determinar um plano de consulta razoável (não necessariamente ótimo) em um tempo razoável, o PostgreSQL usa um *Otimizador de Consulta Genético* (consulte [Capítulo 61](geqo.md)) quando o número de junções excede um limite (consulte [geqo_threshold](runtime-config-query.md#GUC-GEQO-THRESHOLD)).

O procedimento de busca do planejador, na verdade, funciona com estruturas de dados chamadas *caminhos*, que são simplesmente representações reduzidas de planos que contêm apenas a quantidade de informações que o planejador precisa para tomar suas decisões. Após o caminho mais barato ser determinado, uma árvore de *plano* completa é construída para ser passada ao executor. Isso representa o plano de execução desejado com detalhes suficientes para o executor executá-lo. No restante desta seção, vamos ignorar a distinção entre caminhos e planos.

### 51.5.1. Gerando Planos Possíveis [#](#PLANNER-OPTIMIZER-GENERATING-POSSIBLE-PLANS)

O planejador/otimizador começa gerando planos para digitalizar cada relação individual (tabela) usada na consulta. Os planos possíveis são determinados pelos índices disponíveis em cada relação. Há sempre a possibilidade de realizar uma varredura sequencial em uma relação, então um plano de varredura sequencial é sempre criado. Suponha que um índice seja definido em uma relação (por exemplo, um índice B-tree) e uma consulta contenha a restrição `relation.attribute OPR constant`. Se `relation.attribute` coincidir com a chave do índice B-tree e `OPR` for um dos operadores listados na *classe de operadores* do índice, outro plano é criado usando o índice B-tree para digitalizar a relação. Se houver outros índices presentes e as restrições na consulta coincidir com uma chave de um índice, outros planos serão considerados. Planos de varredura de índice também são gerados para índices que têm uma ordem de classificação que pode corresponder à cláusula `ORDER BY` (se houver) da consulta, ou uma ordem de classificação que pode ser útil para junção de fusão (ver abaixo).

Se a consulta exigir a junção de duas ou mais relações, os planos para a junção das relações são considerados após todas as opções viáveis terem sido encontradas para a digitalização de relações individuais. As três estratégias de junção disponíveis são:

* *join em laço aninhado*: A relação correta é verificada uma vez para cada linha encontrada na relação esquerda. Essa estratégia é fácil de implementar, mas pode ser muito demorada. (No entanto, se a relação correta puder ser verificada com uma verificação de índice, essa pode ser uma boa estratégia. É possível usar valores da linha atual da relação esquerda como chaves para a verificação de índice da direita.)
* *join de junção*: Cada relação é ordenada nos atributos de junção antes de começar a junção. Em seguida, as duas relações são verificadas em paralelo, e as linhas correspondentes são combinadas para formar as linhas de junção. Esse tipo de junção é atraente porque cada relação só precisa ser verificada uma vez. A ordenação necessária pode ser realizada por meio de uma etapa de ordenação explícita ou por verificação da relação na ordem correta usando um índice na chave de junção. * *join de hash*: a relação correta é primeiro verificada e carregada em uma tabela de hash, usando seus atributos de junção como chaves de hash. Em seguida, a relação esquerda é verificada e os valores apropriados de cada linha encontrada são usados como chaves de hash para localizar as linhas correspondentes na tabela.

Quando a consulta envolve mais de duas relações, o resultado final deve ser construído por uma árvore de etapas de junção, cada uma com duas entradas. O planejador examina diferentes sequências possíveis de junção para encontrar a mais barata.

Se a consulta utilizar menos de [geqo_threshold](runtime-config-query.md#GUC-GEQO-THRESHOLD) relações, uma busca quase exaustiva é realizada para encontrar a melhor sequência de junção. O planejador considera preferencialmente junções entre quaisquer duas relações para as quais exista uma cláusula de junção correspondente na qualificação `WHERE` (ou seja, para as quais exista uma restrição como `where rel1.attr1=rel2.attr2`). Os pares de junção sem cláusula de junção são considerados apenas quando não há outra escolha, ou seja, uma relação específica não tem cláusulas de junção disponíveis para qualquer outra relação. Todos os planos possíveis são gerados para cada par de junção considerado pelo planejador, e o que é (estimado ser) o mais barato é escolhido.

Quando o `geqo_threshold` é excedido, as sequências de junção consideradas são determinadas por heurísticas, conforme descrito no [Capítulo 61](geqo.md). Caso contrário, o processo é o mesmo.

A árvore de plano final consiste em varreduras sequenciais ou indexadas das relações de base, além de nós de loop aninhado, junção de fusão ou junção de hash, conforme necessário, além de quaisquer etapas auxiliares necessárias, como nós de classificação ou nós de cálculo de função agregada. A maioria desses tipos de nós de plano tem a capacidade adicional de realizar *seleção* (descartar linhas que não atendem a uma condição Booleana especificada) e *projeção* (computação de um conjunto de colunas derivadas com base em valores de coluna dados, ou seja, avaliação de expressões escalares quando necessário). Uma das responsabilidades do planejador é anexar condições de seleção da cláusula `WHERE` e cálculo das expressões de saída necessárias aos nós mais apropriados da árvore de plano.
