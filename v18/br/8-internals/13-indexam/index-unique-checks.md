## 63.5. Verificação de unicidade do índice [#](#INDEX-UNIQUE-CHECKS)

O PostgreSQL aplica restrições de unicidade SQL usando índices *uniques*, que são índices que não permitem múltiplas entradas com chaves idênticas. Um método de acesso que suporte essa funcionalidade define `amcanunique` como verdadeiro. (Atualmente, apenas o b-tree o suporta.) As colunas listadas na cláusula `INCLUDE` não são consideradas ao aplicar a unicidade.

Devido ao MVCC, é sempre necessário permitir que entradas duplicadas existam fisicamente em um índice: as entradas podem se referir a versões sucessivas de uma única linha lógica. O comportamento que realmente queremos impor é que nenhum instantâneo MVCC possa incluir duas linhas com chaves de índice iguais. Isso se decompõe nos seguintes casos que devem ser verificados ao inserir uma nova linha em um índice único:

* Se uma linha válida conflitante tiver sido excluída pela transação atual, está tudo bem. (Em particular, uma vez que uma atualização sempre exclui a versão da linha antiga antes de inserir a nova versão, isso permitirá uma atualização em uma linha sem alterar a chave.)
* Se uma linha conflitante tiver sido inserida por uma transação ainda não comprometida, o insertador deve esperar para ver se essa transação se compromete. Se ela for revertida, não há conflito. Se ela se compromete sem excluir novamente a linha conflitante, há uma violação de unicidade. (Na prática, apenas esperamos que a outra transação termine e, em seguida, repitamos a verificação de visibilidade em sua totalidade.)
* Da mesma forma, se uma linha válida conflitante tiver sido excluída por uma transação ainda não comprometida, o insertador deve esperar que essa transação se comprometa ou abortado, e, em seguida, repetir o teste.

Além disso, imediatamente antes de relatar uma violação de unicidade de acordo com as regras acima, o método de acesso deve reverificar a vitalidade da linha que está sendo inserida. Se ela for comprometida morta, então nenhuma violação deve ser relatada. (Este caso não pode ocorrer durante o cenário comum de inserção de uma linha que foi criada recentemente pela transação atual. Pode acontecer durante `CREATE UNIQUE INDEX CONCURRENTLY`, no entanto.)

Precisamos que o método de acesso ao índice aplique esses testes por si só, o que significa que ele deve acessar o heap para verificar o status do commit de qualquer linha que seja mostrada como tendo uma chave duplicada de acordo com o conteúdo do índice. Sem dúvida, isso é feio e não modular, mas economiza trabalho redundante: se fôssemos fazer uma sondagem separada, a busca pelo índice de uma linha em conflito seria essencialmente repetida enquanto se encontra o lugar para inserir a entrada do índice da nova linha. Além disso, não há uma maneira óbvia de evitar condições de corrida, a menos que a verificação de conflito seja uma parte integral da inserção da nova entrada de índice.

Se a restrição única for adiável, há uma complexidade adicional: precisamos ser capazes de inserir uma entrada de índice para uma nova linha, mas adiar qualquer erro de violação de unicidade até o final da declaração ou até mesmo mais tarde. Para evitar pesquisas repetidas desnecessárias do índice, o método de acesso ao índice deve realizar uma verificação preliminar de unicidade durante a inserção inicial. Se isso mostrar que definitivamente não há tupla em disputa, estamos concluídos. Caso contrário, programamos uma reverificação para ocorrer quando é hora de impor a restrição. Se, no momento da reverificação, tanto a tupla inserida quanto outra tupla com a mesma chave estiverem em execução, então o erro deve ser relatado. (Note que, para esse propósito, “em execução” realmente significa “qualquer tupla na cadeia HOT da entrada do índice está em execução”.) Para implementar isso, a função `aminsert` é passada um parâmetro `checkUnique` com um dos seguintes valores:

* `UNIQUE_CHECK_NO` indica que não deve ser realizada nenhuma verificação de unicidade (este não é um índice único).
* `UNIQUE_CHECK_YES` indica que este é um índice único não diferível, e a verificação de unicidade deve ser realizada imediatamente, conforme descrito acima.
* `UNIQUE_CHECK_PARTIAL` indica que a restrição de unicidade é diferível. O PostgreSQL usará este modo para inserir a entrada do índice de cada linha. O método de acesso deve permitir entradas duplicadas no índice e reportar quaisquer potenciais duplicatas, retornando falso em `aminsert`. Para cada linha para a qual falso é retornado, um recheck diferido será agendado.

O método de acesso deve identificar quaisquer linhas que possam violar a restrição de unicidade, mas não é um erro que ele informe falsos positivos. Isso permite que a verificação seja feita sem esperar que outras transações sejam concluídas; os conflitos relatados aqui não são tratados como erros e serão verificados novamente mais tarde, quando já não poderão mais ser conflitos.
* `UNIQUE_CHECK_EXISTING` indica que esta é uma verificação deferida de uma linha que foi relatada como uma potencial violação de unicidade. Embora isso seja implementado chamando `aminsert`, o método de acesso não deve *inserir* uma nova entrada de índice neste caso. A entrada de índice já está presente. Em vez disso, o método de acesso deve verificar se há outra entrada de índice ao vivo. Se sim, e se a linha alvo também estiver ao vivo, informe o erro.

Recomenda-se que, em uma chamada `UNIQUE_CHECK_EXISTING`, o método de acesso verifique ainda que a linha alvo realmente tenha uma entrada existente no índice e informe um erro se não o tiver. Isso é uma boa ideia porque os valores dos tuplos do índice passados para `aminsert` terão sido recomputados. Se a definição do índice envolver funções que não são realmente imutáveis, podemos estar verificando a área errada do índice. Verificar que a linha alvo é encontrada na revalidação verifica que estamos procurando pelos mesmos valores de tupla que foram usados na inserção original.
